生醫資料格式：從 CSV 到 HDF5 / Zarr / Parquet

當 FASTQ.gz 已經數十 GB、單細胞矩陣是 200k × 30k 的 sparse、影像 dataset 進 TB——純文字 + pandas 不再夠用。本章帶你看懂 HDF5、Zarr、Parquet、Arrow 的差異，以及如何在 R 與 Python 間共享同一份檔案。

When FASTQ.gz reaches tens of GB, single-cell matrices are 200k × 30k sparse, and imaging datasets enter TB scale — plain text + pandas no longer suffice. This chapter walks through HDF5, Zarr, Parquet, and Arrow, plus how to share the same file between R and Python.

動機

為什麼純文字檔不夠了？

CSV / TSV / FASTQ 是純文字，可讀性高、跨平台、但有三類致命缺點：

  • 體積膨脹：單細胞 raw count matrix (200k cells × 30k genes) 存成 CSV 要 ~30GB；存成稀疏 HDF5 只要 ~500MB。
  • 無 schema：欄位型別、單位、缺失值表示法不明確，每讀一次要重新猜。
  • 無隨機存取：要拿第 1,000,000 列必須掃過前 999,999 列；對大型 dataset 完全不可行。

本章六個格式：HDF5、Zarr、Parquet、Arrow / Feather、AnnData/Loom、索引化 BAM/VCF，各自解決上述問題的一面。

CSV/TSV/FASTQ are plain text — human-readable, cross-platform, but with three fatal flaws:

  • Size explosion: A single-cell raw count matrix (200k cells × 30k genes) is ~30GB as CSV but only ~500MB as sparse HDF5.
  • No schema: Column types, units, and missing-value conventions are ambiguous and must be re-inferred every read.
  • No random access: Fetching row 1,000,000 requires scanning rows 1 through 999,999 — impractical at scale.

The six formats here — HDF5, Zarr, Parquet, Arrow/Feather, AnnData/Loom, indexed BAM/VCF — each solve a facet of these problems.

💡
學習策略：不要記每個 API。記住「資料形狀 → 格式」的對應：n-dim array → HDF5/Zarr；table → Parquet/Arrow；read-once stream → 文字壓縮就好。 Learning strategy: Don't memorize every API. Remember the "shape → format" mapping: n-d arrays → HDF5/Zarr; tables → Parquet/Arrow; read-once stream → plain compressed text is fine.
主題 1

一、HDF5 — 階層式科學資料容器

HDF5 是 NASA / NIH 等大型科學計畫的事實標準。一個 .h5 檔像一個「檔案系統內的檔案系統」：可以有 group (像目錄)、dataset (像 NumPy array)、attribute (metadata)。10x Cellranger 輸出 filtered_feature_bc_matrix.h5 就是 HDF5。Bioconductor 的 HDF5Array、DelayedArray 也建立在 HDF5 之上。

HDF5 is the de-facto standard for large scientific projects (NASA, NIH). A .h5 file behaves like a filesystem inside a file: groups (directories), datasets (NumPy arrays), attributes (metadata). 10x Cellranger's filtered_feature_bc_matrix.h5 is HDF5. Bioconductor's HDF5Array and DelayedArray are also HDF5-backed.

h5py.py — Python read/write
import h5py, numpy as np

# Write: chunked + compressed dataset, 30k × 200k sparse-like
with h5py.File("counts.h5", "w") as f:
    grp = f.create_group("sample01")
    grp.create_dataset(
        "X", data=np.random.poisson(1, (30000, 200000)),
        chunks=(500, 500),       # <-- chunk size matters!
        compression="gzip", compression_opts=4,
    )
    grp.attrs["genome"] = "GRCh38"
    grp.attrs["date"]   = "2026-05-15"

# Read only one slice — never loads full matrix
with h5py.File("counts.h5", "r") as f:
    chunk = f["sample01/X"][0:100, 0:1000]   # lazy
    genome = f["sample01"].attrs["genome"]
rhdf5.R — Bioconductor side
library(rhdf5)
library(HDF5Array)

# Write
h5createFile("counts.h5")
h5createGroup("counts.h5", "sample01")
h5write(matrix(rpois(1e6, 1), 1000, 1000),
        file="counts.h5", name="sample01/X")

# DelayedArray-backed: on-disk, looks like in-memory matrix
X <- HDF5Array("counts.h5", "sample01/X")
dim(X)              # 1000 1000 — but nothing loaded yet
gene_means <- rowMeans(X)  # computed in chunks

chunk.py — chunk shape matters
# Chunk shape decides the access pattern
# Row-major access? chunks=(1000, ncols) — one row = one I/O
# Per-gene access? chunks=(nrows, 1000) — one gene = one I/O
# Random tiles? chunks=(500, 500) — balanced

# Compression vs random access: gzip = small but slower to decompress
#                              lzf  = ~2x faster, less ratio
#                              blosc/zstd (h5py-plugin) = best of both

import hdf5plugin
f.create_dataset("X", data=X, chunks=(500,500),
                 **hdf5plugin.Blosc(cname="zstd", clevel=5))
⚠️
注意：HDF5 是「單檔案」，並行寫入需要 MPI 版本 (Parallel HDF5)。多 worker 同時寫入普通 HDF5 會造成檔案損毀——這正是 Zarr 出現的原因。 Note: HDF5 is single-file; concurrent writes require Parallel HDF5 (MPI). Multiple workers writing to plain HDF5 corrupt the file — which is exactly why Zarr exists.
主題 2

二、Zarr — Cloud-Native 的下一代 HDF5

Zarr 把 HDF5 的概念「拆檔案」：每個 chunk 是一個獨立的小檔 (或 S3 object)，整個 array 是一個目錄。優點：

  • 並行寫：不同 worker 寫不同 chunk，毫無衝突。
  • 雲端原生：直接讀 s3://bucket/array.zarr/，無需先下載整檔。
  • OME-Zarr 是 OME / 影像領域的標準 (顯微影像、空間轉錄)；AnnData on Zarr 是單細胞的趨勢。

Zarr breaks HDF5's "one file" constraint: each chunk is a separate small file (or S3 object), and the whole array is a directory. Benefits:

  • Parallel writes: Different workers write different chunks without conflict.
  • Cloud-native: Read s3://bucket/array.zarr/ directly — no full-file download.
  • OME-Zarr is the OME imaging standard (microscopy, spatial transcriptomics); AnnData on Zarr is the single-cell direction.
zarr_basic.py
import zarr, numpy as np

# Create a chunked array; results in a directory `counts.zarr/`
z = zarr.open("counts.zarr", mode="w",
              shape=(30000, 200000), chunks=(500, 500),
              dtype="int32", compressor=zarr.Blosc(cname="zstd"))
z[0:100, 0:100] = np.random.poisson(1, (100,100))

# Hierarchy like HDF5 groups
root = zarr.open("experiment.zarr", mode="w")
root.create_dataset("sample01/X", shape=(30000,200000), chunks=(500,500))
root["sample01"].attrs["genome"] = "GRCh38"
zarr_cloud.py — read directly from S3
import zarr, s3fs

主題 3

三、Parquet 與 Arrow — 表格式資料的最強組合

Apache Parquet 是磁碟上的 columnar 格式：同一欄位連續存放、可單獨壓縮、查詢時只讀需要的欄位 (predicate pushdown)。Apache Arrow 是記憶體中的 columnar 表達；同樣的 bytes 可以在 R / Python / Julia / DuckDB / Spark 間 zero-copy 共享。Feather 是 Arrow 的磁碟版 (適合短期 cache)，Parquet 適合長期儲存。

Apache Parquet is an on-disk columnar format: columns stored contiguously, compressed independently, with predicate pushdown so queries only read needed columns. Apache Arrow is the in-memory columnar representation — the same bytes can be zero-copy shared across R / Python / Julia / DuckDB / Spark. Feather is Arrow-on-disk (short-term cache); Parquet is for long-term storage.

parquet.py — read/write + partition
import pandas as pd, pyarrow as pa, pyarrow.parquet as pq

variants = pd.DataFrame({
    "chrom": ["chr1"]*3 + ["chr2"]*2,
    "pos":   [100, 200, 300, 400, 500],
    "qual":  [30.1, 42.0, 28.5, 35.0, 50.2],
})

# Plain write
variants.to_parquet("variants.parquet", compression="zstd")

# Partitioned: one directory per chrom → reads chr1 without touching chr2
variants.to_parquet("variants_partitioned/", partition_cols=["chrom"])
# variants_partitioned/chrom=chr1/part-0.parquet
# variants_partitioned/chrom=chr2/part-0.parquet
arrow.R — read same file in R
library(arrow)
library(dplyr)

# Read parquet written by Python — exact same bytes, same types
df <- read_parquet("variants.parquet")

# Open a partitioned dataset lazily
ds <- open_dataset("variants_partitioned/")
ds %>% filter(chrom == "chr1", qual > 30) %>%
       select(chrom, pos, qual) %>%
       collect()      # pushdown: chr2 partition never opened
predicate.py — pushdown saves I/O
import pyarrow.parquet as pq

# Read ONLY rows where qual > 30 and ONLY columns chrom, pos
tbl = pq.read_table(
    "variants.parquet",
    columns=["chrom", "pos"],          # column projection
    filters=[("qual", ">", 30)],         # predicate pushdown
)
df = tbl.to_pandas()

# DuckDB on top of Parquet — even more pushdown
import duckdb
duckdb.sql("""
    SELECT chrom, COUNT(*) AS n
    FROM 'variants_partitioned/*/*.parquet'
    WHERE qual > 30
    GROUP BY chrom
""").df()
zerocopy.py — Arrow as IPC
import pyarrow as pa, pyarrow.feather as ft

# Feather v2 = Arrow IPC file (no compression by default → very fast)
ft.write_feather(variants, "variants.feather")
df2 = ft.read_feather("variants.feather")

# Memory map → zero-copy share between processes
mm = pa.memory_map("variants.feather", "r")
reader = pa.ipc.open_file(mm)
tbl = reader.read_all()       # no allocation, just pointer

# Hand a pyarrow.Table directly to DuckDB / Polars — zero-copy
import polars as pl
pl_df = pl.from_arrow(tbl)

四、領域格式：AnnData、Loom、BAM/VCF index

除了通用的 HDF5/Zarr/Parquet，生醫領域有幾個專屬格式：

  • AnnData (.h5ad) — 單細胞主流，HDF5-based，欄位包含 X (count matrix)、obs (cell metadata)、var (gene metadata)、obsm/varm (embeddings)。也支援 .zarr。
  • MuData — multi-modal AnnData 容器 (RNA + ATAC + ADT + spatial)。
  • Loom — Linnarsson lab 提出、HDF5-based 的單細胞格式，velocyto 用它。
  • BAM/CRAM + .bai/.crai — alignment 二進位格式，bgzip + tabix 提供區段隨機存取。VCF.gz + .tbi 同理。

Besides the general-purpose HDF5/Zarr/Parquet, the bio domain has several dedicated formats:

  • AnnData (.h5ad) — single-cell standard, HDF5-based, with X (count matrix), obs (cell metadata), var (gene metadata), obsm/varm (embeddings). Also supports .zarr.
  • MuData — multi-modal AnnData container (RNA + ATAC + ADT + spatial).
  • Loom — Linnarsson-lab single-cell format, HDF5-based; used by velocyto.
  • BAM/CRAM + .bai/.crai — binary alignment formats; bgzip + tabix provide region-based random access. Same for VCF.gz + .tbi.
anndata.py — on-disk schema
import anndata as ad, h5py

# .h5ad is HDF5 under the hood — inspect with h5py
with h5py.File("pbmc.h5ad") as f:
    print(list(f.keys()))
    # ['X', 'obs', 'obsm', 'obsp', 'raw', 'uns', 'var', 'varm']
    print(f["X"].attrs["encoding-type"])  # 'csr_matrix'

# Backed mode — keep matrix on disk, only metadata in RAM
adata = ad.read_h5ad("pbmc.h5ad", backed="r")
sub = adata[adata.obs.cell_type == "T cell"].to_memory()

📝 自我檢測 (3 題)

Q1. 為什麼 OME 影像社群選擇 Zarr 而非 HDF5？

Q1. Why did the OME imaging community pick Zarr over HDF5?

A. HDF5 不支援壓縮A. HDF5 doesn't support compression
B. Zarr 速度永遠比 HDF5 快B. Zarr is always faster than HDF5
C. Zarr 是多檔結構，支援並行寫與直接讀 S3C. Zarr is multi-file, supports parallel writes and direct S3 reads
D. HDF5 只能存 1D arrayD. HDF5 only stores 1D arrays
答案 C。HDF5 是單檔結構，並行寫需 MPI 版本；要從 S3 讀必須整檔下載。Zarr 每個 chunk 是獨立 object，天然適合雲端與分散式運算。
Answer: C. HDF5 is single-file; concurrent writes need MPI; S3 reads require full download. Zarr stores each chunk as an independent object — cloud- and distributed-native.

Q2. 你有 5 億列 variant annotation 表，常見查詢是 WHERE chrom='chr17' AND qual>30。哪個格式最適合？

Q2. You have 500M variant annotation rows; typical query is WHERE chrom='chr17' AND qual>30. Best format?

A. TSV.gzA. TSV.gz
B. Parquet，按 chrom 分區B. Parquet, partitioned by chrom
C. HDF5 + gzipC. HDF5 + gzip
D. PickleD. Pickle
答案 B。Parquet 的 columnar 結構 + predicate pushdown 讓 WHERE qual>30 只讀 qual 欄；按 chrom 分區後 WHERE chrom='chr17' 只開該目錄，其他分區檔案完全不開啟。
Answer: B. Parquet's columnar layout + predicate pushdown means WHERE qual>30 only reads the qual column; partitioning by chrom means WHERE chrom='chr17' only opens that directory.

Q3. Feather (Arrow IPC) 相對於 Parquet 的核心優勢是？

Q3. Feather (Arrow IPC) vs Parquet — what's Feather's core advantage?

A. 讀寫速度更快、可 memory-map zero-copy，適合短期 cacheA. Faster read/write, memory-mappable zero-copy, ideal for short-term cache
B. 壓縮率更高B. Higher compression ratio
C. 支援 predicate pushdownC. Supports predicate pushdown
D. 適合 PB 等級長期儲存D. Ideal for PB-scale long-term storage
答案 A。Feather 是 Arrow 在記憶體的 layout 直接存檔，幾乎免反序列化；但壓縮 / 長期儲存與跨版本相容性都不如 Parquet——所以實務上 Parquet 用於 long-term，Feather 用於 short-term cache 與跨語言交換。
Answer: A. Feather is Arrow's in-memory layout flushed to disk — nearly zero deserialization. But compression and long-term/version-compat lag Parquet — so in practice Parquet = long-term, Feather = short-term cache & cross-language IPC.
