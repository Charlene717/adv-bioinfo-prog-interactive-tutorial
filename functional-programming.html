函數式程式設計：純函式、map/reduce、closure

為什麼 FP 讓 bioinformatics pipeline 更易於並行、測試、與重現？從 Python 的 functools / itertools 到 R 的 purrr，把 for-loop 重寫成可組合的高階函式。

Why FP makes bioinformatics pipelines easier to parallelize, test, and reproduce — from Python's functools / itertools to R's purrr, rewrite for-loops as composable higher-order functions.

動機

為什麼函數式對 bioinformatics 重要？

bioinformatics pipeline 的本質：「把同一個函式套到 N 個樣本上、收集結果」。這正是 map 的形狀。一旦你的程式以「純函式 + map/reduce」表達，下列三件事就免費送你：

  • 並行化：純函式無副作用，map 可無痛換成 parallel::mclapply、multiprocessing.Pool、Snakemake jobs。
  • 重現性：沒有全域狀態與隱藏副作用，給相同輸入永遠得到相同輸出——這是科學的最低要求。
  • 可測性：純函式不需要 mock 檔案系統 / 資料庫 / 網路，pytest / testthat 一行解決。

本頁六個主題：純函式 vs 副作用、map / filter / reduce、closure & partial、函式組合、不可變性、safely / possibly 容錯。

The essence of a bioinformatics pipeline: "apply the same function to N samples, collect results." That's literally the shape of map. Once your code is expressed as "pure functions + map/reduce," three gifts come for free:

  • Parallelism: pure functions have no side effects, so map swaps trivially for parallel::mclapply, multiprocessing.Pool, or Snakemake jobs.
  • Reproducibility: no global state, no hidden side effects — same input always yields same output. The bedrock of science.
  • Testability: pure functions need no filesystem / DB / network mocks — pytest / testthat one-liner.

Topics: pure vs side effects, map / filter / reduce, closures & partial, composition, immutability, safely / possibly fault tolerance.

💡
學習策略：不必把每個 for-loop 都改寫成 FP。判斷準則：「這個迴圈是 transform (每個元素獨立)？還是 sequential (前後依賴狀態)？」前者用 map，後者保留 for-loop。 Learning strategy: Don't FP-ify every for-loop. The test: "Is this loop a transform (each element independent) or sequential (state depends on prior iteration)?" Use map for the former, keep the for-loop for the latter.
主題 1

一、純函式與副作用

純函式滿足兩條件：(1) 相同輸入永遠回傳相同輸出 (referential transparency)；(2) 不修改外部狀態 (no side effect)。常見副作用：寫檔案、改全域變數、發網路請求、印 log、改傳入物件。

A pure function satisfies two conditions: (1) same input always yields same output (referential transparency); (2) no external state mutation (no side effect). Common side effects: writing files, mutating globals, network calls, printing logs, mutating the argument.

impure.py — three hidden problems
import pandas as pd

THRESHOLD = 0.05             # 1. hidden global dependency
log = []                       # 2. external mutable state

def filter_sig_genes(df):
    df["is_sig"] = df.padj < THRESHOLD  # 3. mutates caller's df!
    log.append(f"filtered at {THRESHOLD}")
    return df[df.is_sig]

# Problems:
#  - Changing THRESHOLD silently changes behavior of every call
#  - The input df is mutated → caller's df is altered (action at a distance)
#  - Hard to test: depends on globals; logs accumulate across tests
pure.py — explicit inputs, return new object
def filter_sig_genes(df: pd.DataFrame, threshold: float = 0.05) -> pd.DataFrame:
    """Return a NEW DataFrame of significant genes; do not mutate input."""
    out = df.copy()
    out["is_sig"] = out.padj < threshold
    return out.loc[out.is_sig].drop(columns="is_sig")

# Benefits:
#  - Threshold is explicit — no spooky-action-at-a-distance
#  - Input df is untouched — safe in pipelines & parallel maps
#  - Trivially testable: assert filter_sig_genes(toy_df, 0.05) == expected
impure.R — hidden global, mutation
THRESHOLD <- 0.05
log_msgs  <- character()

filter_sig <- function(df) {
  df$is_sig <<- df$padj < THRESHOLD          # <<- modifies parent env!
  log_msgs <<- c(log_msgs, "filtered")        # side effect
  df[df$is_sig, ]
}

# R fortunately doesn't have shared mutable objects, but <<- still pollutes
# the enclosing env and creates the same testability nightmare.
pure.R — explicit args, return new df
filter_sig <- function(df, threshold = 0.05) {
  df |>
    dplyr::mutate(is_sig = padj < threshold) |>
    dplyr::filter(is_sig) |>
    dplyr::select(-is_sig)
}

# Tests are now one-liners:
# expect_equal(filter_sig(toy_df, 0.05) |> nrow(), 3)

三、map / filter / reduce — 三大基本動詞




三個動詞概括 99% 的 transform 需求：map = 對每個元素套用同一函式；filter = 留下符合條件的元素；reduce = 把序列摺疊成單一值。Python 用 map / filter / functools.reduce 或 list comprehension；R 用 purrr::map_* / purrr::keep / purrr::reduce。






Three verbs cover 99% of transformations: map applies the same function to each element; filter keeps elements that satisfy a predicate; reduce folds a sequence into a single value. Python: map / filter / functools.reduce or comprehensions. R: purrr::map_* / purrr::keep / purrr::reduce.

map.py — Python idioms
from functools import reduce
from itertools import chain
import operator as op

sample_paths = ["s1.fq.gz", "s2.fq.gz", "s3.fq.gz"]

# map: count reads per sample
counts = [count_reads(p) for p in sample_paths]
# or:  counts = list(map(count_reads, sample_paths))

# filter: keep samples with >= 1M reads
big = [p for p, c in zip(sample_paths, counts) if c >= 1_000_000]

# reduce: total reads across all samples
total = reduce(op.add, counts, 0)            # or just sum(counts)

# map then flatten: gather all variants from all VCFs into one list
all_variants = list(chain.from_iterable(parse_vcf(p) for p in vcf_paths))

library(purrr)

sample_paths <- c("s1.fq.gz", "s2.fq.gz", "s3.fq.gz")

# map() returns a list; typed variants force the output type

# keep / discard = filter
big <- sample_paths |> keep(\(p) count_reads(p) >= 1e6)

# reduce — combine list of data.frames into one
all_qc <- map(sample_paths, run_qc) |> reduce(bind_rows)

# map_dfr is the idiomatic shortcut: returns a single combined data.frame

主題 3


四、Closure 與 partial application




Closure：函式「夾帶」其定義時可見的變數。Partial application：把多參數函式的一部分參數先「凍住」，得到一個只剩剩餘參數的新函式——常用於設定預設值、適配第三方 API。






Closure: a function carries the variables visible at its definition site. Partial application: pre-bind some args of a multi-arg function to get a new function with fewer params — used for setting defaults or adapting third-party APIs.















closure.py — factory pattern
def make_qc_filter(min_features: int, max_mt: float):
    """Return a filter configured for a specific QC setup."""
    def _filter(adata):
        return adata[
            (adata.obs.nFeature >= min_features) &
            (adata.obs.pct_mt   <= max_mt)
        ].copy()
    return _filter

# Each closure remembers its OWN config
strict = make_qc_filter(min_features=500, max_mt=5)
loose  = make_qc_filter(min_features=200, max_mt=20)

# Now pass `strict` or `loose` anywhere a filter is expected
clean = strict(adata)



partial.py — functools.partial
from functools import partial
import pandas as pd

def read_tsv(path, sep="\t", comment="#", header=0):
    return pd.read_csv(path, sep=sep, comment=comment, header=header)

# Pre-bind defaults that fit MAF files (no header, # comments)
read_maf = partial(read_tsv, comment="#", header=None)
read_gtf = partial(read_tsv, comment="#", header=None, sep="\t")

mafs = [read_maf(p) for p in maf_paths]   # clean, intent-revealing

# In a parallel map, partial avoids capturing variables in lambdas
from multiprocessing import Pool
with Pool(8) as p:
    counts = p.map(partial(count_reads, min_q=20), sample_paths)



closure.R
make_qc_filter <- function(min_features, max_mt) {
  function(meta) {
    meta |>
      dplyr::filter(nFeature >= min_features, pct_mt <= max_mt)
  }
}

strict <- make_qc_filter(500, 5)
loose  <- make_qc_filter(200, 20)

# Each closure carries its own bindings (peek inside):
environment(strict)$min_features   # 500
environment(loose)$min_features    # 200

主題 4


五、函式組合與管線




「組合 (compose)」是把多個小函式串成新函式：(f ∘ g)(x) = f(g(x))。bioinformatics pipeline 天然是組合形式：parse_vcf → filter_pass → annotate → write_tsv。Python 沒有原生 compose (但可手寫或用 toolz)；R 有 purrr::compose 與管線。






"Composition" chains small functions into a new one: (f ∘ g)(x) = f(g(x)). Bioinformatics pipelines are naturally compositional: parse_vcf → filter_pass → annotate → write_tsv. Python lacks a built-in compose (write it or use toolz); R has purrr::compose and the pipe.













compose.py — manual or via toolz
from functools import reduce

def compose(*fns):
    """compose(f, g, h)(x) == f(g(h(x)))  — right to left, like math."""
    return lambda x: reduce(lambda acc, fn: fn(acc), reversed(fns), x)

# Small, pure, single-purpose units
def drop_low_qual(rec):   return rec if rec.qual >= 30 else None
def to_bed(rec):          return f"{rec.chrom}\t{rec.start}\t{rec.end}"
def add_annot(rec):       ...

pipeline = compose(to_bed, add_annot, drop_low_qual)
lines = [pipeline(r) for r in variants if pipeline(r) is not None]

# With toolz / cytoolz (preferred for production):
# from toolz import compose, pipe
# pipe(variants, partial(filter, drop_low_qual), partial(map, add_annot), list)



compose.R — three options
library(purrr)

drop_low_qual <- function(rec) if (rec$qual >= 30) rec else NULL
add_annot     <- function(rec) { rec$annot <- lookup(rec$pos); rec }
to_bed        <- function(rec) sprintf("%s\t%d\t%d", rec$chrom, rec$start, rec$end)

# Option 1: purrr::compose — math order (right to left)
pipeline <- compose(to_bed, add_annot, drop_low_qual)

# Option 2: purrr::compose with .dir = "forward" (read left to right)
pipeline2 <- compose(drop_low_qual, add_annot, to_bed, .dir = "forward")

# Option 3: just use |> for a one-off pipeline (most idiomatic in R)
bed_lines <- variants |>
  discard(\(r) is.null(drop_low_qual(r))) |>
  map(add_annot) |>
  map_chr(to_bed)











主題 5


六、不可變性 (immutability)




不可變物件無法被修改——任何「修改」操作都回傳新物件。好處：跨執行緒安全、易於 cache、可作 dict/set key。R 的多數結構天生不可變 (copy-on-modify)；Python 中你需要刻意選擇 tuple、frozenset、@dataclass(frozen=True)，並避免 in-place 操作 (df.drop(...) 而非 df.drop(..., inplace=True))。






Immutable objects cannot be modified — any "modification" returns a new object. Benefits: thread-safe, easy to cache, usable as dict/set keys. Most R structures are immutable by default (copy-on-modify); in Python you must deliberately pick tuple, frozenset, @dataclass(frozen=True) and avoid in-place ops (use df.drop(...), not df.drop(..., inplace=True)).













immut.py — choose immutable structures
from dataclasses import dataclass

@dataclass(frozen=True, slots=True)   # frozen → immutable & hashable
class Variant:
    chrom: str
    pos: int
    ref: str
    alt: str

# Now usable as a dict key — perfect for de-duplication / lookups
seen: set[Variant] = set()
for rec in all_variants:
    v = Variant(rec.chrom, rec.pos, rec.ref, rec.alt)
    if v not in seen:
        seen.add(v)
        write(rec)

# tuple instead of list when the size is fixed
gene_coord: tuple[str, int, int] = ("chr17", 43044295, 43125483)



immut_check.py — pandas: avoid inplace
import pandas as pd

# ❌ Mutates caller's df — surprises in pipelines, breaks parallel maps
def drop_failed(df):
    df.drop(df[df.qual < 30].index, inplace=True)
    return df

# ✅ Return a new df; caller's df untouched
def drop_failed(df: pd.DataFrame) -> pd.DataFrame:
    return df.loc[df.qual >= 30].reset_index(drop=True)

# Bonus: chain pure ops — each step yields a new df
result = (df
    .pipe(drop_failed)
    .pipe(annotate_genes)
    .pipe(lambda d: d.sort_values("chrom"))
)











主題 6


七、安全的 map — safely / possibly / quietly




當你對 1000 個樣本跑 map(samples, run_pipeline)，第 372 個樣本壞了——預設行為是 整批失敗。purrr 的 safely() / possibly() / quietly() 把不純的副作用「函式化」，讓壞的樣本只變成「壞的回傳值」而非中斷流程。






When running map(samples, run_pipeline) over 1000 samples, sample #372 fails — by default the whole batch crashes. purrr's safely() / possibly() / quietly() turn impure side effects into pure return values: a bad sample becomes a "bad return," not a halted pipeline.














safely.R — wrap so errors become return values
library(purrr)

# safely() returns list(result, error). One always exists, one is NULL.
safe_run <- safely(run_pipeline)

results <- map(sample_ids, safe_run)
# results[[1]] is list(result = ..., error = NULL) or vice versa

# Separate the wheat from the chaff
ok       <- results |> map("result") |> compact()        # successes
failures <- results |> map("error")  |> compact()        # errors

# transpose() is the canonical way to flip list-of-lists
res_t <- transpose(results)
ok_results <- res_t$result |> compact()
errors     <- res_t$error  |> compact()



possibly.R — supply a default on error
# possibly() returns the value or .otherwise if it fails — no error wrapper
poss_count <- possibly(count_reads, otherwise = NA_integer_)

counts <- map_int(fastq_paths, poss_count)
# NA where count_reads() threw an error; integer elsewhere

# quietly() also captures warnings / messages / output
quiet_lm <- quietly(stats::lm)
out <- quiet_lm(mpg ~ wt, data = mtcars)
out$result; out$output; out$warnings; out$messages



python_equiv.py — manual or via returns lib
def safely(fn):
    """Mimic purrr::safely — return (result, error) tuple."""
    def wrapper(*a, **kw):
        try:
            return (fn(*a, **kw), None)
        except Exception as e:
            return (None, e)
    return wrapper

safe_run = safely(run_pipeline)
results = [safe_run(s) for s in sample_ids]
ok      = [r for r, e in results if e is None]
errors  = [(s, e) for s, (r, e) in zip(sample_ids, results) if e]

# Or use a Result-monad library like `returns` (more idiomatic):
# from returns.result import safe
# safe(run_pipeline)(sample)  # returns Success/Failure

該用 FP 還是 for-loop？





情境 → 工具對照


Q1→「每個元素獨立 transform」 → map / list comp / purrr::map_*"Each element transforms independently" → map / list comp / purrr::map_*


Q2→「迴圈內依賴上一輪結果 (running sum, accumulator)」 → functools.reduce / purrr::accumulate"Iteration depends on prior result (running sum/accumulator)" → functools.reduce / purrr::accumulate

Q5→「要把函式部分參數凍住」 → functools.partial / purrr::partial"Pre-bind some args" → functools.partial / purrr::partial

Q1. 下列哪個函式是「純函式」？


Q1. Which of the following is a pure function?




A. def add_one(xs): xs.append(0); return [x+1 for x in xs]A. def add_one(xs): xs.append(0); return [x+1 for x in xs]


B. def now_plus(x): return x + time.time()B. def now_plus(x): return x + time.time()


C. def normalize(x, mu, sigma): return (x - mu) / sigmaC. def normalize(x, mu, sigma): return (x - mu) / sigma


D. def save_result(x): with open('out.txt','w') as f: f.write(str(x))D. def save_result(x): with open('out.txt','w') as f: f.write(str(x))




答案 C。A 修改了傳入 list (side effect)；B 依賴當前時間 (非 referential transparent)；D 寫檔案 (side effect)。只有 C 滿足兩條件。


Answer: C. A mutates the input (side effect); B depends on current time (not referentially transparent); D writes a file (side effect). Only C satisfies both purity conditions.







Q2. 對 1000 個樣本跑分析，希望失敗的樣本變成 NA 而不是中斷整批執行——應該用？


Q2. Running analysis on 1000 samples, you want failed samples to become NA instead of halting the batch — use?




A. map(samples, run_pipeline) + 在外層 tryCatchA. map(samples, run_pipeline) wrapped in outer tryCatch


B. map(samples, possibly(run_pipeline, otherwise = NA))B. map(samples, possibly(run_pipeline, otherwise = NA))


C. walk(samples, run_pipeline)C. walk(samples, run_pipeline)


D. reduce(samples, run_pipeline)D. reduce(samples, run_pipeline)




答案 B。possibly() 把失敗時的回傳值設為 .otherwise，整批繼續執行。A 的 tryCatch 在外層只能 catch 第一個錯誤，後面樣本就不跑了。


Answer: B. possibly() supplies .otherwise on failure so the batch keeps going. The outer tryCatch in A would catch the first error and skip the rest.







Q3. 關於「不可變性」在 bioinformatics pipeline 的好處，下列何者 不正確？


Q3. Which is NOT a benefit of immutability in bioinformatics pipelines?




A. 跨執行緒 / 跨 process 共享資料時安全 (無 race condition)A. Safe to share across threads/processes (no race conditions)


B. 同一物件可作為 cache (lru_cache / joblib.Memory) 的 keyB. Object can serve as a cache key (lru_cache / joblib.Memory)


C. 函式不會被「action at a distance」式地修改傳入物件C. Functions can't silently mutate inputs ("action at a distance")


D. 不可變物件總是比可變物件更省記憶體D. Immutable objects always use less RAM than mutable ones




答案 D。不可變性常意味著「修改 = 複製新物件」，反而可能 多用 記憶體 (R 的 copy-on-modify 即如此)。A/B/C 才是真正好處。


Answer: D. Immutability often means "modify = copy" — which can use more memory (R's copy-on-modify is the canonical example). A/B/C are the real benefits.
