為什麼結構化日誌比 print 重要？

當你的 pipeline 跑 3 天後在第 487 個樣本掛掉，下列三個問題決定你能不能修：

  • 哪個樣本？ print(f"processing {sample}") 太多會淹沒在 stdout，且 cluster 通常只保留 stderr / log file。
  • 哪個步驟？ Snakemake / Nextflow job 內部需要結構化 log 才知道是 align / call / annotate 階段失敗。
  • 什麼狀態？ 失敗當下的記憶體用量、處理進度、傳入參數，缺一不可。

用 結構化日誌 (每筆 log 帶 timestamp / level / sample_id / step / extra fields) + 分層的例外，這三個問題在 cluster 上 grep 就能答出。本頁六個主題：例外階層、自訂 Exception、R 的條件系統、結構化日誌 (Python / R)、workflow engine 整合、Sentry 概覽。

When your pipeline crashes at sample #487 after 3 days, three questions decide whether you can fix it:

  • Which sample? print(f"processing {sample}") floods stdout; clusters usually keep only stderr/log files.
  • Which step? Inside a Snakemake/Nextflow job, structured logs tell you whether align / call / annotate failed.
  • What was the state? Memory usage, progress, parameters at the moment of failure — all needed.

With structured logging (every log has timestamp / level / sample_id / step / extra fields) plus a layered exception hierarchy, these become a grep away on a cluster. Topics: exception hierarchy, custom exceptions, R conditions, structured logging (Py/R), workflow integration, Sentry.

💡
核心法則：「log 是給未來的你看的」。三 個月後你只能看到 log file，無法重現當下的環境。當下多寫一行 log.info("loaded %s reads from %s", n, path, extra={"sample":sid})，未來省你三小時 debug。 Core rule: "Logs are written for future-you." Three months later you only have the log file — no rerun of the environment. One extra line now (log.info("loaded %s reads from %s", n, path, extra={"sample":sid})) saves three hours of future debugging.
主題 1

一、Python 例外階層與 raise from

Python 的 Exception 是 class 階層。捕捉 BaseException 等於連 KeyboardInterrupt 都吞掉——絕對不要這樣做。實務上的順序：(1) 捕最具體的，(2) 越上層越廣，(3) raise from 串接因果鏈。

Python's Exception forms a class hierarchy. Catching BaseException swallows even KeyboardInterrupt — never do that. In practice: (1) catch the most specific first, (2) broader cases later, (3) raise from to chain the cause.

try_except.py — order matters: specific first
import json, logging
log = logging.getLogger(__name__)

def load_config(path: str) -> dict:
    try:
        with open(path) as f:
            return json.load(f)
    except FileNotFoundError:
        log.error("Config %s not found", path)
        raise                                # let caller decide
    except json.JSONDecodeError as e:
        log.error("Config %s malformed: %s", path, e)
        raise ValueError(f"Invalid JSON in {path}") from e
    except PermissionError:
        log.error("No read permission on %s", path)
        raise
    # Do NOT add a bare `except Exception` — let unknown errors bubble.
custom_exc.py — domain-specific hierarchy
# Build a small hierarchy for your library's errors
class PipelineError(Exception):
    """Base for all errors raised by this pipeline."""

class InputValidationError(PipelineError):
    """User-supplied input is invalid."""

class QCFailedError(PipelineError):
    """Sample failed QC thresholds."""
    def __init__(self, sample_id: str, n_kept: int, n_min: int):
        self.sample_id = sample_id
        self.n_kept    = n_kept
        self.n_min     = n_min
        super().__init__(
            f"Sample {sample_id} kept {n_kept} cells < threshold {n_min}"
        )

class ExternalToolError(PipelineError):
    """Wraps non-zero exit from samtools / bwa / etc."""

# Users can catch either the umbrella or the specific subtype:
try:
    run_pipeline()
except QCFailedError as e:
    log.warning("QC failed for %s: kept %d / required %d", e.sample_id, e.n_kept, e.n_min)
except PipelineError:
    log.exception("Pipeline failure")
raise_from.py — preserve the cause
def parse_vcf(path):
    try:
        return _low_level_parse(path)
    except (UnicodeDecodeError, IndexError) as e:
        # Wrap into a domain error but KEEP the original traceback:
        raise InputValidationError(f"Cannot parse VCF {path}") from e

# Traceback now shows:
#   IndexError: list index out of range
#   The above exception was the direct cause of the following exception:
#   InputValidationError: Cannot parse VCF /tmp/x.vcf

# Use `raise X from None` to SUPPRESS the cause (rarely needed)
def safe_int(s):
    try:
        return int(s)
    except ValueError:
        raise InputValidationError(f"Not an int: {s!r}") from None
exception_groups.py — Python 3.11+ ExceptionGroup
# Collect failures from many parallel tasks WITHOUT losing any
def run_all(samples):
    errors = []
    results = []
    for s in samples:
        try:
            results.append(run_one(s))
        except PipelineError as e:
            errors.append(e)
    if errors:
        # Python 3.11+ groups exceptions; clients catch via except*
        raise ExceptionGroup("some samples failed", errors)
    return results

try:
    run_all(samples)
except* QCFailedError as eg:
    for e in eg.exceptions:
        log.warning("QC fail: %s", e)
except* ExternalToolError as eg:
    log.error("%d tool failures", len(eg.exceptions))
🚫
反模式：except: pass 與 except Exception: pass。它們吞掉所有錯誤包含 KeyboardInterrupt / SystemExit，讓 pipeline 在錯誤狀態下「靜悄悄繼續跑」——這是 silent corruption。除非你明確要做 fallback，否則永遠加上 log 並 re-raise。 Anti-pattern: except: pass and except Exception: pass. They swallow everything including KeyboardInterrupt / SystemExit, letting the pipeline silently continue in a broken state — silent corruption. Unless you have an explicit fallback, always log and re-raise.

二、R 的條件系統：tryCatch / withCallingHandlers

R 的「條件 (condition)」比 Python exception 更廣，分三種：error、warning、message。兩個處理函式有關鍵差別：

  • tryCatch：unwind handler —— 條件發生時直接跳到 handler，不再回到原本的程式碼。
  • withCallingHandlers：calling handler —— 條件發生時呼叫 handler，handler 結束後原程式碼繼續執行 (除非 handler 用 invokeRestart() 跳走)。warning / message 通常用 calling，error 用 try。

R "conditions" are broader than Python exceptions, in three flavors: error, warning, message. Two handlers differ critically:

  • tryCatch: unwinding handler — on a condition, jump to the handler and do not return to the original code.
  • withCallingHandlers: calling handler — invoke the handler and then resume the original code (unless the handler jumps via invokeRestart()). Calling is typical for warnings/messages, unwinding for errors.
tryCatch.R — unwind: jump out on error
load_counts <- function(path) {
  tryCatch({
    df <- readr::read_tsv(path, show_col_types = FALSE)
    if (nrow(df) == 0) stop("empty file", call. = FALSE)
    df
  },
  # Order from most specific to most general
  fileNotFound = function(c) {
    message("missing file: ", path)
    NULL
  },
  error = function(c) {
    futile.logger::flog.error("Failed to load %s: %s", path, conditionMessage(c))
    NULL
  },
  finally = {
    futile.logger::flog.debug("load_counts done for %s", path)
  })
}
calling.R — log warnings without aborting
# Collect every warning into a list while letting the code keep running
run_with_warnings <- function(samples) {
  warns <- list()
  results <- withCallingHandlers(
    lapply(samples, run_one),
    warning = function(w) {
      warns[[length(warns) + 1]] <<- w
      invokeRestart("muffleWarning")   # suppress; keep running
    },
    message = function(m) {
      futile.logger::flog.info(conditionMessage(m))
      invokeRestart("muffleMessage")
    }
  )
  list(results = results, warnings = warns)
}
custom_cond.R — typed conditions (recommended)
# R also supports custom condition classes — use them like Python custom exceptions
qc_failed <- function(sample_id, n_kept, n_min) {
  structure(
    class = c("qc_failed", "error", "condition"),
    list(
      message = sprintf("%s: kept %d < min %d", sample_id, n_kept, n_min),
      sample_id = sample_id, n_kept = n_kept, n_min = n_min
    )
  )
}

# raise:
stop(qc_failed("SRR123", n_kept = 50, n_min = 100))

# catch the specific class:
tryCatch(
  run_pipeline(),
  qc_failed = function(c) {
    futile.logger::flog.warn("QC fail %s: %d/%d", c$sample_id, c$n_kept, c$n_min)
  }
)

四、Python 結構化日誌：logging / loguru

Python 內建 logging 強大但設定冗長；loguru 是社群最愛的封裝，一行就能輸出 JSON 並支援 rotation。生產環境兩者都用——logging 因為相容性，loguru 用於快速腳本。

Python's stdlib logging is powerful but verbose; loguru is the community-favorite wrapper — JSON output and rotation in one line. Both are used in production: logging for ecosystem compat, loguru for scripts.

stdlib_logging.py — production setup
import logging, sys
from logging.handlers import RotatingFileHandler

# Configure ONCE at app entry point — never inside library code
def setup_logging(level: str = "INFO", log_file: str = "pipeline.log"):
    fmt = logging.Formatter(
        "%(asctime)s | %(levelname)-8s | %(name)s | %(message)s",
        datefmt="%Y-%m-%dT%H:%M:%S"
    )
    root = logging.getLogger()
    root.setLevel(level)
    root.handlers.clear()

    # Console: human-readable
    sh = logging.StreamHandler(sys.stderr)
    sh.setFormatter(fmt); root.addHandler(sh)

    # File: rotating, 10 MB × 5 backups
    fh = RotatingFileHandler(log_file, maxBytes=10_000_000, backupCount=5)
    fh.setFormatter(fmt); root.addHandler(fh)

# In each module:
log = logging.getLogger(__name__)        # inherits root config
log.info("loaded %d reads", n_reads)
log.exception("alignment failed")     # auto-includes traceback

json_log.py — machine-readable for cluster grep
import logging, json, sys

class JsonFormatter(logging.Formatter):
    def format(self, record):
        payload = {
            "ts":     self.formatTime(record, "%Y-%m-%dT%H:%M:%S"),
            "level":  record.levelname,
            "logger": record.name,
            "msg":    record.getMessage(),
        }
        # Pull custom context from record.__dict__ (set via extra=...)
        for k in ("sample_id", "step", "duration_s"):
            if k in record.__dict__:
                payload[k] = record.__dict__[k]
        if record.exc_info:
            payload["traceback"] = self.formatException(record.exc_info)
        return json.dumps(payload)

handler = logging.StreamHandler(sys.stderr)
handler.setFormatter(JsonFormatter())
logging.getLogger().addHandler(handler)
logging.getLogger().setLevel("INFO")

log = logging.getLogger(__name__)
log.info("loaded reads", extra={"sample_id": "SRR123", "step": "load"})
# {"ts":"2026-05-15T12:30:00","level":"INFO","msg":"loaded reads","sample_id":"SRR123","step":"load"}

五、R 結構化日誌：logger / futile.logger / lgr

R 有三個常見 logging 套件：logger (現代、輕量、與 base R 整合)、futile.logger (Bioconductor 套件常用)、lgr (功能最完整、最像 Python logging)。新專案建議 logger 或 lgr。

Three common R logging packages: logger (modern, light, integrates with base R), futile.logger (popular in Bioconductor), lgr (most full-featured, closest to Python's logging). New projects: pick logger or lgr.

六、Workflow Engine 整合 + Sentry

Snakemake 與 Nextflow 都會把 rule / process 的 stdout + stderr 自動收進 log 檔。要讓你的 Python / R log 真的進得去，必須輸出到 stderr（或工作流引擎指定的檔案），絕對不要 用 print() 到 stdout——那會污染 piped output。

Sentry 是雲端 error tracking 服務：未捕捉的例外會自動回報 (含 traceback、環境變數、最近 logs)，多次相同 error 自動 deduplication。即使是學術研究，當你的工具被許多人使用後，Sentry 比 GitHub Issues 更早讓你發現問題。

Snakemake and Nextflow automatically capture each rule/process's stdout + stderr into log files. For your Python/R logs to land there, route to stderr (or the engine-specified log file). Never print() to stdout — that pollutes piped output.

Sentry is a cloud error-tracking service: uncaught exceptions are auto-reported (with traceback, env vars, recent logs), and identical errors are auto-deduplicated. Even for academic tools, once others use your code, Sentry surfaces issues faster than GitHub Issues.

snakemake.py — log per rule
# Snakefile
rule align:
    input:  "data/{sample}.fq.gz"
    output: "out/{sample}.bam"
    log:    "logs/align/{sample}.log"           # <-- Snakemake redirects here
    shell:  """
        bwa mem -t {threads} ref.fa {input} 2> {log} | \\
        samtools sort -@ {threads} -o {output} - 2>> {log}
    """

# In a `run:` block (Python), configure logging to write to log[0]:
rule call_variants:
    input:  "out/{sample}.bam"
    output: "out/{sample}.vcf"
    log:    "logs/call/{sample}.log"
    run:
        import logging
        logging.basicConfig(filename=log[0], level="INFO",
            format="%(asctime)s | %(levelname)s | %(message)s")
        logging.info(f"calling variants for {wildcards.sample}")
        call_variants(input[0], output[0])

📝 自我檢測 (3 題)

Q1. Logger 門檻設為 WARNING。下列哪些 log 會被輸出？(多選思考)

Q1. Logger threshold = WARNING. Which logs are emitted? (think multi)

A. DEBUG + INFO + WARNING + ERROR + CRITICAL 全部A. DEBUG + INFO + WARNING + ERROR + CRITICAL — all of them
B. 只有 WARNING + ERROR + CRITICAL (≥ threshold 才通過)B. Only WARNING + ERROR + CRITICAL (≥ threshold passes)
C. 只有 WARNING (剛好等於門檻)C. Only WARNING (exactly matching the threshold)
D. 只有 DEBUG + INFO (小於門檻才輸出)D. Only DEBUG + INFO (below-threshold are emitted)
答案 B。log level 門檻是 下限——大於等於門檻的 log 通過。WARNING 門檻會放行 WARN / ERROR / CRITICAL，過濾 DEBUG / INFO。
Answer: B. The threshold is a floor — logs at or above the threshold pass. A WARNING threshold lets WARN/ERROR/CRITICAL through and filters DEBUG/INFO.

Q2. 下列哪個是 反模式 (anti-pattern)？

Q2. Which is an anti-pattern?

A. except FileNotFoundError: log.error(...); raiseA. except FileNotFoundError: log.error(...); raise
B. raise InputError(...) from original_eB. raise InputError(...) from original_e
C. 在 entry point 一次性 logging.basicConfig()C. Calling logging.basicConfig() once at entry point
D. try: run() except Exception: passD. try: run() except Exception: pass
答案 D。「裸 except + pass」靜默吞掉所有錯誤，包含 KeyboardInterrupt，導致 pipeline 在錯誤狀態下繼續執行 (silent corruption)。A/B/C 都是正確做法。
Answer: D. "Bare except + pass" silently swallows everything (even KeyboardInterrupt), letting the pipeline continue in a broken state (silent corruption). A/B/C are all correct practices.

Q3. R 中要「收集所有 warning 但讓程式繼續執行」，應該使用？

Q3. In R, to "collect all warnings but keep running," use?

A. withCallingHandlers + invokeRestart("muffleWarning")A. withCallingHandlers + invokeRestart("muffleWarning")
B. tryCatch(warning = function(w) ...)B. tryCatch(warning = function(w) ...)
C. suppressWarnings()C. suppressWarnings()
D. options(warn = -1)D. options(warn = -1)
答案 A。withCallingHandlers 是 calling handler——handler 結束後原程式碼會繼續。tryCatch 是 unwind handler，遇 warning 後就跳出整段 code。suppressWarnings / options(warn=-1) 是把 warning 丟掉，不是收集。
Answer: A. withCallingHandlers uses a calling handler — original code resumes after the handler. tryCatch is unwinding (jumps out on warning). suppressWarnings / options(warn=-1) discard warnings rather than collect them.
