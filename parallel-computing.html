平行運算：把 1 核變 N 核（但要小心 Amdahl）

GIL、threading vs multiprocessing、joblib、future、furrr、BiocParallel、共享記憶體、Dask——選對工具、避開 pickling 與 fork 地雷。

GIL, threading vs multiprocessing, joblib, future, furrr, BiocParallel, shared memory, Dask — pick the right tool and dodge the pickling & fork pitfalls.

原理

GIL：為什麼 Python threading 對 CPU 沒幫助

CPython 的 Global Interpreter Lock (GIL) 在同一時間只允許一個 Python bytecode instruction 執行。意思是：起 4 個 thread 跑 4 個純 Python 數值迴圈，總時間幾乎不變（甚至更慢，因為要切換）。但 GIL 在進入 I/O 系統呼叫（network、disk）或 NumPy/Numba/C extension 時會釋放——這時候 threading 才能真正並行。

  • I/O bound（網路、檔案）：用 threading 或 async（見 A1）。
  • CPU bound（純 Python 計算）：用 multiprocessing（每個 process 各有自己的 GIL）。
  • CPU bound 但底層是 NumPy/Numba/BLAS：本身已釋放 GIL，threading 也有效。

Python 3.13 引入「no-GIL build (PEP 703)」，未來 5 年會逐漸主流——但目前主流 wheel 仍預設有 GIL。R 沒有 GIL，但 R 本身 single-threaded（除非用 BLAS 多執行緒或 fork）。

CPython's Global Interpreter Lock permits only one thread to execute Python bytecode at a time. So 4 threads running 4 pure-Python numeric loops finish in essentially the same wall-clock as 1 thread (often worse, due to context switching). But the GIL is released during I/O syscalls (network, disk) and inside NumPy / Numba / C extensions — there, threading truly parallelizes.

  • I/O bound (network, files): use threading or async (see A1).
  • CPU bound (pure Python compute): use multiprocessing — each process has its own GIL.
  • CPU bound but NumPy/Numba/BLAS underneath: GIL is released internally — threading also works.

Python 3.13 ships an experimental no-GIL build (PEP 703); mainstream over the next 5 years. But today's wheels assume the GIL. R has no GIL but is single-threaded by default (except multi-threaded BLAS and explicit fork).

Reach for concurrent.futures (high-level) or joblib (ecosystem standard) first; use raw multiprocessing only when writing low-level frameworks.

futures.py — high-level fan-out
from concurrent.futures import ProcessPoolExecutor, as_completed
from pathlib import Path

def count_reads(bam_path: Path) -> tuple[str, int]:
    import pysam   # import inside the worker (avoid fork issues)
    with pysam.AlignmentFile(bam_path, "rb") as bam:
        return bam_path.name, bam.count()

bam_files = list(Path("data").glob("*.bam"))

with ProcessPoolExecutor(max_workers=8) as ex:
    futures = {ex.submit(count_reads, p): p for p in bam_files}
    for fut in as_completed(futures):
        name, n = fut.result()
        print(f"{name}: {n:,} reads")

shared_mem.py — avoid copying big arrays
from multiprocessing import shared_memory
from concurrent.futures import ProcessPoolExecutor
import numpy as np

# Big expression matrix that we don't want to pickle 8 times
X = np.load("expr.npy")              # 4 GB
shm = shared_memory.SharedMemory(create=True, size=X.nbytes)
buf = np.ndarray(X.shape, dtype=X.dtype, buffer=shm.buf)
buf[:] = X[:]

def worker(name, shape, dtype, row_range):
    shm_local = shared_memory.SharedMemory(name=name)
    arr = np.ndarray(shape, dtype=dtype, buffer=shm_local.buf)
    sub = arr[row_range[0]:row_range[1]]
    return sub.mean(axis=0)

chunks = np.array_split(np.arange(X.shape[0]), 8)
ranges = [(c[0], c[-1]+1) for c in chunks]
with ProcessPoolExecutor(8) as ex:
    means = list(ex.map(worker, [shm.name]*8, [X.shape]*8, [X.dtype]*8, ranges))

shm.close(); shm.unlink()

主題 2

二、multiprocessing 的三大地雷

① fork vs spawn
Linux 預設 fork（複製整個 process），macOS 14+ 與 Windows 預設 spawn（重新啟動 Python）。在 fork 模式下，CUDA context、OpenMP thread pool、log 檔 handle 都會留在 child 中造成 deadlock 或 corruption。建議 mp.set_start_method("spawn", force=True) 寫死。
Linux defaults to fork (clones the process); macOS 14+ and Windows default to spawn (fresh Python). With fork, CUDA contexts, OpenMP thread pools, and log file handles all carry over and can deadlock or corrupt. Pin mp.set_start_method("spawn", force=True).
② pickling 開銷
傳給 worker 的參數會被 pickle 序列化、複製到子行程、再 unpickle。傳一個 4GB matrix 給 8 個 worker = 32 GB 多餘 RAM。解法：shared_memory、joblib 的 mmap_mode、或把資料放檔案讓 worker 自己讀。
Worker args get pickled, copied to child, then unpickled. Sending a 4 GB matrix to 8 workers = 32 GB of extra RAM. Solutions: shared_memory, joblib's mmap_mode, or let workers read the file themselves.
③ 不可 pickle 的物件
Lambda、closure、開啟的檔案、SQLite connection、socket、CUDA context 都 不能 pickle。傳 lambda 給 Pool.map 會炸。解法：頂層 def 函式；用 cloudpickle（joblib loky backend 已內建）；在 worker 內部才開啟連線。
Lambdas, closures, open files, SQLite connections, sockets, CUDA contexts are NOT picklable. Passing a lambda to Pool.map crashes. Fixes: top-level def functions; cloudpickle (joblib loky uses it); open connections inside the worker.

三、R 的平行運算生態

R 沒有 GIL，但預設 single-threaded。三套主流框架：

  • future + furrr：把 purrr::map 換成 furrr::future_map，一行切換 multisession（spawn）/ multicore（fork，僅 Linux/macOS）/ cluster / remote。
  • foreach + doParallel：傳統做法，foreach(i = 1:N) %dopar% { ... }。
  • BiocParallel：Bioconductor 標準介面，提供 SerialParam、MulticoreParam、SnowParam、BatchtoolsParam（SGE/SLURM/Torque），可在套件作者層級統一抽象。

R has no GIL but is single-threaded by default. Three mainstream frameworks:

  • future + furrr: Replace purrr::map with furrr::future_map; one line switches between multisession (spawn) / multicore (fork; Linux/macOS only) / cluster / remote.
  • foreach + doParallel: Traditional approach, foreach(i = 1:N) %dopar% { ... }.
  • BiocParallel: Bioconductor standard interface providing SerialParam, MulticoreParam, SnowParam, BatchtoolsParam (SGE/SLURM/Torque) — lets package authors abstract over backends.
furrr.R — drop-in parallel purrr
library(furrr)
plan(multisession, workers = 8)   # or: multicore / cluster / remote

results <- future_map(sample_ids, function(sid) {
  obj <- readRDS(paste0("data/", sid, ".rds"))
  obj <- NormalizeData(obj)
  obj <- FindVariableFeatures(obj)
  list(id = sid, var = VariableFeatures(obj))
}, .progress = TRUE, .options = furrr_options(seed = 42))
# seed = 42 → reproducible parallel RNG (L'Ecuyer-CMRG streams)
foreach.R — traditional
library(foreach); library(doParallel)
cl <- makeCluster(8)
registerDoParallel(cl)

results <- foreach(sid = sample_ids,
                   .packages = c("Seurat"),
                   .errorhandling = "pass") %dopar% {
  obj <- readRDS(paste0("data/", sid, ".rds"))
  process(obj)
}
stopCluster(cl)
biocparallel.R — Bioconductor standard
library(BiocParallel)

# Set a default backend for the whole session
register(MulticoreParam(workers = 8))

# Now any Bioconductor function that accepts BPPARAM uses it
res <- bplapply(sample_ids, process_one_sample)

# Or run on a SLURM cluster

                        resources = list(walltime = 3600, memory = "8G"))

四、Amdahl 定律：為什麼不會線性加速

如果你的程式 90% 可平行、10% 必須序列化，無論用多少核心，加速上限是 10×。Amdahl's Law：
S(N) = 1 / (s + (1-s)/N)，其中 s 是序列化比例。
另外還有 實際 overhead：行程啟動、pickling、I/O 競爭、cache thrashing——讓真實曲線比理論更平。

If 90% of your program is parallelizable and 10% must be serial, no number of cores beats 10× speedup. Amdahl's Law:
S(N) = 1 / (s + (1-s)/N), where s is the serial fraction.
On top of that, real-world overhead (process startup, pickling, I/O contention, cache thrashing) flattens the curve further.

該用哪個平行工具？

📝 自我檢測 (3 題)

Q1. 一段純 Python 數值迴圈，用 4 個 thread (ThreadPoolExecutor) 並行，結果幾乎沒變快。為什麼？

Q1. A pure-Python numeric loop, run on 4 threads via ThreadPoolExecutor, gains almost no speedup. Why?

A. ThreadPoolExecutor 有 bugA. ThreadPoolExecutor is buggy
B. CPU 只有 1 核B. CPU has only 1 core
C. GIL 同時只允許一個 thread 跑 Python bytecode；改用 ProcessPoolExecutor 或讓 hot loop 走 NumPy/Numba 釋放 GILC. The GIL allows only one thread to run Python bytecode at a time; use ProcessPoolExecutor or push the hot loop into NumPy/Numba which release the GIL
D. Python 自動偵測並降速D. Python detects threads and throttles
答案 C。純 Python compute 在 thread 中會搶 GIL；要嘛換 process pool，要嘛把熱迴圈交給釋放 GIL 的 C extension（NumPy / Numba @njit(nogil=True)）。
Answer: C. Pure-Python compute serializes on the GIL across threads; either use a process pool, or push the hot loop into a GIL-releasing C extension (NumPy / Numba @njit(nogil=True)).

Q2. Amdahl 定律：序列化部分占 20%、可平行部分占 80%，無限多核心時加速上限是？

Q2. Amdahl: 20% serial, 80% parallel. With infinite cores, the maximum speedup is?

A. 5×（= 1/0.2）A. 5× (= 1/0.2)
B. 80×B. 80×
C. 無限大C. Infinite
D. 由 CPU 廠商決定D. Depends on the CPU vendor
答案 A。當 N→∞，S = 1 / (s + (1-s)/N) → 1/s = 1/0.2 = 5。「最後那 20% 不能平行」決定了天花板。
Answer: A. As N→∞, S = 1/(s + (1-s)/N) → 1/s = 1/0.2 = 5. The unparallelizable 20% sets the ceiling.

Q3. 你把一個 4GB NumPy 矩陣傳給 ProcessPoolExecutor 上的 8 個 worker，記憶體爆增到 36GB。最佳解法？

Q3. You hand a 4 GB NumPy matrix to 8 workers via ProcessPoolExecutor and RAM blows up to 36 GB. Best fix?

A. 多買 RAMA. Buy more RAM
B. 用 multiprocessing.shared_memory 或 joblib 的 mmap_mode，讓 8 worker 共讀同一塊B. Use multiprocessing.shared_memory or joblib's mmap_mode so workers share one buffer
C. 改用 threadingC. Switch to threading
D. 把矩陣切成 8 個小檔再傳D. Split into 8 small files
答案 B。傳遞 NumPy array 給 worker 預設會 pickle → 複製 → unpickle，8 個 worker 就是 8 份複本。shared_memory 與 memmap 都是 single-copy 解法。Threading 行不通（GIL）；切檔反而更慢且仍會複製。
Answer: B. Passing arrays to workers defaults to pickle → copy → unpickle; 8 workers = 8 copies. shared_memory and memmap give single-copy access. Threading won't help (GIL); splitting files makes I/O worse and still copies.
