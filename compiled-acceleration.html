向量化與編譯加速：把 hot loop 變成原生速度

NumPy 向量化、Numba JIT、Cython、Rcpp、data.table——把熱迴圈從 Python/R 解譯器的牢籠裡解救出來。

NumPy vectorization, Numba JIT, Cython, Rcpp, data.table — free your hot loops from the Python / R interpreter cage.

原理

為什麼 Python / R 的迴圈這麼慢？

純 Python 的 for i in range(N): 每次迭代都要：(1) 解譯 bytecode，(2) 處理 reference count，(3) 把每個整數包成 PyObject（24 bytes 起跳）。一個對 1 億元素做加法的 loop，純 Python ≈ 20 秒，NumPy ≈ 0.1 秒，Numba JIT ≈ 0.05 秒，C ≈ 0.04 秒。

解套方法可分成兩類：

  • 向量化 (vectorize)：用 NumPy / data.table / 矩陣運算把 loop 推給底層 C/Fortran/BLAS。
  • 編譯 (compile)：把 Python/R 程式碼編成機器碼——Numba 用 LLVM，Cython 走 C 編譯器，Rcpp 直接編 C++。

A plain Python for i in range(N): pays three taxes per iteration: (1) interpreting bytecode, (2) reference-count bookkeeping, (3) boxing each int into a PyObject (24 bytes minimum). For adding 100M numbers: pure Python ≈ 20s, NumPy ≈ 0.1s, Numba JIT ≈ 0.05s, C ≈ 0.04s.

Two strategies:

  • Vectorize: Push the loop down to C / Fortran / BLAS via NumPy / data.table / matrix ops.
  • Compile: Translate Python/R into machine code — Numba uses LLVM, Cython invokes a C compiler, Rcpp compiles C++ directly.
💡
嘗試順序：向量化 → Numba @njit → 平行化（Block B3）→ Cython / Rcpp / C 擴充。每往下一階複雜度都顯著上升，加速比卻只是「再快幾倍」。多數時候停在 Numba 就夠。 Try-order: Vectorize → Numba @njit → parallelize (B3) → Cython / Rcpp / C. Each step adds significant complexity but only modest extra speedup. Most projects stop at Numba.
主題 1

一、NumPy 向量化：第一道防線

絕大多數「Python 太慢」其實是「沒向量化」。常見技巧：用 broadcasting 取代外層迴圈，用 np.where / boolean masking 取代 if，用 np.einsum / @ 做矩陣運算，用 np.add.reduceat 做 segmented sum。

"Python is slow" usually means "not vectorized." Common moves: broadcasting instead of outer loops, np.where/boolean masks instead of if, np.einsum/@ for matrix products, np.add.reduceat for segmented sums.

slow.py — 2.4 s on 1M cells
import numpy as np

def cpm_loop(counts: np.ndarray) -> np.ndarray:
    """Counts-per-million normalization, naive."""
    n_cells, n_genes = counts.shape
    out = np.zeros_like(counts, dtype=np.float32)
    for i in range(n_cells):
        total = 0
        for j in range(n_genes):
            total += counts[i, j]
        for j in range(n_genes):
            out[i, j] = counts[i, j] / total * 1e6
    return out
fast.py — 12 ms on the same matrix (200× faster)
import numpy as np

def cpm(counts: np.ndarray) -> np.ndarray:
    totals = counts.sum(axis=1, keepdims=True)   # shape (n,1)
    return (counts / totals * 1e6).astype(np.float32)

# Key tricks:
# - sum(axis=1, keepdims=True) gives (n,1) which broadcasts over (n,m)
# - everything runs in C; no Python-level loop
advanced.py — einsum + reduceat
import numpy as np

# Gene × sample correlation, vectorized
def gene_gene_corr(X: np.ndarray) -> np.ndarray:
    Xc = X - X.mean(axis=1, keepdims=True)
    norms = np.linalg.norm(Xc, axis=1, keepdims=True)
    Xn = Xc / norms
    return Xn @ Xn.T              # BLAS gemm under the hood

# Segmented sum: aggregate read counts per gene window
def window_counts(positions: np.ndarray, bin_edges: np.ndarray):
    idx = np.searchsorted(bin_edges, positions)
    return np.bincount(idx, minlength=len(bin_edges))
⚠️
陷阱：不要把 NumPy array 用 Python loop 走訪——那等於失去向量化。for x in arr: ... 比 for x in arr.tolist(): ... 還慢，因為每次都要做 unboxing。 Pitfall: Don't iterate a NumPy array with a Python loop — you've lost the vectorization. for x in arr: ... is even slower than for x in arr.tolist(): ... because each step unboxes a scalar.
主題 2

二、Numba：一個 decorator 換 50–200× 速度

Numba 透過 LLVM 把 Python 函式 JIT 編譯成機器碼。對純數值迴圈（k-mer counting、edit distance、smith-waterman 內層）通常能取得 50–200× 加速；加 parallel=True 可再乘上核心數。

Numba JIT-compiles Python functions to machine code via LLVM. For pure-numeric loops (k-mer counting, edit distance, Smith-Waterman inner loops) it typically achieves 50–200× speedup; parallel=True multiplies by core count.

njit.py — k-mer counting
from numba import njit
import numpy as np

@njit(cache=True)   # 'n'o-Python mode; cache compiled binary to disk
def count_kmers(seq: np.ndarray, k: int) -> np.ndarray:
    """seq is uint8 (A=0,C=1,G=2,T=3); returns 4**k length array."""
    n_kmers = 4 ** k
    counts = np.zeros(n_kmers, dtype=np.int64)
    for i in range(len(seq) - k + 1):
        idx = 0
        for j in range(k):
            idx = idx * 4 + seq[i + j]
        counts[idx] += 1
    return counts

# Pure-Python: ~14 s for 1M bp, k=6
# Numba njit: ~80 ms — 175× faster
parallel.py — parallel=True over rows
from numba import njit, prange
import numpy as np

@njit(parallel=True, fastmath=True, cache=True)
def pairwise_euclidean(X: np.ndarray) -> np.ndarray:
    n, d = X.shape
    D = np.zeros((n, n))
    for i in prange(n):                    # parallel range
        for j in range(i + 1, n):
            s = 0.0
            for k in range(d):
                diff = X[i, k] - X[j, k]
                s += diff * diff
            D[i, j] = D[j, i] = np.sqrt(s)
    return D
# 8-core machine: ~7× extra on top of njit (not full 8× due to overhead)
vectorize.py — element-wise ufunc
from numba import vectorize, float32, float64

@vectorize([float32(float32), float64(float64)], target="parallel")
def phred_to_prob(q):
    return 10.0 ** (-q / 10.0)

# Now use as if it were a NumPy ufunc
probs = phred_to_prob(qual_array)         # broadcasts, multi-threaded

#   - pandas DataFrames / Series

from numba import njit, types
from numba.typed import Dict

@njit
def unique_counts(arr):
    d = Dict.empty(key_type=types.int64, value_type=types.int64)
    for x in arr:
        d[x] = d.get(x, 0) + 1
    return d

三、Cython：把熱迴圈寫成 C 速度

Cython 是 Python 的超集，允許你用 cdef 宣告 C 型別變數、cdef class 寫 C-extension class、memoryview 安全地存取 NumPy buffer。pysam、scikit-learn、pandas 內部都大量使用 Cython。比 Numba 多了 build 步驟，但對需要分發 binary wheel 的 Bioconductor/PyPI 套件是必要的。

Cython is a Python superset that lets you declare C-typed variables with cdef, write C-extension classes via cdef class, and safely access NumPy buffers via memoryviews. pysam, scikit-learn, and pandas all use Cython internally. It has a build step (unlike Numba), but is the standard for PyPI/Bioconductor packages that need pre-compiled binary wheels.

edit_dist.pyx — Levenshtein with memoryviews
# cython: boundscheck=False, wraparound=False, cdivision=True
cimport numpy as cnp
import numpy as np

def edit_distance(bytes s1, bytes s2) -> int:
    cdef Py_ssize_t i, j, n = len(s1), m = len(s2)
    cdef cnp.int32_t[:, ::1] dp = np.zeros((n+1, m+1), dtype=np.int32)
    cdef int cost
    for i in range(n+1): dp[i, 0] = i
    for j in range(m+1): dp[0, j] = j
    for i in range(1, n+1):
        for j in range(1, m+1):
            cost = 0 if s1[i-1] == s2[j-1] else 1
            dp[i, j] = min(dp[i-1, j] + 1,
                           dp[i, j-1] + 1,
                           dp[i-1, j-1] + cost)
    return dp[n, m]
setup.py — build the extension
from setuptools import setup
from Cython.Build import cythonize
import numpy

setup(
    ext_modules=cythonize("edit_dist.pyx", language_level="3"),
    include_dirs=[numpy.get_include()],
)

# Build:  python setup.py build_ext --inplace
# Use:    from edit_dist import edit_distance
# Typical speedup vs pure Python: 100–500×

四、R 端：data.table 與 Rcpp

R 的兩條主要加速路徑：(1) 用 data.table 取代 data.frame——by-reference 修改、key-indexed join、底層 C 實作；(2) 用 Rcpp 寫 C++ 函式直接呼叫，RcppArmadillo 給線性代數類似 NumPy 的語法。

Two main R speedup paths: (1) replace data.frame with data.table — by-reference mutation, key-indexed joins, C internals; (2) write C++ via Rcpp; RcppArmadillo gives NumPy-like linear-algebra syntax.

datatable.R — 10–100× faster than dplyr for big tables
library(data.table)

# Read 5GB BED-like file in ~12 s (vs ~3 min for read.table)
dt <- fread("variants.tsv.gz")

# In-place mutate — NO copy
dt[, gc := (G + C) / (A + T + G + C)]

# Group-by aggregation — fully C-backed
result <- dt[qual > 30, .(n = .N, mean_dp = mean(DP)), by = chrom]

# Key-indexed join: pre-sort once, then binary search
setkey(dt, chrom, pos)
hits <- dt[regions, on = .(chrom, pos >= start, pos <= end)]
rcpp_inline.R — compile C++ inline
library(Rcpp)

cppFunction('
  NumericVector cpp_rolling_mean(NumericVector x, int w) {
    int n = x.size();
    NumericVector out(n - w + 1);
    double s = 0;
    for (int i = 0; i < w; ++i) s += x[i];
    out[0] = s / w;
    for (int i = w; i < n; ++i) {
      s += x[i] - x[i - w];
      out[i - w + 1] = s / w;
    }
    return out;
  }
')

# 10M-element vector, w=100:
#   R for-loop:        ~9.0 s
#   zoo::rollmean:     ~1.2 s
#   cpp_rolling_mean:  ~0.08 s  (~110×)
arma.cpp — RcppArmadillo for matrix math
// PCA-style eigen decomp using Armadillo
#include <RcppArmadillo.h>
// [[Rcpp::depends(RcppArmadillo)]]
// [[Rcpp::export]]
arma::mat cpp_pca(const arma::mat& X, int k) {
    arma::mat Xc = X.each_row() - arma::mean(X, 0);
    arma::mat U, V;
    arma::vec s;
    arma::svd_econ(U, s, V, Xc);
    return U.cols(0, k - 1) * arma::diagmat(s.head(k));
}
// sourceCpp("pca.cpp") in R; calls into OpenBLAS, similar perf to NumPy

📝 自我檢測 (3 題)

Q1. 為什麼 for x in numpy_array: 比 numpy_array.sum() 慢很多？

Q1. Why is for x in numpy_array: much slower than numpy_array.sum()?

A. NumPy 用 SIMD，Python 沒A. NumPy uses SIMD, Python does not
B. Python loop 每次都要 unbox/boxing PyObject、跑 bytecode；.sum() 整個 loop 跑在 C 層級B. Python loop boxes/unboxes a PyObject per step and runs bytecode; .sum() runs the whole loop in C
C. .sum() 用 GPUC. .sum() uses the GPU
D. NumPy 自動平行化D. NumPy auto-parallelizes
答案 B。核心開銷是 Python 解譯器與 PyObject overhead。SIMD 雖也有幫助，但量級遠不及把 loop 整段下放到 C。
Answer: B. The dominant cost is interpreter + PyObject overhead. SIMD helps too, but pushing the loop into C is far more impactful.

Q2. 你的 Numba @njit 函式第一次呼叫比第二次慢 200 倍，為什麼？

Q2. Your @njit function is 200× slower on the first call than subsequent ones. Why?

A. Numba 有 bugA. Numba has a bug
B. Cache missB. CPU cache miss
C. 第一次呼叫包含 JIT 編譯時間；可加 cache=True 把編好的二進位存起來C. The first call includes JIT compile time; add cache=True to persist the binary
D. GIL 釋放D. GIL release
答案 C。Numba lazy-compile：見到第一組型別才動工。Benchmark 時記得「先 warm-up 一次」再計時；分發時用 AOT 編譯或 cache=True。
Answer: C. Numba compiles lazily on first type signature seen. Always warm up before benchmarking; ship with AOT compilation or cache=True.

Q3. 你要寫一個會被 1000 個下游使用者 pip install 的工具，含一段純數值 hot loop。最合適的加速方案？

Q3. You're shipping a tool to 1000 users via pip install, containing a pure-numeric hot loop. Best accelerator?

A. 純 Python，反正使用者電腦快A. Pure Python; users have fast laptops
B. Numba JIT，第一次跑慢沒關係B. Numba JIT — first run slow is fine
C. 要求使用者裝 R 並改用 RcppC. Require users to install R and use Rcpp
D. Cython + 預先在 CI 編好 manylinux/macOS/Windows wheelsD. Cython + pre-built manylinux/macOS/Windows wheels in CI
答案 D。Numba 對 end-user 而言有 LLVM 重依賴、首次延遲、且編譯失敗時很難 debug。Cython 預編 wheel 是 pysam/scikit-learn/numpy 等大套件的標準做法。
Answer: D. Numba imposes a heavy LLVM dependency on end users, first-call latency, and tricky failure modes. Pre-compiled Cython wheels are the standard approach (pysam, scikit-learn, NumPy all do this).
