套件開發與發布：PyPI、CRAN、Bioconductor

從 scripts/ 到 pip install mypkg 或 install.packages("mypkg")——pyproject.toml、uv、src/ layout、Sphinx、PyPI trusted publishing、usethis、pkgdown、BiocCheck。

From scripts/ to pip install mypkg or install.packages("mypkg") — pyproject.toml, uv, src/ layout, Sphinx, PyPI trusted publishing, usethis, pkgdown, BiocCheck.

動機

為什麼要做成套件？

把分析放在 scripts/ 資料夾然後互相 import 看似省事，但會碰到：

  • 無版本可指定：「請用 v1.2.3 重現我的圖」——可是你只有 GitHub main。
  • 依賴混亂：同事裝完發現要手動補 5 個包，每個版本還不一定相容。
  • 不能 cite：審稿要 DOI/Zenodo 連結，scripts 沒辦法給。
  • 無法重用：下一個 paper 要把 QC 函式抽出來，發現它 import 了一個寫死路徑的 config。

本章六個主題：Python 現代打包、uv 與 versioning、文件 (Sphinx + pkgdown)、PyPI 發布、R 套件流程 (usethis / R CMD check)、Bioconductor / R-universe。

Keeping analyses in scripts/ with cross-imports feels light but breaks down:

  • No version pin: "Reproduce with v1.2.3" — but you only have GitHub main.
  • Dependency chaos: Colleagues install and manually patch 5 packages, hoping versions align.
  • Can't cite: Reviewers ask for a DOI/Zenodo link — scripts can't give one.
  • Can't reuse: Next paper, you try to lift the QC function — turns out it imports a hard-coded config path.

The six topics: Modern Python packaging, uv + versioning, Docs (Sphinx + pkgdown), PyPI publishing, R package workflow (usethis / R CMD check), Bioconductor / R-universe.

💡
口訣：「能 import 的、能 pip install 的、能 cite DOI 的——這是 reproducible research 的三層門檻。」 Mantra: "Importable, pip-installable, cite-by-DOI — the three tiers of reproducible research."
主題 1

一、Python 現代打包

2025 標準作法：所有 metadata 集中在 pyproject.toml (PEP 621)，build backend 可選 hatchling (推薦)、setuptools、poetry-core、flit。源碼放在 src/mypkg/ 避免 import shadow 問題。

2025 default: all metadata in pyproject.toml (PEP 621); build backend is hatchling (recommended), setuptools, poetry-core, or flit. Source lives in src/mypkg/ to prevent import-shadowing.

src/scqc/cli.py — entry point
import argparse, sys
from .qc import filter_cells
from .io import read_anndata, write_anndata
from . import __version__

def main(argv: list[str] | None = None) -> int:
    p = argparse.ArgumentParser(prog="scqc")
    p.add_argument("-V", "--version", action="version", version=__version__)
    p.add_argument("input");  p.add_argument("output")
    p.add_argument("--min-counts", type=int, default=500)
    args = p.parse_args(argv)

    adata = read_anndata(args.input)
    adata = filter_cells(adata, min_counts=args.min_counts)
    write_anndata(adata, args.output)
    return 0

if __name__ == "__main__":
    sys.exit(main())

二、uv 與版本控制

uv 是 Astral 用 Rust 寫的新一代 Python 套件管理工具，速度比 pip/poetry 快 10-100×，整合 lock file、virtualenv、Python interpreter 安裝。版本號用 SemVer (MAJOR.MINOR.PATCH)，並用 hatch-vcs 從 git tag 自動讀取。

uv is Astral's Rust-written next-gen Python package manager — 10-100× faster than pip/poetry, integrates lock file, virtualenv, and Python interpreter install. Versions follow SemVer (MAJOR.MINOR.PATCH); hatch-vcs derives them from git tags.

uv-workflow.sh
# Install (one-time)
curl -LsSf https://astral.sh/uv/install.sh | sh

# In a new package directory
uv init --lib scqc            # scaffold src/scqc
cd scqc
uv add anndata scanpy pydantic
uv add --dev pytest ruff mypy

# Run something inside the venv (auto-created)
uv run pytest
uv run python -c "import scqc; print(scqc.__version__)"

# Sync from lock file (CI)
uv sync --all-extras

# Build wheel + sdist
uv build           # or: python -m build

# Publish (uses trusted publishing if configured)
uv publish
semver.txt — Semantic Versioning
MAJOR.MINOR.PATCH        (e.g. 2.4.1)

  PATCH 0 → 1     bug fixes only, no API change
                  e.g. 1.2.0 → 1.2.1

  MINOR 4 → 5     new feature, backward-compatible
                  e.g. 1.2.0 → 1.3.0

  MAJOR 1 → 2     breaking API change
                  e.g. 1.2.0 → 2.0.0

Pre-release & build metadata (PEP 440):
  1.0.0a1         alpha
  1.0.0b1         beta
  1.0.0rc1        release candidate
  1.0.0.dev3      dev snapshot
  1.0.0.post1     post-release fix
  1.0.0+gpu       local build identifier

Bioconductor rule (separate):
  Devel branch  : x.y.z where y is odd  (1.3.x = devel)
  Release branch: x.y.z where y is even (1.4.x = stable)
release.sh — cut a new version
# 1. Update CHANGELOG.md (or use towncrier / git-cliff)
git-cliff --tag v0.4.0 --output CHANGELOG.md

# 2. Commit changelog
git add CHANGELOG.md
git commit -m "chore: prepare v0.4.0"

# 3. Tag — hatch-vcs reads this
git tag -a v0.4.0 -m "v0.4.0: add doublet detection"
git push origin main --tags

# 4. GitHub Actions workflow (release.yml) auto-builds & publishes to PyPI
#    via OIDC trusted publishing (no API token in repo!)

Python: Sphinx + MyST (Markdown) + furo / pydata-sphinx-theme + sphinx-autoapi. R: pkgdown. Both free-host on ReadTheDocs / Cloudflare Pages / GitHub Pages.

四、PyPI 發布

2024 後 PyPI 推薦 Trusted Publishing——GitHub Actions 透過 OIDC token 認證，不再需要在 repo 存 API key (避免外洩風險)。流程：(1) PyPI 後台註冊 trusted publisher，(2) GitHub workflow 加 permissions: id-token: write。

Since 2024 PyPI recommends Trusted Publishing — GitHub Actions authenticates via OIDC token, no API key in the repo (preventing leaks). Steps: (1) register a trusted publisher in PyPI settings; (2) add permissions: id-token: write to the workflow.

.github/workflows/publish.yml
name: Publish to PyPI
on:
  release:
    types: [published]      # fires when you click "Publish release" on GitHub

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: {fetch-depth: 0}        # for hatch-vcs to see tags
      - uses: astral-sh/setup-uv@v3
      - run: uv build
      - uses: actions/upload-artifact@v4
        with: {name: dist, path: dist/}

  publish:
    needs: build
    runs-on: ubuntu-latest
    environment: pypi                 # GitHub env with PyPI binding
    permissions:
      id-token: write                 # required for OIDC trusted publishing
    steps:
      - uses: actions/download-artifact@v4
        with: {name: dist, path: dist/}
      - uses: pypa/gh-action-pypi-publish@release/v1
        # No password needed! OIDC handles auth.
testpypi.sh — TestPyPI dry-run before real release
# TestPyPI is a sandbox at https://test.pypi.org
# Useful for verifying metadata, README rendering, install flow.

uv build
uv publish --publish-url https://test.pypi.org/legacy/

# Try installing from TestPyPI
pip install --index-url https://test.pypi.org/simple/ \
            --extra-index-url https://pypi.org/simple/ \
            scqc

# Once happy: tag and let GitHub Actions publish to real PyPI.
⚠️
注意：PyPI 不允許「刪除版本後重複使用版本號」。如果你 publish 了 0.1.0，那個版本號永遠用掉了——即使刪除也不能 republish。永遠先用 TestPyPI 試跑。 Warning: PyPI does not allow "delete version then reuse the number." Once 0.1.0 is published, that number is permanently burned — even after deletion. Always test on TestPyPI first.

R/qc.R — roxygen2 docstring
#' Filter low-quality cells
#'
#' Removes cells whose total counts fall below `min_counts`.
#'
#' @param sce A `SingleCellExperiment` with `colSums(counts(sce))` available.
#' @param min_counts Numeric; minimum total UMIs per cell. Default 500.
#' @return A filtered `SingleCellExperiment`.
#' @examples
#' sce <- make_tiny_sce()
#' filter_cells(sce, min_counts = 100)
#' @export
filter_cells <- function(sce, min_counts = 500) {
  stopifnot(min_counts >= 0)
  keep <- colSums(SummarizedExperiment::assay(sce, "counts")) >= min_counts
  sce[, keep]
}

主題 6

六、Bioconductor 與 R-universe

Bioconductor 是生醫資訊 R 套件的官方倉庫 (CRAN 旁的兄弟)，門檻高 (要過 BiocCheck、有 vignettes、用 S4 物件、半年一個版本)，但有「同個 Bioc 版本所有套件都會被一起 build/test」的優點，依賴一致性極高。R-universe 是 rOpenSci 提供的個人/組織級 CRAN-like 服務，幾分鐘上架，適合介於 GitHub 與 CRAN 之間的快速發布。

Bioconductor is the official biomedical R package repository (CRAN's sibling). High bar (BiocCheck, vignettes, S4 objects, twice-yearly release), but every package in a Bioc release is built and tested together — exceptional dependency consistency. R-universe (by rOpenSci) is a personal/org-level CRAN-like service — up in minutes; great for between-GitHub-and-CRAN distribution.

📝 自我檢測 (3 題)

Q1. 為什麼 src/ layout 比 flat layout 安全？

Q1. Why is src/ layout safer than flat layout?

A. src/ 跑得比較快A. src/ runs faster
B. PyPI 規定必須用 src/B. PyPI requires src/
C. 避免「測試從專案根目錄 import 到沒安裝版本」的隱性 bugC. Prevents tests accidentally importing the un-installed copy from the project root
D. src/ 比較容易上 Git LFSD. src/ works better with Git LFS
答案 C。Flat layout (套件目錄與 tests/ 同層) 在跑 pytest 時可能直接 import 沒裝過的本機程式，遮蔽你 pip install 後該測的版本；src/ 強迫先 install 再測，更接近終端使用者體驗。
Answer: C. Flat layout (package next to tests/) lets pytest pick up the un-installed source, hiding what users would actually get after pip install. src/ forces install-then-test, mirroring the real install path.

Q2. Trusted Publishing 相對於傳統 API token 的主要安全優點？

Q2. Main security advantage of Trusted Publishing over API tokens?

A. 不需在 repo / secrets 保存長期 token，靠 OIDC 短期 token 驗證A. No long-lived token in repo/secrets — OIDC short-lived token authenticates
B. 上傳速度比較快B. Faster upload
C. 只能對 GitHub 私有 repo 使用C. Only works on private GitHub repos
D. 可以一次發布到 PyPI + BioconductorD. Publishes to PyPI + Bioconductor at once
答案 A。傳統 API token 一旦外洩 (push 到公開 repo、log 印出)，攻擊者可長期惡意 publish。OIDC trusted publishing 用短期 token，過期即無效；且 PyPI 端綁定特定 GitHub repo + workflow，攻擊面大幅縮小。
Answer: A. Long-lived API tokens, once leaked (pushed to a public repo, printed in logs), let attackers publish indefinitely. OIDC tokens are short-lived and PyPI binds them to a specific GitHub repo + workflow — drastically smaller attack surface.

Q3. 下列關於 Bioconductor 的描述何者正確？

Q3. Which statement about Bioconductor is correct?

A. 提交後幾分鐘自動上架 (像 PyPI)A. Auto-publishes within minutes (like PyPI)
B. 半年一個 release；所有套件被一起 build/test，依賴一致性高B. Twice-yearly release; all packages built/tested together, strong dependency consistency
C. 不允許用 S4 物件C. Forbids S4 classes
D. 比 CRAN 標準寬鬆D. Has looser standards than CRAN
答案 B。Bioconductor 半年一個 release (4 月 / 10 月)，devel 版本平行進行。提交流程嚴格 (BiocCheck 比 R CMD check 嚴)，要有 vignette、用 S4、有 biocViews。代價是嚴謹度——這也是它成為生醫資訊金標準的原因。
Answer: B. Bioconductor releases twice yearly (April / October), with a parallel devel branch. Submission is strict (BiocCheck is harsher than R CMD check) — vignette required, S4 expected, biocViews mandatory. The strictness is exactly why Bioc is the bioinformatics gold standard.
