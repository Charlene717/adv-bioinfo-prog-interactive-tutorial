HPC 與雲端：把分析推上叢集與雲

當你的分析從「我的工作站」成長到「1000 個樣本、需要 200 顆 CPU」，HPC 與雲端是必經之路。本章涵蓋 SLURM、AWS Batch、GCP、Terra，以及為什麼成本可以差 10 倍。

When analyses outgrow your workstation — 1000 samples needing 200 CPUs — HPC and cloud become inevitable. This chapter covers SLURM, AWS Batch, GCP, Terra, and why costs can differ by 10×.

動機

HPC vs 雲端：何時用哪個？

機構級 HPC (大學 / 醫院 / 研究所) 通常已有 SLURM 叢集——對於大多數研究專案，HPC 是 最便宜 (對使用者免費或低成本) 且 最快上手 的選擇。但 HPC 有兩個限制：

  • 排隊時間：高負載時 job 可能排隊 12-48 小時。
  • 固定配額：機構配額用完就必須等。

雲端 (AWS / GCP / Azure) 適合：突發大量運算 (burst)、機構沒有 HPC、需要 GPU / 特殊硬體、與外部協作者共享環境、需要 絕對的彈性 (按秒計費)。但雲端有兩大坑：egress 費 (從雲端下載資料貴) 與 storage 累積 (S3 月費)。

Institutional HPC (universities, hospitals, research institutes) usually have SLURM clusters — for most research projects, HPC is the cheapest (free or low-cost to users) and fastest to start. But HPC has two limits:

  • Queue time: jobs may wait 12-48h under heavy load.
  • Fixed quota: once your allocation runs out, you wait.

Cloud (AWS/GCP/Azure) fits: burst-scale compute, institutions without HPC, GPU/specialty hardware needs, external collaborators, true elasticity (per-second billing). Two main traps: egress fees (downloading from cloud is expensive) and storage creep (S3 monthly bills).

HPC

SLURM 必備 — 提交、監控、debug

SLURM (Simple Linux Utility for Resource Management) 是 HPC 最廣用的排程器。掌握 5 個指令就能應付 80% 場景：sbatch (提交)、squeue (排隊狀態)、scancel (取消)、sacct (歷史 / 帳務)、sinfo (節點資源)。

SLURM (Simple Linux Utility for Resource Management) is the most widely used HPC scheduler. Five commands cover 80% of cases: sbatch (submit), squeue (queue), scancel (cancel), sacct (history/accounting), sinfo (node info).

job.sbatch — single job
#!/bin/bash
#SBATCH --job-name=bwa-align
#SBATCH --partition=compute
#SBATCH --cpus-per-task=16
#SBATCH --mem=32G
#SBATCH --time=04:00:00
#SBATCH --output=logs/%x-%j.out
#SBATCH --error=logs/%x-%j.err
#SBATCH --mail-type=END,FAIL
#SBATCH --mail-user=user@univ.edu

module load bwa/0.7.17 samtools/1.19
source ~/miniforge3/etc/profile.d/conda.sh && conda activate align

bwa mem -t $SLURM_CPUS_PER_TASK ref.fa R1.fq.gz R2.fq.gz \
  | samtools sort -@ $SLURM_CPUS_PER_TASK -o sample.bam -
array.sbatch — 200 samples in one submission
#!/bin/bash
#SBATCH --array=1-200%50          # 200 tasks, max 50 concurrent
#SBATCH --cpus-per-task=8
#SBATCH --mem=16G
#SBATCH --time=02:00:00
#SBATCH --output=logs/%A_%a.out

SAMPLE=$(sed -n "${SLURM_ARRAY_TASK_ID}p" samples.tsv | cut -f1)
echo "Processing $SAMPLE on $(hostname)"
fastp -i raw/${SAMPLE}_R1.fq.gz -I raw/${SAMPLE}_R2.fq.gz \
      -o trim/${SAMPLE}_R1.fq.gz -O trim/${SAMPLE}_R2.fq.gz \
      -w $SLURM_CPUS_PER_TASK
monitor.sh — daily routine
# My jobs right now
squeue -u $USER

# Why is my job not running?
squeue -u $USER -o "%i %P %j %T %R"
# R(easons): Priority, Resources, ReqNodeNotAvail, AssocGrpCPULimit

# Cancel everything
scancel -u $USER

# Cancel one job
scancel 12345678

# How much CPU / memory did finished jobs use? (key for right-sizing!)
sacct -j 12345678 --format=JobID,JobName,State,Elapsed,MaxRSS,ReqMem,AllocCPUS

# Resource info — which partitions / nodes available?
sinfo -o "%P %a %l %D %C"   # partition, avail, time, nodes, cpus

常見錯誤 — over-requesting：申請 64 CPU / 256 GB 但實際用 4 CPU / 8 GB，會被排程器降低優先序，且浪費機構配額。提交後用 sacct --format=MaxRSS,AllocCPUS,Elapsed 確認真實用量，下次調低申請。 Common mistake — over-requesting: Asking for 64 CPU / 256 GB but using only 4 / 8 GB lowers your scheduling priority and wastes institutional allocation. Run sacct --format=MaxRSS,AllocCPUS,Elapsed after jobs to right-size next time.

實作

AWS Batch — 突發大量運算

AWS Batch 把「我要跑 1000 個獨立 job」這件事抽象化：你給它 job definition (Docker image + 指令 + 資源)，它自動開機器 (含 spot 機器)、執行、輸出 log、關機。生物學上常見配置：spot 機器跑 trim/align，on-demand 跑 variant calling (避免被中斷)。

AWS Batch abstracts "run 1000 independent jobs": you provide a job definition (Docker image + command + resources), and it provisions machines (including spot), runs jobs, captures logs, and tears down. Common bio pattern: spot instances for trim/align, on-demand for variant calling (avoid interruption).

submit_array.py — fan out 200 samples
import boto3, json
batch = boto3.client("batch", region_name="us-east-1")

samples = open("samples.tsv").read().splitlines()

for i, sid in enumerate(samples):
    resp = batch.submit_job(
        jobName=f"align-{sid}",
        jobQueue="genomics-spot-queue",
        jobDefinition="bwa-align:5",
        containerOverrides={
            "command": ["bash", "/work/align.sh", sid],
            "environment": [{"name": "SAMPLE", "value": sid}],
            "resourceRequirements": [
                {"type": "VCPU",   "value": "16"},
                {"type": "MEMORY", "value": "32768"},
            ],
        },
        retryStrategy={"attempts": 3},   # spot may be reclaimed
    )
    print(resp["jobId"], sid)

cost_estimate.py — back-of-envelope
# 1000-sample WGS variant calling, AWS us-east-1, 2026 pricing (approx)
samples       = 1000
align_h       = 2.0    # bwa-mem2 16 vcpu / 32GB
call_h        = 1.5    # gatk HaplotypeCaller
spot_price_h  = 0.20   # c6i.4xlarge spot — varies by region
on_demand_h   = 0.68   # c6i.4xlarge on-demand

cost_align = samples * align_h * spot_price_h
cost_call  = samples * call_h  * on_demand_h
storage    = samples * 35 * 0.023   # 35GB BAM, S3 $0.023/GB-mo
egress     = 0                       # keep results in S3 — no download = no egress

print(f"Align (spot): ${cost_align:,.0f}")
print(f"Call  (on-demand): ${cost_call:,.0f}")
print(f"Storage 1mo: ${storage:,.0f}")
print(f"Total: ${cost_align + cost_call + storage:,.0f}")
# Typical: ~$1.7-2.2 per sample for spot-heavy mix.
實務

生物學家在雲端的 8 大省錢策略

💰

1. 善用 spot / preemptible

spot 機器比 on-demand 便宜 60-90%。對於 idempotent 的 trim / align job 配 retry policy，被中斷就重跑。對 variant calling 等較長 job 用 on-demand。

Spot is 60-90% cheaper than on-demand. Pair with retry policies for idempotent jobs (trim/align). Use on-demand for long-running steps (variant calling).

🗺️

2. 不要 download reference genome

把 reference (hg38, gencode index, etc.) 放 EFS / shared bucket，所有 worker 從 mount 直讀。避免每個 worker 都從 S3 抓 reference。

Put references (hg38, gencode index, etc.) on EFS or a shared bucket; mount and read in every worker. Avoid each worker pulling references from S3.

📤

3. 在雲端做完才下載

Egress 是 AWS 最大金錢黑洞 ($0.09/GB)。1000 個 BAM (35 TB) 下載要 $3,150。把 BAM 留在 S3，只下載 VCF / report。

Egress is AWS's biggest money trap ($0.09/GB). Downloading 1000 BAMs (35 TB) costs $3,150. Keep BAMs in S3; download only VCFs/reports.

🗄️

4. 分層儲存

剛產出的 BAM 放 S3 Standard，3 個月後 lifecycle policy 自動移到 S3 IA ($0.0125/GB)，1 年後到 Glacier ($0.004/GB)。

Fresh BAMs in S3 Standard; lifecycle policy auto-moves to S3 IA at 3 months ($0.0125/GB), Glacier at 1 year ($0.004/GB).

📦

5. 用 BioContainers 而非自建

Quay.io / BioContainers 已有幾乎所有生物學工具的 multi-arch image，省下 build + scan 時間。配合 Snakemake / Nextflow 的 conda+container 整合。

Quay.io / BioContainers ships multi-arch images for nearly every bio tool — saving build + scan time. Pair with Snakemake/Nextflow conda+container integration.

📊

6. 用 Tower / Cromwell UI 觀測

「黑箱跑了 6 小時」是雲端最貴的事故。Seqera Tower / Cromwell UI 顯示每個 task CPU、RAM、time，方便 right-sizing 與 debug。

"6 hours of black-box running" is the most expensive cloud accident. Seqera Tower / Cromwell UI show CPU/RAM/time per task — essential for right-sizing and debugging.

🌐

7. 善用 Open Data Registry

1000 Genomes、GTEx、TCGA、ENCODE、UK Biobank Open Tier 等公開資料在 AWS Open Data Registry 與 GCP Public Datasets 都有原生 mirror，從同 region 讀取「免費」。

1000 Genomes, GTEx, TCGA, ENCODE, UK Biobank Open Tier are mirrored on AWS Open Data Registry and GCP Public Datasets — reading from the same region is "free."

🔐

8. 預算告警與 tagging

設定每月 budget alert ($X 警示、$2X 切斷)。每個資源用 Tag 標註 project / PI，月底用 Cost Explorer 拆帳避免 "誰在燒錢"。

Set monthly budget alerts ($X warn, $2X hard cap). Tag every resource with project/PI; use Cost Explorer to attribute spend at month-end.

📝 自我檢測 (3 題)

Q1. 你在 AWS S3 存了 1000 個 BAM (共 35 TB) 並打算每天從本地 download 全部分析——這個流程最大的成本問題是？

Q1. You store 1000 BAMs (35 TB) on S3 and plan to download all of them to local for daily analysis — what's the biggest cost problem?

A. S3 儲存費用太高A. S3 storage is too expensive
B. EC2 不夠快B. EC2 isn't fast enough
C. Egress (從 AWS 下載) 費用 ≈ $3,150 (35 TB × $0.09)，遠高於計算與儲存C. Egress (download from AWS) ≈ $3,150 (35 TB × $0.09), far higher than compute or storage
D. 沒問題D. No problem
答案 C。Egress 是雲端最大金錢黑洞。應該在雲端 (同 region 的 EC2) 完成分析，只下載小體積的 VCF / report。
Answer: C. Egress is the cloud's biggest money trap. Compute in cloud (same-region EC2), download only small VCFs/reports.

Q2. 在 SLURM 提交 array job 200 個樣本但同時只想跑 50 個，正確語法是？

Q2. Submit a SLURM array of 200 samples but cap concurrent runs at 50?

A. --array=1-200 --max=50A. --array=1-200 --max=50
B. --array=1-200%50B. --array=1-200%50
C. --array=1-50 --ntasks=200C. --array=1-50 --ntasks=200
D. 提交 200 次 sbatchD. Run sbatch 200 times
答案 B。%N 是 SLURM array 的並行上限語法。1-200%50 代表 200 個 task 同時最多跑 50 個。
Answer: B. %N caps concurrent array tasks. 1-200%50 means 200 tasks, max 50 concurrent.

Q3. 下列何者最適合用 AWS Batch 的 spot instance？

Q3. Which is best suited for AWS Batch spot instances?

A. 24 小時的 long-running GATK joint callingA. A 24-hour long-running GATK joint calling
B. 必須 ACID 的資料庫B. An ACID database
C. 2 小時 idempotent 的 trim/align，配 retry policyC. A 2-hour idempotent trim/align with retry policy
D. 即時面對使用者的 Web 服務D. A user-facing live web service
答案 C。Spot 可被 AWS 隨時回收，最適合短時、idempotent (可重跑) 的 batch job。配 retry 處理偶發中斷可節省 60-90% 成本。
Answer: C. Spot can be reclaimed anytime — ideal for short, idempotent batch jobs. Retries handle interruptions and save 60-90% cost.
