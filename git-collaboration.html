Git 進階協作：rebase、hooks、LFS、Flow 模型

從 commit/push 升級到實驗室級協作——rebase vs merge、interactive squash、reflog 救援、pre-commit 自動化、Git LFS 管 FASTQ/BAM、GitHub Flow vs GitFlow、PR 禮儀與 conventional commits。

From commit/push to lab-grade collaboration — rebase vs merge, interactive squash, reflog rescue, pre-commit automation, Git LFS for FASTQ/BAM, GitHub Flow vs GitFlow, PR etiquette, conventional commits.

動機

為什麼研究 git 需要「進階化」？

大多數實驗室的 git 用法只有 git add -A && git commit -m "update"。短期能用，但當你 (a) 要把 PR 提交給 nf-core，(b) 重現一年前的 figure，(c) 接手別人寫到一半的分析——三種痛點就會浮現：

  • 歷史一團亂：50 個 "fix typo" commit 混在一起，無法 bisect 找出哪一行害 reviewer 抱怨結果不對。
  • 不能 push：把 80GB 的 BAM 不小心 commit 進來，整個 repo 變成黑洞，clone 要 3 小時。
  • 分支管理失控：main / dev / new-figure / fix-bug-from-2024-12-03 全部 diverge，merge conflict 解到崩潰。

本章六個主題：rebase 與互動式重整、reflog 救命網、pre-commit hooks、Git LFS、Flow 模型、PR 禮儀。

Most lab git usage is just git add -A && git commit -m "update". Works short-term, but breaks when (a) you submit a PR to nf-core, (b) reproduce a figure from last year, (c) inherit half-written analysis. Three pains emerge:

  • Messy history: 50 "fix typo" commits jumbled together — impossible to bisect when a reviewer complains the numbers changed.
  • Cannot push: An 80GB BAM accidentally committed turns the repo into a black hole — clones take 3 hours.
  • Branch chaos: main / dev / new-figure / fix-bug-from-2024-12-03 all diverge, merge conflicts mount.

The six topics: rebase + interactive squash, reflog safety net, pre-commit hooks, Git LFS, flow models, PR etiquette.

💡
心法：commit 是給「未來的你」與「reviewer」的訊息，不是給機器。一個 commit = 一個邏輯改動 + 一句說得清楚的訊息。 Mindset: A commit is a message to future-you and to reviewers, not to the machine. One commit = one logical change + one clear message.
主題 1

一、Rebase vs Merge — 何時用哪個？

merge 保留分支歷史 (產生 merge commit)，rebase 把你的 commit 重新「貼」到目標分支頂端，產生線性歷史。原則：本地、私人分支用 rebase；已 push 給別人的 commit 不要 rebase。

merge preserves branch history (creates a merge commit); rebase replays your commits on top of the target, yielding linear history. Rule: rebase locally on private branches; never rebase commits others have pulled.

操作 歷史形狀 衝突解決 是否安全於共享分支 典型用途
git merge保留分支 + merge commit一次解完✅把 feature 合回 main
git rebase線性 (改寫 SHA)可能反覆解❌清理私人分支再開 PR
git merge --squash單一 commit 進 main一次解完✅壓成單一 commit
cherry-pick單一 commit 移植只該 commit✅hotfix 跨分支移植
rebase.sh — keep your branch on top of main
# You're on feature/scrnaseq-qc, main has moved forward.
# Goal: replay your commits on the new main tip.
git fetch origin
git checkout feature/scrnaseq-qc
git rebase origin/main           # replays each of your commits

# If a conflict appears:
#   1. edit the file, git add <file>
#   2. git rebase --continue
#   3. or: git rebase --abort to bail out

# After rebase, your branch's SHAs changed → force-push (use --force-with-lease!)
git push --force-with-lease origin feature/scrnaseq-qc
interactive.sh — squash + reword 50 WIP commits
# Show last 10 commits in editor for editing
git rebase -i HEAD~10

# Editor opens; change "pick" to:
#   s / squash   = combine into previous commit
#   r / reword   = keep commit, edit message
#   d / drop     = delete this commit
#   e / edit     = pause to amend code
#   f / fixup    = like squash but discard the message

# Example before:
# pick a1b2c3d add qc function
# pick e4f5g6h fix typo
# pick i7j8k9l fix typo again
# pick m0n1o2p add tests

# After editing:
# pick a1b2c3d add qc function
# f    e4f5g6h fix typo
# f    i7j8k9l fix typo again
# pick m0n1o2p add tests

# Result: 4 → 2 clean commits
cherry-pick.sh — hotfix to release branch
# A critical fix landed on main as commit abc1234.
# You need it on the v1.2-release branch too.
git checkout v1.2-release
git cherry-pick abc1234

# Cherry-pick a range
git cherry-pick abc1234^..def5678

# Cherry-pick without auto-commit (lets you edit before commit)
git cherry-pick -n abc1234
⚠️
The Golden Rule of Rebase：千萬不要 rebase 已 push 到 shared branch 的 commit。如果同事已從 origin/feature 拉下來，你 rebase 後 force-push，他們本機歷史會與遠端衝突，可能遺失工作。 The Golden Rule of Rebase: Never rebase commits already pushed to a shared branch. If a teammate pulled origin/feature, your rebase + force-push leaves their history conflicting with the remote — they may lose work.
主題 2

二、Reflog — 你的時光機

Reflog 紀錄你的 HEAD 在過去 90 天 (預設) 移動過的所有位置——包括被 reset --hard 砍掉的 commit、被 rebase 改寫掉的 SHA。只要 commit 曾在你本機存在，就一定救得回來。

Reflog records every position HEAD has been in the last 90 days (default) — including commits killed by reset --hard or rewritten by rebase. If a commit ever existed locally, you can recover it.

recover.sh — undo a fatal reset --hard
# Disaster: you ran `git reset --hard HEAD~5` and lost 5 commits
git reflog
# Output:
# 8c4f1a2 HEAD@{0}: reset: moving to HEAD~5
# 3d9e0b7 HEAD@{1}: commit: add UMAP figure  ← what you want
# 7a1c2d8 HEAD@{2}: commit: refactor QC

git reset --hard HEAD@{1}     # back to the good state
# All "lost" commits and files are restored
undo-rebase.sh — recover from a bad rebase
git rebase -i HEAD~20         # oops, dropped 3 important commits

git reflog
# 1a2b3c4 HEAD@{0}: rebase finished
# f5e6d7c HEAD@{1}: rebase: started
# 9c8b7a6 HEAD@{2}: commit: pre-rebase state  ← restore this

git reset --hard HEAD@{2}
# Branch is back to exactly where it was before rebase
💡
口訣：把 git reflog 想成「git 操作的瀏覽器歷史」。當你下了破壞性指令 (reset/rebase/checkout -f) 之後恐慌時，先深呼吸，然後 reflog。 Mantra: Think of git reflog as your "browser history of git operations." After any destructive command (reset/rebase/checkout -f), breathe, then reflog.
主題 3

三、Hooks 與 pre-commit framework

Git hooks 是 commit/push 等事件觸發的 shell scripts，放在 .git/hooks/。但本機 hook 不能版本化、不能跨平台 — 因此社群用 pre-commit framework (Python 工具) 來把 hooks 設定寫進 .pre-commit-config.yaml，commit 前自動跑 black / ruff / styler / nbstripout 等檢查器。

Git hooks are shell scripts triggered by events (commit/push), stored in .git/hooks/. But local hooks aren't versioned and aren't portable — the community uses the pre-commit framework (a Python tool) to declare hooks in .pre-commit-config.yaml, running black / ruff / styler / nbstripout automatically before each commit.

.pre-commit-config.yaml
# Run with: pre-commit run --all-files
repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
      - id: check-added-large-files
        args: ['--maxkb=500']    # block files > 500KB
      - id: check-merge-conflict

  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.5.0
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format

  - repo: https://github.com/psf/black
    rev: 24.4.2
    hooks:
      - id: black

  - repo: https://github.com/lorenzwalthert/precommit
    rev: v0.4.3
    hooks:
      - id: style-files          # R: styler
      - id: lintr                 # R: lintr

  - repo: https://github.com/kynan/nbstripout
    rev: 0.7.1
    hooks:
      - id: nbstripout            # strip Jupyter output before commit
install.sh
# One-time setup per developer
pip install pre-commit            # or: brew install pre-commit / uv tool install pre-commit
pre-commit install                # wires .git/hooks/pre-commit

# Run all hooks against all files (CI uses this too)
pre-commit run --all-files

# Bypass (rarely): only for emergencies
git commit --no-verify -m "WIP: stash before going home"

# Auto-update hook versions
pre-commit autoupdate
raw-hook.sh — old-school .git/hooks/pre-commit
#!/bin/sh
# .git/hooks/pre-commit  (must be chmod +x)
# Refuse commit if any staged Python file fails ruff

files=$(git diff --cached --name-only --diff-filter=ACM | grep '\.py$')
if [ -n "$files" ]; then
    ruff check $files || {
        echo "❌ ruff failed; commit aborted"
        exit 1
    }
fi
exit 0
主題 4

四、Git LFS、submodules、subtree

Git 是為原始碼設計的，每次 commit 整個 binary file 會把 repo 撐爆。生醫資訊有兩個常見需求：

  • 大檔案 (FASTQ/BAM/VCF/.h5ad)：用 Git LFS，repo 只存 pointer，實際內容存在 LFS server。
  • 共用其他 repo (e.g. 共用 reference pipeline)：用 submodule (link to specific SHA) 或 subtree (copy into your repo)。

Git was built for source code; committing binaries blows up the repo. Two common bioinformatics needs:

  • Large files (FASTQ/BAM/VCF/.h5ad): use Git LFS — repo holds a pointer, real bytes live on the LFS server.
  • Share another repo (e.g. common reference pipeline): use submodule (link to a specific SHA) or subtree (copy into your repo).
git-lfs.sh
# One-time install
git lfs install

# Track binary extensions — writes to .gitattributes (commit this!)
git lfs track "*.bam" "*.bai" "*.fastq.gz" "*.h5ad" "*.vcf.gz"
git add .gitattributes

# Now binaries are stored as pointers
git add tests/data/mini.bam
git commit -m "add test BAM via LFS"
git push

# Pointer file looks like:
#   version https://git-lfs.github.com/spec/v1
#   oid sha256:abc123...
#   size 1048576

# Caveat: GitHub free LFS quota is small (1GB storage, 1GB bandwidth/month).
# Heavy users go to Zenodo, S3, or self-hosted LFS server.
submodule.sh
# Pin to a specific SHA of an external repo
git submodule add https://github.com/nf-core/modules external/nf-core-modules
git commit -m "add nf-core/modules as submodule"

# Clone with submodules
git clone --recurse-submodules <url>

# Update to newer SHA in submodule
cd external/nf-core-modules
git checkout main && git pull
cd ../..
git add external/nf-core-modules
git commit -m "bump nf-core/modules"

# Pros: small repo; clear separation. Cons: extra clone step, confusing for newcomers.
subtree.sh — copy another repo into yours
# Pull in another repo's history under prefix=external/utils
git subtree add --prefix=external/utils \
    https://github.com/lab/utils.git main --squash

# Later, pull upstream changes
git subtree pull --prefix=external/utils \
    https://github.com/lab/utils.git main --squash

# Push your local changes back upstream
git subtree push --prefix=external/utils \
    https://github.com/lab/utils.git contrib

# Pros: clone is one step; no extra config. Cons: bigger repo, harder upgrades.

branch-protection.yml — GitHub repo settings
# Recommended for any analysis repo with > 1 author
protection:
  required_pull_request_reviews:
    required_approving_review_count: 1
    dismiss_stale_reviews: true
  required_status_checks:
    strict: true
    contexts:
      - "pytest (3.11)"
      - "pytest (3.12)"
      - "pre-commit"
      - "R CMD check"
  enforce_admins: false
  restrictions: null
  required_signatures: true      # GPG-signed commits only
  allow_force_pushes: false
  allow_deletions: false
signed-commits.sh
# Generate or import a GPG key (or use SSH signing in modern git)
gpg --full-generate-key
gpg --list-secret-keys --keyid-format=long
# Tell git to use it
git config --global user.signingkey ABCD1234EF567890
git config --global commit.gpgsign true
git config --global tag.gpgsign true

# Modern alternative: SSH key signing (git 2.34+)
git config --global gpg.format ssh
git config --global user.signingkey ~/.ssh/id_ed25519.pub

# Verify
git log --show-signature -1
💡
PR 禮儀 (生醫資訊版)：(1) PR < 400 行最容易被審；(2) 描述「為什麼」而不是「做了什麼」(diff 已說了 what)；(3) 附上跑過的 reproducible 結果或截圖；(4) 一個 PR 一個邏輯改動 — 別把 feature + refactor + style 混在一起；(5) reviewer 給 nit 標記時可以拒絕，但給 must 標記時要修。 PR etiquette (bioinformatics flavor): (1) Keep PRs < 400 LOC for fastest review; (2) describe the "why," not the "what" (the diff already says what); (3) include reproducible results or screenshots; (4) one PR = one logical change — don't mix feature + refactor + style; (5) reviewers' "nit:" comments can be declined; "must:" comments must be addressed.

📝 自我檢測 (3 題)

Q1. 同事已從 origin/feature 拉下來工作。你接著對 feature 做 git rebase main 然後 force-push。預期會發生？

Q1. Your teammate pulled origin/feature. You then run git rebase main on feature and force-push. What happens?

A. 同事的本機分支自動更新成新歷史A. Their local branch auto-updates to the new history
B. 同事的本機歷史與 remote 衝突，下次 pull 會 diverge 或遺失 commitB. Their local history conflicts with remote; next pull diverges or loses commits
C. GitHub 會 block force-pushC. GitHub blocks the force-push
D. 沒影響，rebase 只動 SHA 不動內容D. No impact — rebase only changes SHAs
答案 B。The Golden Rule of Rebase：不要 rebase 已分享給別人的 commit。要嘛開新分支做 squash merge，要嘛跟團隊溝通並 enable branch protection 防止 force-push。
Answer: B. The Golden Rule of Rebase: never rebase commits others have pulled. Use squash-merge on a new branch, or enable branch protection blocking force-pushes.

Q2. 下列何者最適合用 Git LFS？

Q2. Which file is best stored via Git LFS?

A. analysis.py (3KB Python script)A. analysis.py (3KB Python script)
B. README.mdB. README.md
C. tests/data/mini.bam (5MB test fixture)C. tests/data/mini.bam (5MB test fixture)
D. requirements.txtD. requirements.txt
答案 C。LFS 為 binary / 大檔案而設計，BAM 是典型例子。文字檔有 git diff 才有意義 — 不要進 LFS。注意 GitHub 免費 LFS 額度小，超過要付費或自架。
Answer: C. LFS is designed for binaries / large files; BAM is canonical. Text files need git diff — don't LFS them. Note GitHub's free LFS quota is small; heavy users pay or self-host.

Q3. pre-commit framework 相對於原生 .git/hooks/pre-commit 的主要優點？

Q3. Main advantage of the pre-commit framework over native .git/hooks/pre-commit?

A. 設定寫進 .pre-commit-config.yaml 可版本化、跨平台、跨語言A. Config in .pre-commit-config.yaml is versioned, cross-platform, multi-language
B. 跑得比較快B. Runs faster
C. 是 git 原生功能C. It's built into git
D. 能 block 大檔案被 pushD. Can block large files from being pushed
答案 A。原生 hooks 在 .git/hooks/ 不會被 commit，每個 developer 要手動裝；pre-commit framework 把設定寫進 repo 根目錄，pre-commit install 即可同步，且支援 Python/R/JS/shell 共存。
Answer: A. Native hooks live in .git/hooks/ which isn't committed — each dev installs manually. The framework keeps config at repo root, syncs via pre-commit install, and supports Python/R/JS/shell in one place.
