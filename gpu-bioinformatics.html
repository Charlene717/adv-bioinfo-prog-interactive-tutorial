GPU 加速生醫運算：何時值得搬到 GPU 上

CuPy、RAPIDS cuDF/cuML、rapids-singlecell、JAX、PyTorch——把 100 萬細胞 scRNA-seq 從 2 小時壓到 5 分鐘，但要先理解資料搬運與記憶體配額。

CuPy, RAPIDS cuDF/cuML, rapids-singlecell, JAX, PyTorch — compress a 1M-cell scRNA-seq from 2 hours to 5 minutes, but only after understanding host↔device transfer and memory budgets.

原則

GPU 何時真的幫得上忙？

GPU 不是萬靈丹。一張 A100 有 6912 個 CUDA core，但每個 core 比 CPU core 慢很多——GPU 贏在大規模、相同操作、資料密集；輸在分支、隨機存取、小資料。最常見適合 GPU 的生醫工作：

  • 大批次線性代數：scRNA-seq PCA、UMAP、neighbor graph（cuML 比 sklearn 快 10–50×）。
  • 深度學習：AlphaFold、ESM、scVI、Geneformer——本來就是給 GPU 設計的。
  • 影像處理：病理切片、cryo-EM 重建、顯微影像分割。
  • 序列比對：NVIDIA Parabricks (GATK on GPU)、CUDA-Smith-Waterman；short-read mapping 用 GPU 加速 4–10×。

不適合 GPU：VCF 解析、merge sort、Snakemake 排程——這些是 I/O 或分支密集，搬到 GPU 反而更慢。

GPU is not a silver bullet. An A100 has 6912 CUDA cores, but each is much slower than a CPU core — GPUs win at large, uniform, data-parallel ops; lose at branching, random access, small data. Typical bioinformatics wins:

  • Large-batch linear algebra: scRNA-seq PCA, UMAP, neighbor graphs (cuML is 10–50× faster than sklearn).
  • Deep learning: AlphaFold, ESM, scVI, Geneformer — built for GPUs.
  • Imaging: H&E slides, cryo-EM reconstruction, microscopy segmentation.
  • Sequence alignment: NVIDIA Parabricks (GATK on GPU), CUDA-Smith-Waterman; short-read mapping accelerated 4–10×.

NOT for GPU: VCF parsing, merge-sort, Snakemake scheduling — these are I/O- or branch-heavy and slow down on a GPU.

主題 1

一、CuPy 與 RAPIDS：drop-in 替代

NVIDIA + open-source 的 RAPIDS 生態系想做一件事：讓 NumPy / pandas / scikit-learn / NetworkX 的程式幾乎不用改就能跑在 GPU 上。

The NVIDIA + open-source RAPIDS stack has one mission: let NumPy / pandas / sklearn / NetworkX code run on GPU with almost zero changes.

cupy.py — numpy on GPU
import cupy as cp
import numpy as np

# CPU NumPy → GPU CuPy: change one import
X_cpu = np.random.randn(100_000, 2000).astype(np.float32)   # 800 MB
X_gpu = cp.asarray(X_cpu)                                          # transfer to device

# Same API as NumPy
mean   = X_gpu.mean(axis=0)
cov    = (X_gpu - mean).T @ (X_gpu - mean) / (X_gpu.shape[0] - 1)
eigvals, eigvecs = cp.linalg.eigh(cov)

# Bring back to CPU only when needed (e.g. for plotting)
loadings = cp.asnumpy(eigvecs[:, -10:])

⚠️
Host↔Device 搬運是隱性 bottleneck：PCIe 5.0 約 64 GB/s，相比 A100 內部 2 TB/s 慢 30 倍。盡量「進 GPU 一次、做完所有事再出來」。每次 cp.asnumpy() 都是一次同步障壁，會讓 GPU 等到資料拷出去。 Host↔Device transfer is a hidden bottleneck: PCIe 5.0 is ~64 GB/s vs A100's internal 2 TB/s — 30× slower. Move to GPU once, do everything, then move back. Each cp.asnumpy() is a sync barrier; the GPU waits until the copy finishes.
主題 2

二、JAX 與 PyTorch：autodiff + GPU

當你需要可微分計算（VI、deep learning、bayesian inference），JAX 與 PyTorch 提供：(1) NumPy-like API，(2) 自動微分，(3) JIT 編譯到 GPU/TPU，(4) 多 GPU/多機器（jax.pmap / PyTorch DDP）。

When you need differentiable computation (variational inference, deep learning, Bayesian models), JAX and PyTorch give you: (1) NumPy-like API, (2) automatic differentiation, (3) JIT to GPU/TPU, (4) multi-GPU/multi-node (jax.pmap / PyTorch DDP).

jax.py — jit + grad + pmap
import jax, jax.numpy as jnp
from jax import jit, grad, vmap, pmap

def neg_log_lik(params, X, y):
    mu = X @ params["w"] + params["b"]
    return jnp.mean((mu - y) ** 2)

# jit-compile the gradient — runs entirely on GPU
grad_fn = jit(grad(neg_log_lik))

for step in range(1000):
    g = grad_fn(params, X, y)
    params = jax.tree.map(lambda p, gp: p - 1e-3 * gp, params, g)

# vmap: vectorize over a new axis (e.g. multiple chromosomes)
batched = vmap(neg_log_lik, in_axes=(None, 0, 0))

# pmap: replicate across multiple GPUs (data parallel)
multi_gpu = pmap(grad_fn)
torch.py — typical training loop
import torch
import torch.nn as nn

device = torch.device("cuda" if torch.cuda.is_available() else "cpu")

# scVI-style VAE on counts
class CountVAE(nn.Module):
    def __init__(self, n_genes, latent=10):
        super().__init__()
        self.enc = nn.Sequential(nn.Linear(n_genes, 128), nn.ReLU(),
                                  nn.Linear(128, latent * 2))
        self.dec = nn.Sequential(nn.Linear(latent, 128), nn.ReLU(),
                                  nn.Linear(128, n_genes))
    def forward(self, x):
        h = self.enc(x); mu, log_var = h.chunk(2, dim=-1)
        z = mu + torch.randn_like(mu) * (log_var * 0.5).exp()
        return self.dec(z), mu, log_var

model = CountVAE(n_genes=2000).to(device)
opt   = torch.optim.Adam(model.parameters(), lr=1e-3)
for batch in loader:
    x = batch.to(device, non_blocking=True)
    recon, mu, lv = model(x)
    loss = ((recon - x) ** 2).mean() - 0.5 * (1 + lv - mu**2 - lv.exp()).mean()
    opt.zero_grad(); loss.backward(); opt.step()

💡
選 GPU 的原則：(1) 先看 VRAM 夠不夠 —— 一張 24GB L4 跑不動需要 60GB 的 ESM-2 3B；(2) 訓練看 BF16 TFLOPS，inference 看 FP16/INT8；(3) 雲端短跑挑 A100/L4 spot，常駐生產線挑 H100 reserved；(4) Apple Silicon 的 MPS backend 適合在 MacBook 開發小批次原型，正式訓練再上 cloud。 Picking a GPU: (1) Check VRAM first — a 24 GB L4 can't load a 60 GB ESM-2 3B; (2) training cares about BF16 TFLOPS, inference about FP16/INT8; (3) short bursty workloads → A100/L4 spot; long-running prod → H100 reserved; (4) Apple Silicon MPS is fine for laptop-side prototyping, cloud for serious training.
⚠️
CUDA / ROCm / MPS 三大陣營：NVIDIA CUDA 是事實標準（生態最完整）；AMD ROCm 近年 PyTorch 支援已成熟，但 RAPIDS / cuDF 仍 NVIDIA-only；Apple MPS 對 PyTorch 已實用，但訓練速度與穩定度遠不及 CUDA。混合運算記得 torch.backends.cudnn.benchmark = True 來自動找最佳 kernel。 The three GPU camps: NVIDIA CUDA is the de-facto standard (richest ecosystem); AMD ROCm has matured for PyTorch but RAPIDS / cuDF remain NVIDIA-only; Apple MPS is usable for PyTorch but lags CUDA in speed and stability. For mixed workloads enable torch.backends.cudnn.benchmark = True for auto kernel selection.
主題 4

四、多 GPU 策略

三種主流分割：

  • Data parallel (DP / DDP)：每張 GPU 拿完整模型 + 不同 batch，梯度做 all-reduce。最常用，PyTorch DistributedDataParallel、JAX pmap。模型必須 fit 在單卡 VRAM。
  • Model parallel / pipeline：模型切片放在不同 GPU。用於模型大於單卡（ESM-2 15B、AlphaFold-Multimer）。工具：DeepSpeed、FSDP、Megatron-LM。
  • Tensor parallel：單一層的 matrix 切片到多卡，需要快速 NVLink。Mainly LLM 訓練。

Three mainstream partitioning strategies:

  • Data parallel (DP / DDP): Each GPU holds the full model + a different batch; gradients are all-reduced. Most common; PyTorch DistributedDataParallel, JAX pmap. Model must fit on one GPU.
  • Model / pipeline parallel: Slice the model across GPUs. For models larger than one GPU (ESM-2 15B, AlphaFold-Multimer). Tools: DeepSpeed, FSDP, Megatron-LM.
  • Tensor parallel: Slice a single layer's matrix across GPUs; needs fast NVLink. Mostly LLM training.

📝 自我檢測 (3 題)

Q1. 下列任務中，哪一個搬到 GPU 「最不可能」帶來實質加速？

Q1. Which of these is least likely to benefit from moving to GPU?

A. 100 萬細胞 scRNA-seq 的 PCA 與 UMAPA. PCA + UMAP on 1M-cell scRNA-seq
B. 解析 100GB VCF 並做欄位過濾B. Parsing a 100 GB VCF with column filters
C. 訓練 scVI 變分自編碼器C. Training a scVI variational autoencoder
D. 病理切片影像分割D. Pathology slide image segmentation
答案 B。VCF 解析是 I/O + 字串處理 + 分支密集——GPU 強在線性代數和影像，弱在 if/else 和隨機讀。cudf 對「已經在 GPU 上的整潔表」有用，但「解析原始檔」會被 I/O 卡住。
Answer: B. VCF parsing is I/O + string + branch heavy — GPUs excel at linalg/imaging, not if/else and random reads. cudf helps once data is on GPU, but parsing raw files is I/O-bound.

Q2. 你的 PyTorch 訓練在 batch_size=64 時 OOM。下列哪一組做法「不」會減少 GPU 記憶體？

Q2. Your PyTorch training OOMs at batch_size=64. Which option does NOT reduce GPU memory?

A. 降 batch size 並用 gradient accumulationA. Reduce batch size and use gradient accumulation
B. 啟用 mixed precision (autocast + GradScaler)B. Enable mixed precision (autocast + GradScaler)
C. 把 learning rate 從 1e-3 改成 1e-4C. Change learning rate from 1e-3 to 1e-4
D. 啟用 gradient checkpointingD. Enable gradient checkpointing
答案 C。Learning rate 純粹影響數值收斂，與記憶體無關。A 減少 activation、B 把 fp32 換 fp16 省一半 activation、D 用「重算代替存」省 activation——都是常用 OOM 救火法。
Answer: C. Learning rate affects only convergence, not memory. A reduces activations, B halves activation memory via fp16, D trades compute for memory by recomputing activations — all standard OOM remedies.

Q3. 在 RAPIDS 工作流中，你每處理一步就 cp.asnumpy() 把結果搬回 CPU 看一下——為什麼效能會大幅下降？

Q3. In a RAPIDS workflow you cp.asnumpy() back to CPU after every step to inspect — why does performance crater?

A. 每次拷貝都是 host↔device 同步障壁，PCIe 頻寬遠低於 GPU 內部，且強迫 GPU 等到資料傳完A. Each copy is a host↔device sync barrier; PCIe is far slower than internal GPU bandwidth, and the GPU stalls until the copy finishes
B. CuPy 不支援 .asnumpy()B. CuPy doesn't support .asnumpy()
C. 會自動觸發 garbage collectionC. It triggers automatic garbage collection
D. CPU 比 GPU 還慢D. CPUs are slower than GPUs
答案 A。PCIe 5.0 ~64 GB/s 對上 A100 內部 ~2 TB/s 是 30 倍差距；而且 .asnumpy() 必須先讓 GPU 完成手邊所有 kernel 才能拷出——這個同步本身就抹掉並行收益。原則：盡量在 GPU 上跑完整段 pipeline，最後再搬回。 Answer: A. PCIe 5.0 ~64 GB/s vs A100's internal ~2 TB/s is a 30× gap; .asnumpy() also forces the GPU to flush all pending kernels before copying — the sync alone destroys parallelism gains. Rule: run the whole pipeline on GPU, copy back at the end.
