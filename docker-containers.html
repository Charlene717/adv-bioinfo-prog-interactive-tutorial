容器化：Docker / Apptainer 之於生醫資訊

把「我電腦上能跑」變成「任何電腦都能跑、五年後還能跑」——image vs container、多階段建構、BioContainers、Apptainer for HPC、安全與快取。

Turn "works on my machine" into "works anywhere, still works in five years" — images vs containers, multi-stage builds, BioContainers, Apptainer on HPC, security & caching.

動機

為什麼生醫資訊離不開容器？

一個典型 scRNA-seq pipeline 串接 fastqc → STAR → samtools → cellranger → scanpy → scvi-tools——每個工具有自己的 C 函式庫、Python 版本、R 套件需求。Conda 可以管套件，但無法管 glibc 版本、kernel 模組、CUDA 驅動。容器把「OS 之上的一切」都包進去，達成三件事：

  • 環境鎖定：image hash (SHA-256) 唯一對應一份完整環境，五年後 docker pull biocontainers/samtools:1.18 仍取得同一個二進位。
  • 跨平台：你的 Mac、同事的 Linux 工作站、AWS 上的 EC2、HPC 的 SLURM node——同一 image 跑同一行為。
  • 權限分離：容器內看不到 host 的 /home，誤刪 rm -rf 也不會傷到 host filesystem。

本章涵蓋 Docker（單機/雲端標準）、Apptainer（HPC 標準，前身 Singularity）、BioContainers（Bioconda 自動產出的官方 image registry）、與 Snakemake/Nextflow 的整合。

A typical scRNA-seq pipeline chains fastqc → STAR → samtools → cellranger → scanpy → scvi-tools — each tool has its own C libraries, Python version, R-package needs. Conda manages packages but cannot manage glibc, kernel modules, CUDA drivers. Containers package "everything above the OS," giving you three wins:

  • Frozen environments: An image hash (SHA-256) uniquely identifies a full environment; five years later docker pull biocontainers/samtools:1.18 still yields the same binaries.
  • Cross-platform: Your Mac, a colleague's Linux box, an AWS EC2, an HPC SLURM node — same image, same behavior.
  • Isolation: A container cannot see host's /home; an accidental rm -rf never harms the host filesystem.

This chapter covers Docker (single-machine/cloud standard), Apptainer (HPC standard, formerly Singularity), BioContainers (Bioconda-auto-built image registry), and integration with Snakemake/Nextflow.

💡
關鍵概念：「image」是不可變的快照（像光碟），「container」是執行中的 image 實例（像光碟在 player 裡）。一份 image 可以同時跑出 N 個 container。 Key concept: An image is an immutable snapshot (like a CD-ROM); a container is a running instance of that image (the CD in a player). One image → many concurrent containers.
主題 1

一、Dockerfile 核心指令

Dockerfile 是文字檔，每一行是一個 layer。BuildKit 會 cache 每個 layer——只要該行以上沒改，就能命中 cache 大幅加速 rebuild。指令順序原則：變動最少的指令放最上面（base image、apt 套件），常變動的放下面（COPY source、ENTRYPOINT）。

A Dockerfile is plain text where each line becomes a layer. BuildKit caches each layer — as long as everything above is unchanged, the cache hits and rebuilds become fast. Ordering rule: least-changing instructions on top (base image, apt packages), most-changing at the bottom (COPY source, ENTRYPOINT).

Dockerfile — bioinformatics CLI
# syntax=docker/dockerfile:1.7
FROM python:3.12-slim AS base

# 1) System libs — rarely change, stays cached
RUN apt-get update && apt-get install -y --no-install-recommends \
      build-essential libhts-dev samtools bcftools tabix \
    && rm -rf /var/lib/apt/lists/*

# 2) Python deps — re-runs only if requirements.txt changes
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 3) App code — changes most often, last
COPY src/ ./src/
COPY bin/ ./bin/

# 4) Non-root user (security best practice)
RUN useradd -m -u 1000 biouser
USER biouser

ENV PYTHONUNBUFFERED=1 \
    PATH=/app/bin:$PATH

# ENTRYPOINT = always-run binary; CMD = default args (overridable)
ENTRYPOINT ["python", "-m", "variant_qc"]
CMD ["--help"]

build.sh — with BuildKit cache mount
#!/bin/bash
# BuildKit cache speeds up pip / apt by sharing cache across builds
export DOCKER_BUILDKIT=1

docker build \
  --cache-from=type=registry,ref=ghcr.io/lab/variant-qc:buildcache \
  --cache-to=type=registry,ref=ghcr.io/lab/variant-qc:buildcache,mode=max \
  -t ghcr.io/lab/variant-qc:1.2.0 \
  -t ghcr.io/lab/variant-qc:latest \
  .

docker push ghcr.io/lab/variant-qc:1.2.0
⚠️
ENTRYPOINT vs CMD：ENTRYPOINT 是「永遠執行的指令」，CMD 是「可被 docker run image arg 覆寫的預設參數」。錯把所有東西塞進 CMD，使用者就無法乾淨地傳入自己的引數。 ENTRYPOINT vs CMD: ENTRYPOINT is "always runs"; CMD is "default args, overridable by docker run image arg." Putting everything in CMD prevents users from cleanly passing their own arguments.
主題 2

二、多階段建構 (multi-stage) — 縮小 image

編譯 htslib、bwa 需要 GCC、make、autoconf——這些工具加起來 500MB+。但 runtime 只需要編譯出的二進位。多階段建構讓你「在 builder 階段裝編譯工具，最後階段只拷貝產物」，把 image 從 1.5GB 砍到 200MB。

Compiling htslib, bwa needs GCC, make, autoconf — those alone add 500MB+. But runtime only needs the compiled binaries. Multi-stage builds let you "install build tools in a builder stage, copy only artifacts to a final stage," shrinking the image from 1.5GB to 200MB.

Dockerfile.multi — two stages
# ===== STAGE 1: builder (has compilers) =====
FROM ubuntu:22.04 AS builder

RUN apt-get update && apt-get install -y \
      build-essential autoconf wget zlib1g-dev libbz2-dev \
      liblzma-dev libcurl4-openssl-dev libssl-dev

WORKDIR /build
RUN wget https://github.com/samtools/samtools/releases/download/1.18/samtools-1.18.tar.bz2 \
    && tar xjf samtools-1.18.tar.bz2 \
    && cd samtools-1.18 \
    && ./configure --prefix=/install \
    && make -j$(nproc) install

# ===== STAGE 2: runtime (no compilers) =====
FROM ubuntu:22.04 AS runtime

RUN apt-get update && apt-get install -y --no-install-recommends \
      zlib1g libbz2-1.0 liblzma5 libcurl4 libssl3 \
    && rm -rf /var/lib/apt/lists/*

# Copy ONLY the compiled binary, leave compilers behind
COPY --from=builder /install /usr/local

RUN useradd -m biouser
USER biouser
ENTRYPOINT ["samtools"]
size_compare.txt
# Single-stage build with compilers retained:
$ docker images variant-qc:single
REPOSITORY    TAG    IMAGE ID     SIZE
variant-qc    single 7a3f...      1.52 GB

# Multi-stage build:
$ docker images variant-qc:multi
REPOSITORY    TAG    IMAGE ID     SIZE
variant-qc    multi  9c2e...      218 MB

# 7× smaller → faster push/pull, smaller attack surface

主題 4

四、Apptainer (Singularity) — HPC 上的選擇

大學 HPC 幾乎不開放 Docker（Docker daemon 以 root 執行，是 multi-user 集群的安全噩夢）。Apptainer（2021 從 Singularity 改名）解決這個問題：

  • 無 daemon：每個 user 各自啟動 container，root 權限不外漏。
  • 單檔 image：SIF 檔（Singularity Image Format）可以放在 NFS、Lustre 上分享。
  • 相容 Docker：直接 apptainer pull docker://biocontainers/samtools:1.18 轉換現有 Docker image。
  • 自動掛載 $HOME：container 內可直接讀寫 host 的 home，與 Docker 預設行為相反。

University HPCs rarely allow Docker (the daemon runs as root — a multi-user nightmare). Apptainer (renamed from Singularity in 2021) fixes this:

  • No daemon: Each user runs their own container; no root leak.
  • Single-file image: SIF (Singularity Image Format) can sit on NFS/Lustre and be shared.
  • Docker-compatible: apptainer pull docker://biocontainers/samtools:1.18 converts existing Docker images.
  • Auto-mounts $HOME: Inside the container you can read/write the host home directly — opposite of Docker's default.
apptainer_basics.sh
# 1. Pull a Docker image as SIF
apptainer pull docker://quay.io/biocontainers/samtools:1.18--h50ea8bc_1
# → produces samtools_1.18--h50ea8bc_1.sif (~30 MB)

# 2. Run a command inside the container
apptainer exec samtools_1.18--h50ea8bc_1.sif samtools view -c sample.bam

# 3. Bind host paths — needed when working files live outside $HOME
apptainer exec --bind /scratch:/scratch,/projects:/projects \
    samtools_1.18.sif samtools sort /scratch/sample.bam -o /scratch/sorted.bam

# 4. Run with no host filesystem leak (more isolated)
apptainer exec --containall --bind /scratch:/scratch tool.sif my_cmd

六、Compose、安全、與漏洞掃描




當你的開發環境要同時跑 PostgreSQL + Redis + Jupyter 三個服務，docker compose 讓你用一個 YAML 同時啟動。Production 與資安實務則要求：(1) 不以 root 跑 container；(2) 唯讀檔案系統；(3) 定期掃描 image 漏洞（trivy）。






For dev environments running PostgreSQL + Redis + Jupyter together, docker compose spins them up from one YAML. Production hygiene: (1) don't run as root; (2) read-only filesystem; (3) scan images for CVEs (trivy).

trivy_scan.sh — CVE scan in CI
# Install: brew install trivy  /  apt install trivy

# Scan image for HIGH/CRITICAL CVEs only; fail CI on findings
trivy image \
    --severity HIGH,CRITICAL \
    --exit-code 1 \
    --ignore-unfixed \
    ghcr.io/lab/variant-qc:1.2.0

# Also scan filesystem during build (catches secrets, leaked .env)
trivy fs --scanners secret,vuln .

📝 自我檢測 (3 題)





Q1. 在 Dockerfile 中，COPY requirements.txt . 放在 RUN pip install 之前的主要目的是？


Q1. Why place COPY requirements.txt . before RUN pip install in a Dockerfile?




A. 因為 pip 需要 requirements.txt 才能執行A. pip needs requirements.txt to run


B. 利用 layer cache：只要 requirements.txt 不變，pip install 步驟就能 cache 命中，重 build 飛快B. Layer caching: as long as requirements.txt is unchanged, pip install hits cache and rebuilds are fast


C. 為了安全C. For security


D. 沒有差別，只是慣例D. No difference; just convention




答案 B。把不變的依賴單獨拷貝可以最大化 cache 命中。若一起 COPY . .，任何 source code 改動都會讓 pip install 重跑。


Answer: B. Copying invariant deps separately maximizes cache hits. If you COPY . ., any source change forces pip install to re-run.







Q2. 大學 HPC 通常拒絕安裝 Docker 的主要原因？


Q2. Why do most university HPCs refuse to install Docker?




A. Docker 需要付授權費A. Docker requires a license fee


B. Docker image 比 Apptainer 大B. Docker images are larger than Apptainer's


C. Docker daemon 以 root 執行，使用者可逃逸成 root，多用戶集群安全風險高C. The Docker daemon runs as root; users can escalate to root — a security risk on multi-user clusters


D. Docker 不能跑 GPUD. Docker cannot use GPUs




答案 C。Apptainer 設計時就是為了解決 HPC 環境多用戶共享的 root 風險，所以容器以呼叫者的 UID 執行，不需 daemon。


Answer: C. Apptainer was designed exactly to remove the rooted-daemon risk on shared HPC; containers run as the calling user's UID, no daemon required.







Q3. 下列哪個做法最能把 1.5GB 的 image 砍到 200MB？


Q3. Which is the most effective way to shrink a 1.5GB image to 200MB?




A. 多階段建構：builder stage 編譯，最後 stage 只 COPY 二進位A. Multi-stage build: compile in a builder stage, COPY only binaries to the final stage


B. 壓縮成 tar.gz 再發布B. Gzip the image before publishing


C. 把所有 RUN 合併成一行C. Combine all RUN into a single line


D. 改用 ApptainerD. Switch to Apptainer




答案 A。編譯工具鏈（gcc, make…）通常 500MB+，多階段建構讓它們留在 builder stage 不進最終 image。合併 RUN（C）只能省幾 MB。


Answer: A. Compiler toolchains (gcc, make…) are usually 500MB+; multi-stage leaves them in the builder. Combining RUN (C) only saves a few MB.
