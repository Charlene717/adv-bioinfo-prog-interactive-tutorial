補充說明 · Notes

E1 Apptainer 補述：rootless 與 GPU

(A) Rootless / fakeroot 模式：Docker 早期 daemon 需要 root，因此 HPC 才轉向 Apptainer。但 Apptainer 並非「自動 rootless」——若要在 cluster 上 apptainer build，需 admin 啟用 fakeroot 或 setuid；許多 cluster 只允許 apptainer run/exec 既有 SIF，build 必須在本機完成。Docker 端對應的是 rootless Docker / Podman，並非完全等價。

(C) SIF vs OCI：Apptainer 原生格式 SIF 是單一可簽章檔案，方便 reproducibility 與 GPG 簽章；但 push 到 OCI registry (Docker Hub / GHCR) 仍要轉成 OCI layers，反向也要 apptainer pull docker://...。雙向轉換並非無損 — selinux labels、特殊 mounts、healthcheck 都可能丟失。

The previous note covered Apptainer's read-only default and auto bind-mounts. Two more "looks like Docker, behaves differently" pitfalls common on HPC:

(A) Rootless / fakeroot mode: Docker historically required a root daemon — the reason HPC moved to Apptainer. But Apptainer is not "automatically rootless": to apptainer build on a cluster, admins must enable fakeroot or setuid; many clusters allow only apptainer run/exec on pre-built SIF, with builds done locally. The Docker analog is rootless Docker / Podman — not a one-to-one mapping.

(B) GPU passthrough: Docker uses --gpus all (via NVIDIA Container Toolkit). Apptainer uses --nv (NVIDIA) or --rocm (AMD), still relying on the host's CUDA / driver. If the host CUDA major version mismatches the image's libcudart, jobs silently crash or fall back to CPU — a classic "CI passes, HPC fails" cause. Pin a driver-compatible CUDA minor inside the image.

(C) SIF vs OCI: Apptainer's native SIF is a single signable file, friendly for reproducibility and GPG signing — but pushing to an OCI registry (Docker Hub / GHCR) requires conversion to OCI layers, and the reverse apptainer pull docker://... too. Bidirectional conversion is not lossless: SELinux labels, special mounts, and healthchecks can be dropped.

E3/F · FAIR ≠ Open（含 FAIR4RS for software）

Findable (F)：metadata + 持久識別碼 (PID, 如 DOI) 可被搜尋。
• Accessible (A)：透過開放、標準化的協定取得 — 但「需要授權」依然 FAIR。臨床、基因隱私、商業敏感資料 metadata 可公開、資料本身需 controlled access (如 dbGaP、EGA)，這完全符合 FAIR。
• Interoperable (I)：使用 controlled vocabulary、標準 ontology (如 OBO Foundry, MIxS)。
• Reusable (R)：清楚授權 (license)、出處 (provenance)、領域標準。

Open data 是 R 中關於 license 的一個選項（如 CC0 / CC-BY），不是 FAIR 的必要條件。把「不公開資料 = 不 FAIR」當成 grant rejection 理由是錯的——只要 metadata 公開且 access protocol 標準化，受控資料一樣 FAIR。

FAIR4RS (FAIR for Research Software, Chue Hong et al. 2022 RDA) 把 FAIR 延伸到軟體：(F) 程式碼有 DOI（如 Zenodo-GitHub 整合）；(A) 可下載原始碼且有版本資訊；(I) 使用標準依賴聲明、容器化、CFF (Citation File Format)；(R) 開源 license + 測試 + 文件。FAIR4RS 也明確 不要求 軟體必須 open source，但若閉源需提供等效的可重現性保證。

The tutorial cites the FAIR principles (Wilkinson et al. 2016 Sci Data) in its "data / software reproducibility" sections. The most pervasive community misconception is FAIR = Open — which the original paper explicitly rejects. Wilkinson et al. 2016 §Box 2 states:

• Findable (F): metadata + persistent identifier (PID, e.g. DOI) discoverable.
• Accessible (A): retrievable via a standardized, open protocol — but "authentication required" is still FAIR. Clinical, genomic-privacy, or commercially sensitive data can have public metadata while the data itself sits behind controlled access (dbGaP, EGA), and that is fully FAIR-compliant.
• Interoperable (I): controlled vocabularies, standard ontologies (OBO Foundry, MIxS).
• Reusable (R): explicit license, provenance, domain standards.

Open data is one license-related option under R (e.g., CC0, CC-BY), not a FAIR prerequisite. Equating "not public" with "not FAIR" as a grant-rejection rationale is incorrect — controlled-access data is FAIR if metadata is public and the access protocol is standardized.

FAIR4RS (FAIR for Research Software, Chue Hong et al. 2022 RDA) extends FAIR to software: (F) code has a DOI (e.g., Zenodo–GitHub integration); (A) source downloadable with version info; (I) standardized dependency declarations, containerization, CFF (Citation File Format); (R) open license + tests + documentation. FAIR4RS also does not require software to be open source, but closed-source releases must provide equivalent reproducibility guarantees.
