勘誤 · CAVEATS

⚠️ 工具使用提醒與 caveats

以下為各工具實務上的常見誤解、benchmark 結果不一致之處，以及容易在教學中被忽略的細節。並非否定原工具，而是補充使用情境上的限制。

B2 Numba：JIT 編譯開銷 (warm-up cost)

Numba @jit/@njit 第一次呼叫時須花數秒到數十秒做編譯，這個成本在「跑一次小資料」時可能比純 Python 還慢。建議：(1) 用 cache=True 把編譯結果存到 __pycache__；(2) 在 benchmark 時剔除第一次 warm-up；(3) 若僅執行一次小資料，向量化 NumPy 通常較佳。

來源：Numba FAQ — Performance。

B3 BiocParallel：SerialParam 其實沒有平行

code>BiocParallel::SerialParam() 顧名思義是「序列執行」，不會啟動多 worker。許多 Bioconductor 套件（DESeq2、scran、scDblFinder）預設 BPPARAM=SerialParam()，使用者若沒有顯式傳入 MulticoreParam(workers=N) 就只會單核跑。教學中介紹 BiocParallel 時應強調此點。

來源：BiocParallel Introduction vignette。

B3 Python multiprocessing：Windows / Jupyter 的 __main__ 限制

在 Windows（spawn start method）與 Jupyter notebook 環境下，multiprocessing.Pool 要求被執行的函式必須能被 pickle 並從 __main__ import；常出現 AttributeError: Can't pickle local object。實務上：把 worker function 放在獨立 .py 檔，或改用 concurrent.futures.ProcessPoolExecutor 搭配 if __name__ == '__main__': guard。

來源：multiprocessing Programming guidelines。

C2 Polars vs DuckDB benchmark 結果不一致

社群常引用 H2O DB-benchmark 顯示 Polars 比 pandas 快 10×、與 DuckDB 互有勝負。實情：勝負取決於 query 類型（group-by 多列 vs join vs window）、資料分佈與是否 lazy。建議：對自己的 workload 用 scan_parquet() + collect() 來實測，不要單純信任 marketing 數字。

來源：DuckDB Labs db-benchmark。

C1 AnnData backed mode 的隱性寫入限制

AnnData 支援 backed='r' 直接從 HDF5 / Zarr 檔懶載入大型矩陣，但這個模式有兩個常被忽略的限制：(1) backed='r+' 寫入 HDF5 在 anndata < 0.10 時對 sparse matrix 會丟資料；(2) backed mode 下 .X 的部分 slicing 與 copy() 行為與 in-memory 不一致。處理 >10⁶ cells 資料時建議用 Zarr backend，並 pin anndata >= 0.10.5。

來源：AnnData 0.10 release notes。

D2 Coverage % 不是品質指標

教學中常以「Coverage > 80%」作為 PR 通過標準。Coverage.py 官方文件明確指出：高覆蓋率只代表程式碼被執行過，不代表 assertion 真的測到了預期行為。例如 def f(x): return x; test: assert f(1) is not None 有 100% coverage 但測得無物。應搭配 mutation testing（如 mutmut）或同行 PR review 衡量測試品質。

來源：Coverage.py FAQ。

D3 uv / Poetry / pip：lockfile 與 build backend 易混淆

2024 後的 Python 打包工具同時扮演「dependency resolver」與「build backend」兩個角色，但兩者其實獨立：uv 與 poetry 都是 resolver；build backend 則是 hatchling、setuptools、poetry-core、maturin（Rust）等。實務上 uv lock 產生 uv.lock，但 pyproject.toml 內仍須宣告 [build-system] build-backend = ...。混用最常見問題：用 uv 管 deps 但 build backend 是 setuptools.legacy，造成 sdist 缺檔。

來源：PyPA — setup.py deprecation、uv project concepts。

E1 Docker on HPC：Apptainer 不是 1:1 替代

HPC cluster 多用 Apptainer/Singularity 取代 Docker，常以為兩者執行模型相同。事實上：(1) Apptainer 預設 image 是 read-only；(2) $HOME、/tmp、CWD 預設自動 bind mount，不像 Docker 預設隔離；(3) network 預設使用 host network；(4) 不支援 Docker compose。在 HPC 上跑 Snakemake / Nextflow 時要記得 mount 點與 Dockerfile USER 指令的差異。

來源：Apptainer — Docker compatibility。

E2 Snakemake vs Nextflow：選擇取決於團隊，非效能

Wratten et al. 2021 Nat Methods 對 5 大 workflow manager 的 benchmark 中，runtime 差距 < 5%，效能並非選擇依據。實際決策因素為：(1) Snakemake 語法接近 Python，適合 Python 為主的 lab；(2) Nextflow 採 DSL2 + Groovy，nf-core 提供 80+ 預打包 pipeline，適合臨床 NGS 環境；(3) WDL + Cromwell 是 Terra / DNAnexus 生態系預設選擇。請依團隊既有技能與生態系選擇，而非單純看 benchmark 數字。

來源：Wratten et al. 2021 Nat Methods。

E4 FAIR 不等於 open

FAIR 原則（Findable / Accessible / Interoperable / Reusable）並未要求資料公開。一份內部基因體資料只要有 persistent identifier、metadata schema 與授權聲明，即可 FAIR-compliant 而仍保持 controlled access。臨床資料的 IRB / GDPR 限制與 FAIR 不衝突。對外引用 FAIR 時應避免「FAIR = open data」的誤解。

來源：GO FAIR — FAIR Principles；Wilkinson et al. 2016。
