測試策略：pytest、testthat、property-based 與 CI

沒有測試的 pipeline 就像沒有 control 的實驗——你不知道明天它會不會壞。本章教 testing pyramid、pytest fixtures、testthat snapshot、hypothesis property-based、coverage、GitHub Actions、nf-test。

A pipeline without tests is an experiment without a control — you won't know it broke tomorrow. Covers the testing pyramid, pytest fixtures, testthat snapshots, hypothesis property-based, coverage, GitHub Actions, nf-test.

動機

為什麼研究程式需要測試？

「我每次都人工目視 figure，沒問題」——這在 1 個樣本時可行，到 100 個樣本就崩潰。三類痛點：

  • Silent regression：升級 scanpy 0.10 → 0.11，sc.pp.normalize_total 預設改變，你的 UMAP 變形——但沒人發現，因為沒有測試斷言 cluster 數。
  • 邊界 bug：你的程式在 hg38 跑得好，到 mm10 因為 chrY 不存在而 crash——沒測過小鼠檔案就不知道。
  • refactor 恐懼：不敢動 6 個月前的 QC code，因為沒有測試守住既有行為。

本章六個主題：testing pyramid、pytest fixtures、testthat + snapshot、property-based testing、coverage、CI (GitHub Actions + nf-test)。

"I just eyeball every figure" — works for 1 sample, breaks at 100. Three pain points:

  • Silent regression: Upgrading scanpy 0.10 → 0.11 changes sc.pp.normalize_total defaults; your UMAP shifts — undetected, because no test asserted the cluster count.
  • Edge-case bugs: Code works on hg38 but crashes on mm10 because chrY is missing — undetected until you actually try mouse data.
  • Refactor fear: You can't touch six-month-old QC code because no tests guard the existing behaviour.

The six topics: testing pyramid, pytest fixtures, testthat + snapshots, property-based, coverage, CI (GitHub Actions + nf-test).

口訣：「測試不是要找 bug，而是要在 refactor / dependency upgrade 時讓你敢動。」 Mantra: "Tests aren't for finding bugs — they're so you can refactor and upgrade dependencies without fear."
主題 1

一、Testing Pyramid — 70/20/10 法則

金字塔從下而上：單元測試 (Unit) 多而快、整合測試 (Integration) 少而中速、端到端 (E2E) 極少而慢。健康的測試集大致 70% unit、20% integration、10% e2e。

Bottom-up: Unit tests are many and fast, Integration tests are fewer and medium, E2E tests are very few and slow. A healthy suite is roughly 70% unit, 20% integration, 10% e2e.

主題 2

二、pytest — fixtures, parametrize, monkeypatch

pytest 是 Python 生態最常用的測試框架，比 unittest 簡潔，支援 fixture (共享 setup)、parametrize (一個測試多筆資料)、monkeypatch (替換物件)、tmp_path (隔離暫存資料夾)。

pytest is the de-facto Python test framework. Lighter than unittest; first-class fixtures (shared setup), parametrize (one test, many cases), monkeypatch (replace objects), tmp_path (isolated tmp dir).

tests/test_qc.py
import pytest
from mypkg.qc import filter_cells

def test_filter_removes_low_count(tiny_adata):
    out = filter_cells(tiny_adata, min_counts=100)
    assert out.n_obs < tiny_adata.n_obs
    assert (out.obs["total_counts"] >= 100).all()

def test_filter_keeps_all_when_threshold_zero(tiny_adata):
    out = filter_cells(tiny_adata, min_counts=0)
    assert out.n_obs == tiny_adata.n_obs

def test_filter_invalid_threshold_raises(tiny_adata):
    with pytest.raises(ValueError, match="min_counts must be >= 0"):
        filter_cells(tiny_adata, min_counts=-5)
tests/conftest.py — shared fixtures
import pytest, numpy as np, anndata as ad

@pytest.fixture
def tiny_adata():
    """200 cells × 50 genes synthetic AnnData."""
    rng = np.random.default_rng(42)
    X = rng.poisson(3, size=(200, 50))
    a = ad.AnnData(X)
    a.obs["total_counts"] = X.sum(axis=1)
    return a

@pytest.fixture(scope="session")
def mini_fastq(tmp_path_factory):
    """One-time mini FASTQ used by many tests."""
    p = tmp_path_factory.mktemp("data") / "mini.fq"
    p.write_text("@r1\nACGT\n+\nIIII\n" * 100)
    return p

# Real BAM fixtures: keep TINY files (<1MB) in tests/data/ and commit them.
tests/test_parametrize.py
import pytest
from mypkg.seq import gc_content

@pytest.mark.parametrize("seq,expected", [
    ("AAAA",   0.0),
    ("GGCC",   1.0),

    ("NNNN",   0.0),
    ("",       0.0),

def test_gc_content(seq, expected):
    assert gc_content(seq) == pytest.approx(expected)

# One assertion, 5 cases. pytest reports each as a separate test.

主題 3

三、R: testthat + vdiffr snapshot

R 套件用 testthat 框架。對 ggplot2 圖形可用 vdiffr 做「視覺 snapshot test」——第一次跑會存 SVG，之後每次跑比對是否像素級相同。

R packages use testthat. For ggplot2 figures, vdiffr provides "visual snapshot tests" — first run saves an SVG; subsequent runs compare pixel-for-pixel.

tests/testthat/test-qc.R
library(testthat)
library(mypkg)

test_that("filter_cells removes low-count cells", {
  sce <- make_tiny_sce()
  out <- filter_cells(sce, min_counts = 100)
  expect_lt(ncol(out), ncol(sce))
  expect_true(all(out$total_counts >= 100))
})

test_that("negative threshold errors", {
  sce <- make_tiny_sce()
  expect_error(filter_cells(sce, min_counts = -1),
               "min_counts must be >= 0")
})

test_that("gc_content boundary cases", {
  expect_equal(gc_content("AAAA"), 0)
  expect_equal(gc_content("GGCC"), 1)
  expect_equal(gc_content(""),     0)
})
tests/testthat/test-plot.R — vdiffr snapshot
library(testthat)
library(vdiffr)

test_that("plot_umap produces stable figure", {
  sce <- readRDS(test_path("fixtures", "mini-sce.rds"))
  p <- plot_umap(sce, color_by = "cluster")

  # First run: stores tests/testthat/_snaps/plot/umap.svg
  # Later runs: compare against stored SVG, fail on diff
  expect_doppelganger("umap-by-cluster", p)
})

# When the figure is intentionally changed:
#   testthat::snapshot_review()   # interactive web UI to approve diffs
tests/testthat/helper-setup.R
# Any helper-*.R in tests/testthat/ is auto-sourced before tests
make_tiny_sce <- function(n_cell = 200, n_gene = 50) {
  set.seed(42)
  X <- matrix(rpois(n_cell * n_gene, lambda = 3),
              nrow = n_gene, ncol = n_cell)
  sce <- SingleCellExperiment::SingleCellExperiment(
    assays = list(counts = X)
  )
  sce$total_counts <- colSums(X)
  sce
}
主題 4

四、Property-based testing — Hypothesis

傳統 example-based 測試你寫 case 它跑 case；property-based 你寫「永遠成立的性質」，框架自動生成上百個輸入去找反例。Python 用 hypothesis；R 可用 hedgehog 或 quickcheck 風格。

Example-based: you write cases; the framework runs them. Property-based: you write invariants; the framework generates hundreds of inputs trying to falsify them. Python uses hypothesis; R has hedgehog or quickcheck-style libraries.

tests/test_props.py
from hypothesis import given, strategies as st, settings
from mypkg.seq import reverse_complement, gc_content

# Property 1: reverse_complement is its own inverse
@given(st.text(alphabet="ACGT", min_size=0, max_size=1000))
def test_rc_involution(seq):
    assert reverse_complement(reverse_complement(seq)) == seq

# Property 2: GC content is in [0, 1] for any valid sequence
@given(st.text(alphabet="ACGTN", min_size=1))
def test_gc_bounded(seq):
    gc = gc_content(seq)
    assert 0 <= gc <= 1

# Property 3: reverse_complement preserves length
@given(st.text(alphabet="ACGT", min_size=0, max_size=1000))
def test_rc_length_preserving(seq):
    assert len(reverse_complement(seq)) == len(seq)

# Hypothesis shrinks failing inputs to the minimal counterexample.
# Often finds bugs around empty input, N-only sequences, mixed case.
tests/testthat/test-props.R
library(testthat)
library(hedgehog)   # or: library(quickcheck)

# Property: rev_comp twice returns input
test_that("rev_comp is its own inverse", {
  gen_seq <- gen.sample(c("A","C","G","T"), size = gen.int(1000), replace = TRUE) |>
    gen.map(\(x) paste0(x, collapse = ""))

  forall(gen_seq, function(seq) {
    expect_equal(rev_comp(rev_comp(seq)), seq)
  })
})
⚠️
取捨：property-based 抓到的 bug 常是「邊界」(空輸入、全 N、超長 read)，但寫測試本身需要更多思考——適合套件級函式，不適合一次性 figure script。 Trade-off: property-based catches edge bugs (empty input, all-N, ultra-long reads), but requires more thought. Use for library-level functions, not one-off figure scripts.
主題 5

五、Coverage 與 mocking 外部服務

coverage.py (Python) 與 covr (R) 量化「哪些行被測試到」。Coverage 不等於品質——100% 行覆蓋仍可能漏 case，但 < 60% 通常表示有大片 dead zone。

coverage.py (Python) and covr (R) measure which lines are exercised. Coverage is not a quality metric — 100% line coverage can still miss cases, but < 60% usually signals large dead zones.

coverage.sh — Python
# Install
pip install pytest pytest-cov

# Run with coverage + HTML report
pytest --cov=mypkg --cov-report=html --cov-report=term-missing

# pyproject.toml fragment:
# [tool.coverage.run]
# branch = true
# source = ["mypkg"]
# [tool.coverage.report]
# fail_under = 80
# exclude_lines = ["pragma: no cover", "raise NotImplementedError"]

# Upload to codecov.io in CI
codecov -t $CODECOV_TOKEN
covr.R — R
# In R console
install.packages("covr")

# Local report
covr::report()                          # interactive HTML
covr::package_coverage()                # numeric coverage

# In CI (GitHub Actions): upload to codecov.io
covr::codecov(token = Sys.getenv("CODECOV_TOKEN"))

# Threshold check
cov <- covr::package_coverage()
stopifnot(covr::percent_coverage(cov) >= 80)

主題 6

六、CI: GitHub Actions + nf-test

本機跑得過的測試，要在 CI 也跑得過——三大主流：GitHub Actions、GitLab CI、Circle CI。對 Nextflow pipeline 額外有 nf-test，可斷言 channel 輸出、process 行為、snapshot stdout。

Local pass must equal CI pass. Three majors: GitHub Actions, GitLab CI, Circle CI. For Nextflow pipelines, nf-test asserts channel outputs, process behaviour, snapshot stdout.

.github/workflows/python.yml
name: Python tests
on: [push, pull_request]
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, macos-latest]
        python: ["3.11", "3.12"]
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v3
      - name: Install
        run: uv sync --all-extras
      - name: Lint
        run: |
          uv run ruff check .
          uv run ruff format --check .
      - name: Test
        run: uv run pytest --cov=mypkg --cov-report=xml
      - uses: codecov/codecov-action@v4
        with:
          token: ${{ secrets.CODECOV_TOKEN }}
.github/workflows/r.yml
name: R-CMD-check
on: [push, pull_request]
jobs:
  R-CMD-check:
    runs-on: ${{ matrix.config.os }}
    strategy:
      fail-fast: false
      matrix:
        config:
          - {os: ubuntu-latest,  r: 'release'}
          - {os: ubuntu-latest,  r: 'devel'}
          - {os: macos-latest,   r: 'release'}
          - {os: windows-latest, r: 'release'}
    steps:
      - uses: actions/checkout@v4
      - uses: r-lib/actions/setup-r@v2
        with: {r-version: ${{ matrix.config.r }}}
      - uses: r-lib/actions/setup-r-dependencies@v2
        with: {extra-packages: any::rcmdcheck, any::covr}
      - uses: r-lib/actions/check-r-package@v2
      - name: Coverage
        if: matrix.config.os == 'ubuntu-latest'
        run: covr::codecov()
        shell: Rscript {0}
tests/main.nf.test — nf-test
nextflow_pipeline {
    name "Test main.nf with tiny FASTQ"
    script "main.nf"

    test("single sample") {
        when {
            params {
                input  = "$projectDir/tests/data/samplesheet.csv"
                outdir = "$outputDir"
            }
        }
        then {
            assert workflow.success
            assert path("$outputDir/qc_report.html").exists()
            assert snapshot(
                path("$outputDir/counts.tsv")
            ).match()
        }
    }
}

決策樹

該寫哪種測試？

情境 → 工具對照

Q1→「測單一純函式 (gc_content, parse_cigar)」 → pytest / testthat unit test"Pure function (gc_content, parse_cigar)" → pytest / testthat unit test
Q2→「同一個函式要對許多 case 跑」 → @pytest.mark.parametrize"Same function, many inputs" → @pytest.mark.parametrize
Q3→「測會呼叫 REST API 的函式」 → monkeypatch / responses / httpx-mock"Function calls REST API" → monkeypatch / responses / httpx-mock
Q4→「測 ggplot/seaborn 圖形不會跑掉」 → vdiffr (R) / pytest-mpl (Python)"Verify a ggplot/seaborn figure stays stable" → vdiffr (R) / pytest-mpl (Python)
Q5→「想找邊界 bug」 → property-based (hypothesis / hedgehog)"Hunt for edge-case bugs" → property-based (hypothesis / hedgehog)
Q6→「測整條 Nextflow pipeline」 → nf-test，存 mini fixtures"Test whole Nextflow pipeline" → nf-test with mini fixtures

📝 自我檢測 (3 題)

Q1. Testing pyramid 健康的比例最接近？

Q1. Healthiest testing-pyramid mix?

A. 100% E2E (跑整條 pipeline 才安心)A. 100% E2E (only full pipeline matters)
B. 50% unit / 50% E2E，無 integrationB. 50% unit / 50% E2E, no integration
C. ~70% unit / 20% integration / 10% E2EC. ~70% unit / 20% integration / 10% E2E
D. 只要有 1 個 E2E 就好D. Just one E2E is enough
答案 C。E2E 慢、整批失敗，難 debug；unit 快、明確、refactor 友善。整合測試補上「跨模組」缺口。70/20/10 是常見的健康分布。
Answer: C. E2E is slow and fails in clumps; unit is fast and refactor-friendly. Integration covers cross-module gaps. 70/20/10 is the canonical balance.

Q2. 為什麼測試呼叫 REST API 的函式時要 mock？

Q2. Why mock REST calls in tests?

A. 因為 API 不能在 CI 用A. APIs aren't usable in CI
B. 測試應快速、確定性、不依賴外部服務、不消耗 quotaB. Tests must be fast, deterministic, network-free, quota-free
C. mock 比真實 API 更準確C. Mocks are more accurate than the real API
D. 法律規定不能在 CI 連網D. Law forbids network in CI
答案 B。單元測試的價值在於快速、確定、孤立。打真實 API 會：(1) 慢、(2) 不穩定 (網路掛了 CI 紅)、(3) 燒 quota、(4) 同事的 PR 也會 trigger 呼叫。Mocking 用 responses 或 monkeypatch。
Answer: B. Unit tests must be fast, deterministic, isolated. Real API hits make tests slow, flaky (red CI on network blip), burn quota, and fire on every PR. Use responses or monkeypatch.

Q3. Property-based testing (hypothesis) 跟傳統 example-based testing 最大差別？

Q3. Biggest difference between property-based (hypothesis) and example-based testing?

A. 你寫「永遠成立的性質」，框架自動生成上百輸入試圖證偽A. You declare invariants; the framework generates many inputs to try to falsify them
B. Property-based 不需要寫 assertionB. Property-based needs no assertion
C. Property-based 跑得比 unit test 快C. Property-based runs faster than unit tests
D. Property-based 只能測純函式以外的東西D. Property-based can only test side-effects
答案 A。例：你說「reverse_complement 兩次等於原序列」，hypothesis 自動生成 100 個 random DNA (含空字串、超長、混合大小寫) 來打你的程式。它擅長找邊界 bug 並把反例 shrink 到最小。
Answer: A. Example: declare "reverse_complement twice equals input"; hypothesis generates 100 random DNAs (including empty, very long, mixed case) and tries to break your code. Excellent at edge-case bugs; shrinks failures to a minimal counterexample.
