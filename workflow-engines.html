工作流引擎：Snakemake、Nextflow、WDL

當你的 shell 腳本擴展到 10,000 樣本時就會崩潰——宣告式工作流引擎讓你只描述「誰依賴誰」，由引擎負責平行化、重試、失敗續跑與雲端調度。

Shell scripts collapse at 10,000 samples — declarative workflow engines let you describe only "what depends on what" and delegate parallelization, retries, resume, and cloud scheduling to the engine.

動機

為什麼需要工作流引擎？

典型的 bulk RNA-seq pipeline 至少有 6 個步驟：QC → trim → align → quantify → DESeq2 → report。每個步驟對每個樣本都要跑一次。若你有 200 個樣本，shell 腳本要承擔：

  • 相依管理：哪個 step 必須先完成？輸出檔在哪？
  • 平行調度：哪些 step 可平行？對 SLURM / AWS Batch 怎麼提交？
  • 失敗續跑：第 150 個樣本壞了，重跑時要從哪重啟？
  • 環境隔離：每個 tool 的 conda env / Docker image 該怎麼掛？
  • 觀測性：哪些 job 慢？哪些卡住？

工作流引擎將上述全部抽象化。三大主流：Snakemake (Python 系，生命科學最廣)、Nextflow DSL2 (Groovy/Java 系，nf-core 社群，雲端最強)、WDL/Cromwell (GATK Best Practices、Broad、Terra)。

A typical bulk RNA-seq pipeline has 6+ steps: QC → trim → align → quantify → DESeq2 → report. Each step runs per sample. With 200 samples, a shell script must handle:

  • Dependency tracking: which step must finish first? where are outputs?
  • Parallel scheduling: which steps parallelize? how do we submit to SLURM / AWS Batch?
  • Resume on failure: sample 150 crashed — where do we restart?
  • Environment isolation: how do we attach each tool's conda env / Docker image?
  • Observability: which jobs are slow? which are stuck?

Workflow engines abstract all of this. Three major systems: Snakemake (Python-based, widest in life sciences), Nextflow DSL2 (Groovy/Java, the nf-core community, strongest cloud support), WDL/Cromwell (GATK Best Practices, Broad, Terra).

實作 1

Snakemake — Pythonic 規則檔

Snakefile
configfile: "config.yaml"
SAMPLES = config["samples"]

rule all:
    input: expand("results/vcf/{s}.vcf.gz", s=SAMPLES)

rule trim:
    input:  fq1="raw/{s}_R1.fq.gz", fq2="raw/{s}_R2.fq.gz"
    output: o1="trim/{s}_R1.fq.gz", o2="trim/{s}_R2.fq.gz"
    conda:  "envs/fastp.yaml"
    threads: 4
    shell:  "fastp -i {input.fq1} -I {input.fq2} -o {output.o1} -O {output.o2} -w {threads}"

rule align:
    input:  fq1=rules.trim.output.o1, fq2=rules.trim.output.o2,
            idx=config["bwa_index"]
    output: bam="bam/{s}.bam"
    conda:  "envs/bwa.yaml"
    threads: 16
    shell:  "bwa mem -t {threads} {input.idx} {input.fq1} {input.fq2} | "
            "samtools sort -@ {threads} -o {output.bam} -"

rule call_variants:
    input:  bam="bam/{s}.bam", ref=config["reference"]
    output: vcf="results/vcf/{s}.vcf.gz"
    container: "docker://broadinstitute/gatk:4.5.0"
    shell:  "gatk HaplotypeCaller -R {input.ref} -I {input.bam} -O {output.vcf}"
config.yaml
samples:
  - SRR001
  - SRR002
  - SRR003
reference: refs/hg38.fa
bwa_index: refs/hg38.fa
threads: 16
profiles/slurm/config.yaml — submit to SLURM
executor: slurm
default-resources:
  slurm_partition: "compute"
  mem_mb: 8000
  runtime: 120
set-resources:
  align:
    mem_mb: 32000
    runtime: 240
jobs: 200
use-conda: True
use-apptainer: True

# Run with: snakemake --profile profiles/slurm
💡
核心優勢：Snakemake 用「output 檔已存在 → skip」做續跑——重啟時引擎自動只跑失敗的 jobs。配合 --rerun-incomplete 處理半成品檔。 Key strength: Snakemake resumes via "output exists → skip" — engine reruns only failed jobs. Use --rerun-incomplete for partial outputs.

Nextflow DSL2 — Channel 與 nf-core

Nextflow 的核心抽象是 Channel——資料流。Process 之間靠 channel 傳資料而非靠檔案路徑相依，這讓 Nextflow 對雲端 (S3、GCS) 與容器調度極友好。nf-core 社群已維護 100+ 個生產級 pipeline (rnaseq、sarek、scrnaseq、taxprofiler)。

Nextflow's core abstraction is the Channel — a dataflow stream. Processes pass data via channels rather than file paths, making Nextflow exceptionally cloud-friendly (S3, GCS) and container-aware. The nf-core community maintains 100+ production-grade pipelines (rnaseq, sarek, scrnaseq, taxprofiler).

main.nf — DSL2 pipeline
nextflow.enable.dsl = 2

process TRIM {
    container 'biocontainers/fastp:0.23.4--h5f740d0_1'
    cpus 4
    input: tuple val(sid), path(fq1), path(fq2)
    output: tuple val(sid), path("${sid}_R1.tr.fq.gz"), path("${sid}_R2.tr.fq.gz")
    script:
    """
    fastp -i $fq1 -I $fq2 -o ${sid}_R1.tr.fq.gz -O ${sid}_R2.tr.fq.gz -w $task.cpus
    """
}

process ALIGN {
    container 'biocontainers/bwa-mem2:2.2.1--he513fc3_0'
    cpus 16; memory '32 GB'
    input: tuple val(sid), path(fq1), path(fq2)
           path  idx
    output: tuple val(sid), path("${sid}.bam")
    script:
    """
    bwa-mem2 mem -t $task.cpus $idx $fq1 $fq2 | samtools sort -@ $task.cpus -o ${sid}.bam -
    """
}

workflow {
    samples_ch = Channel.fromFilePairs('raw/*_{R1,R2}.fq.gz')
                         .map { sid, fqs -> tuple(sid, fqs[0], fqs[1]) }
    idx_ch     = Channel.value(file(params.idx))

    TRIM(samples_ch)
    ALIGN(TRIM.out, idx_ch)
}
nextflow.config — multi-environment
params {
    idx = 'refs/hg38.bwa'
    outdir = 'results'
}

profiles {
    standard { docker.enabled = true }
    slurm    { process.executor = 'slurm'; apptainer.enabled = true }
    aws {
        process.executor = 'awsbatch'
        process.queue    = 'genomics-spot'
        aws.region       = 'us-east-1'
        workDir          = 's3://my-bucket/work'
    }
}

WDL / Cromwell — Broad / GATK 生態

WDL (Workflow Description Language) 是 Broad Institute 主導的格式，由 Cromwell 引擎執行。GATK Best Practices、Terra (Broad 雲端平台)、DNAnexus 都以 WDL 為一等公民。若你做大規模 germline / somatic 變異呼叫，WDL 通常是最少抵抗路徑。

WDL (Workflow Description Language) is the Broad Institute-led format executed by Cromwell. GATK Best Practices, Terra (Broad's cloud platform), and DNAnexus all treat WDL as first-class. For large-scale germline/somatic variant calling, WDL is usually the path of least resistance.

align.wdl — WDL 1.1
version 1.1

workflow AlignAndCall {
  input {
    Array[Pair[File, File]] sample_pairs
    File ref_fa
    File ref_idx
  }
  scatter (p in sample_pairs) {
    call Trim  { input: fq1 = p.left, fq2 = p.right }
    call Align { input: fq1 = Trim.o1, fq2 = Trim.o2, ref = ref_fa, idx = ref_idx }
  }
  output { Array[File] bams = Align.bam }
}

task Align {
  input { File fq1; File fq2; File ref; File idx }
  command <<<
    bwa mem -t 16 ~{idx} ~{fq1} ~{fq2} | samtools sort -@ 16 -o out.bam -
  >>>
  runtime {
    docker: "broadinstitute/gatk:4.5.0"
    cpu: 16
    memory: "32G"
  }
  output { File bam = "out.bam" }
}
inputs.json — provide values
{
  "AlignAndCall.sample_pairs": [
    { "left": "s3://bkt/SRR001_R1.fq.gz", "right": "s3://bkt/SRR001_R2.fq.gz" }
  ],
  "AlignAndCall.ref_fa":  "s3://bkt/hg38.fa",
  "AlignAndCall.ref_idx": "s3://bkt/hg38.fa.bwt"
}

Q1. Snakemake 在重啟一個失敗的 pipeline 時，預設用什麼方式判斷哪些 job 不必重跑？

Q1. When restarting a failed Snakemake pipeline, how does it decide which jobs to skip?

A. 查看 SQLite 記錄資料庫A. Reads from a SQLite database
B. 透過 git commit hash 比對B. By git commit hash
C. 檢查 output 檔案是否存在且時間戳合理C. Checks if output files exist with sane timestamps
D. 詢問 SLURM 排程器D. Queries the SLURM scheduler
答案 C。Snakemake 用「output 已存在 → skip」做斷點續跑；若懷疑檔案不完整可用 --rerun-incomplete。
Answer: C. Snakemake resumes by checking output existence; use --rerun-incomplete if files may be partial.

Q2. 你打算在 AWS 上跑 1000 個 RNA-seq 樣本，且希望用一個已驗證的 pipeline。最少阻力的選擇？

Q2. You plan to run 1000 RNA-seq samples on AWS using a validated pipeline. Path of least resistance?

A. 自己寫 bash 加上 GNU parallelA. Write bash + GNU parallel yourself
B. Nextflow + nf-core/rnaseq + AWS Batch profileB. Nextflow + nf-core/rnaseq + AWS Batch profile
C. 用 Excel 紀錄樣本 IDC. Track samples in Excel
D. WDL 是必選 (因為它最快)D. WDL is mandatory (it's the fastest)
答案 B。nf-core/rnaseq 是社群維護、有測試、有 AWS Batch profile 的標準選擇。
Answer: B. nf-core/rnaseq is community-maintained, tested, and ships with an AWS Batch profile.

Q3. Nextflow DSL2 中，process 之間傳遞資料的核心抽象是？

Q3. In Nextflow DSL2, what is the core abstraction passed between processes?

A. 環境變數A. Environment variables
B. 共享記憶體B. Shared memory
C. SQLite 資料表C. SQLite table
D. Channel — 資料流物件D. Channel — a dataflow object
答案 D。Channel 是 Nextflow 的核心；它讓 process 之間以資料流而非檔案路徑相依，這正是 Nextflow 對雲端友好的原因。
Answer: D. Channels are Nextflow's core abstraction. Processes depend on data streams rather than file paths — the key reason Nextflow is cloud-friendly.
