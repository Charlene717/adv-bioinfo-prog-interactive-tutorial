效能剖析與最佳化：先量再改

cProfile、pyinstrument、line_profiler、memory_profiler、py-spy、profvis、bench——用真實資料找出 hot spot，再針對 80/20 那 20% 動手。

cProfile, pyinstrument, line_profiler, memory_profiler, py-spy, profvis, bench — find real hot spots on real data, then target the 20% that matters.

原則

為什麼一定要「先量再改」？

Donald Knuth 的名言：「Premature optimization is the root of all evil.」生醫程式中最常見的浪費是花三天把一個只佔 0.5% 總時間的迴圈優化成 C 擴充——卻沒注意到 92% 的時間其實花在某次重複的 pd.merge。

  • 80/20 法則：絕大多數程式的 80% 時間花在 20% 的程式碼。先用 profiler 抓出那 20%。
  • 三種瓶頸：CPU-bound（計算密集）、I/O-bound（讀檔/網路）、Memory-bound（命中 swap 或 OOM）。三者需要不同工具。
  • 真實資料才準：用 toy data 跑 profiler 經常誤導——10 個 cell 跟 100 萬個 cell 的瓶頸位置不同。

Knuth's famous line: "Premature optimization is the root of all evil." The most common bioinformatics waste is spending three days hand-rolling a C extension for a loop that's 0.5% of runtime — while missing the repeated pd.merge that consumes 92%.

  • 80/20 rule: Most programs spend 80% of time in 20% of code. Use a profiler to find that 20%.
  • Three bottleneck types: CPU-bound, I/O-bound, memory-bound (paging / OOM). Each needs different tools.
  • Profile on real data: Toy datasets mislead — the bottleneck for 10 cells differs from 1M cells.
💡
工作流程：(1) 用 wall-clock 找出哪一步最慢 → (2) 用 deterministic profiler (cProfile) 或 sampling profiler (pyinstrument / py-spy) 找熱點函式 → (3) 用 line_profiler 鎖定熱點函式內哪幾行最貴 → (4) 動手改 → (5) 再量一次確認真的變快。 Workflow: (1) Wall-clock to find the slowest step → (2) deterministic profiler (cProfile) or sampling profiler (pyinstrument / py-spy) to find hot functions → (3) line_profiler to pinpoint expensive lines inside → (4) optimize → (5) re-measure to confirm.

一、Python 剖析工具一覽

四個層次：函式級時間（cProfile / pyinstrument）、行級時間（line_profiler）、記憶體（memory_profiler / tracemalloc）、production 採樣（py-spy）。生醫工作流幾乎一定會用到前三個。

Four levels: function-level time (cProfile / pyinstrument), line-level time (line_profiler), memory (memory_profiler / tracemalloc), production sampling (py-spy). Bioinformatics workflows almost always touch the first three.

cprofile.py — built-in deterministic profiler
import cProfile, pstats
from pstats import SortKey

def main():
    adata = sc.read_h5ad("pbmc_3k.h5ad")
    sc.pp.normalize_total(adata)
    sc.pp.log1p(adata)
    sc.pp.highly_variable_genes(adata, n_top_genes=2000)
    sc.pp.pca(adata, n_comps=50)
    sc.pp.neighbors(adata)
    sc.tl.leiden(adata)

# Run with profiling
cProfile.run("main()", "pipeline.prof")

# Inspect top 15 cumulative-time functions
p = pstats.Stats("pipeline.prof")
p.sort_stats(SortKey.CUMULATIVE).print_stats(15)

# Visualize:
#   pip install snakeviz
#   snakeviz pipeline.prof

line_profiler.py — per-line timing of hot fns
# pip install line_profiler
# Add @profile to functions you want to inspect (no import needed)

@profile
def compute_qc_metrics(adata):
    adata.var["mt"] = adata.var_names.str.startswith("MT-")        # 0.1%
    sc.pp.calculate_qc_metrics(adata, qc_vars=["mt"], inplace=True) # 78%  ← hot
    adata = adata[adata.obs.pct_counts_mt < 15].copy()           # 21%  ← copy() expensive
    return adata

# Run:
#   kernprof -lv my_script.py
# Output shows time/hits per line; you'd then attack the 78% line.
pyspy.sh — production sampling (no code change)
# Install: pip install py-spy (or cargo install py-spy)

# Attach to a running PID, dump flamegraph after 60s
py-spy record -o flame.svg --duration 60 --pid 12345

# Live "top"-like view of a running process
py-spy top --pid 12345

# Inside a Docker container (need --cap-add SYS_PTRACE)
docker run --cap-add SYS_PTRACE my-pipeline py-spy record -o /out/flame.svg -- python run.py
⚠️
採樣 vs 確定性的差異：cProfile 記下每次呼叫，對呼叫頻繁的小函式會放大 overhead；pyinstrument/py-spy 每 1–10ms 抓一次 stack，overhead 低但會漏掉 < 1ms 的快函式。短熱點用 cProfile，長 pipeline 用 pyinstrument。 Sampling vs deterministic: cProfile records every call — overhead balloons for tiny frequently-called functions. pyinstrument / py-spy sample every 1–10ms — low overhead but miss sub-millisecond functions. Use cProfile for short hot spots, pyinstrument for long pipelines.
主題 2

二、記憶體剖析：抓 OOM 與洩漏

scRNA-seq、整段基因組 VCF 都是 RAM hog。memory_profiler 給每行記憶體變化；tracemalloc（stdlib）做 snapshot diff 找洩漏；psutil 監控整個 process。

scRNA-seq and whole-genome VCFs are RAM hogs. memory_profiler shows per-line deltas; tracemalloc (stdlib) takes snapshot diffs to find leaks; psutil tracks whole-process RSS.

tracemalloc.py — find allocation deltas
import tracemalloc

tracemalloc.start(25)  # keep 25-frame stack per allocation

snap1 = tracemalloc.take_snapshot()
adata = run_pipeline()
snap2 = tracemalloc.take_snapshot()

# Top 10 lines that allocated the most NEW memory
for stat in snap2.compare_to(snap1, "lineno")[:10]:
    print(stat)

# Output example:
#   anndata/_core/views.py:88: size=412 MB (+412 MB)
#   scanpy/preprocessing/_pca.py:172: size=87 MB (+87 MB)
memprof.py — per-line RAM
# pip install memory_profiler

@profile
def build_count_matrix(reads):
    counts = pd.DataFrame()                # 0 MB
    for r in reads:                       # 0 MB
        row = pd.DataFrame([r.to_dict()])  # +12 MB
        counts = pd.concat([counts, row])  # +2.3 GB ← O(n²) RAM!
    return counts

# Run:  python -m memory_profiler script.py
# The fix: build a list of dicts, then DataFrame at the end.
psutil.py — whole-process monitoring
import psutil, os, time, threading

def memory_watch(interval=1.0):
    p = psutil.Process(os.getpid())
    peak = 0
    def _loop():
        nonlocal peak
        while True:
            rss = p.memory_info().rss / 1e9
            peak = max(peak, rss)
            time.sleep(interval)
    threading.Thread(target=_loop, daemon=True).start()
    return lambda: peak

get_peak = memory_watch()
run_pipeline()
print(f"Peak RSS: {get_peak():.1f} GB")

三、R 的剖析工具

R 端三大工具：profvis（互動 flamegraph，Rprof 的視覺化）、bench::mark()（比較多種寫法，回報時間+記憶體+GC）、microbenchmark（單純計時，nano 級）。lobstr::obj_size() 用來看 R 物件真實大小（會展開 promise 與 SEXP overhead）。

Three R tools: profvis (interactive flamegraph wrapping Rprof), bench::mark() (compare implementations with time + memory + GC), microbenchmark (simple nano-second timing). lobstr::obj_size() reports the true size of an R object (unwrapping promises and SEXP overhead).

bench.R — compare implementations with mem + GC
library(bench)
library(data.table)

mat <- matrix(rnorm(1e6), 1000, 1000)

bench::mark(
  base   = colSums(mat),
  matrix = .colSums(mat, nrow(mat), ncol(mat)),
  apply  = apply(mat, 2, sum),
  check  = TRUE,
  iterations = 50
)

lobstr.R — true object size
library(lobstr)

# object.size() lies for shared objects; obj_size accounts for sharing
x <- 1:1e6
y <- list(x, x, x)
object.size(y)   # 24,000,200 bytes (3 × x — wrong)
obj_size(y)      # 4,000,104 bytes  (1 × x — correct)

# Compare base data.frame vs data.table vs tibble
df <- data.frame(a = 1:1e6, b = rnorm(1e6))
dt <- data.table::as.data.table(df)
tb <- tibble::as_tibble(df)
obj_sizes(df, dt, tb)
⚠️
R 的 copy-on-modify 陷阱：修改 list 中的一個元素其實會複製整個 list；給 data.frame 加一欄會複製整個 data.frame。tracemem(x) 可印出每次複製的位置，data.table 的 := 才是真正的 in-place 寫入。 R copy-on-modify gotcha: Modifying one element of a list copies the entire list; adding a column to a data.frame copies the whole data.frame. tracemem(x) prints every copy event; only data.table's := is truly in-place.

四、讀懂 flame graph

Flame graph 的橫軸是「相對時間佔比」（不是時間順序），縱軸是 call stack 深度。找寬的、不找高的——寬代表花的時間多。下方互動圖可切換 hot levels 看你會先優化哪一層。

In a flame graph the x-axis is relative time (not timeline order) and the y-axis is call-stack depth. Look wide, not tall — width = time. Toggle hot levels below to see which layer you'd attack first.

主題 4

五、生醫程式三大反模式

下方三段示意是真實 PR review 中最常見的「跑很慢的元兇」：

The three classes below are the most common "why is this so slow?" findings in real PR reviews:

① DataFrame.append in loop
每次 pd.concat 都複製整張表 → O(n²)。一萬列要 30 秒，十萬列要 50 分鐘。
修正：累積成 list[dict]，最後一次 pd.DataFrame(rows)。Pandas 2.0 已將 DataFrame.append 移除。
Every pd.concat copies the whole frame → O(n²). 10K rows: 30s; 100K: 50 minutes.
Fix: Accumulate list[dict], build pd.DataFrame(rows) once. Pandas 2.0 removed DataFrame.append.
② R 中 for-loop 改寫 data.frame 欄
df$new <- f(df$x) 在 loop 中會反覆 copy-on-modify。一千次迴圈可能浪費幾 GB。
修正：vectorize（一次處理整欄）；不得已就用 data.table 的 :=。
df$new <- f(df$x) in a loop forces repeated copy-on-modify; 1K iterations can waste several GB.
Fix: Vectorize (whole-column at once); fallback to data.table's :=.
③ 字串相加組 FASTA
s = s + line 在 Python 中是 O(n²)（CPython 偶有優化但別賴）。
修正：用 list.append，最後 "".join(parts)；或直接 file.write(line) 串流寫出。
s = s + line in Python is O(n²) (CPython sometimes optimizes — don't rely on it).
Fix: Use list.append then "".join(parts); or stream file.write(line) directly.
🚫
Big-O 提醒：scRNA-seq 動輒幾百萬細胞 × 兩萬基因。O(n²) 演算法在這規模等同永遠跑不完。優化前先在腦中算一次：「如果 n 變 10 倍，這函式會慢幾倍？」 Big-O reminder: scRNA-seq is millions of cells × tens of thousands of genes. An O(n²) algorithm at this scale effectively never finishes. Before optimizing, ask: "If n grows 10×, how much slower does this get?"
決策樹

該用哪個 profiler？

情境 → 工具對照

Q1→「我只想知道哪個函式最慢」 → pyinstrument (Py) / profvis (R)"I just want the slowest function" → pyinstrument (Py) / profvis (R)
Q2→「我已知哪個函式慢，要看裡面哪幾行」 → line_profiler"I know the function — which lines inside?" → line_profiler
Q3→「我的 process OOM 了」 → tracemalloc snapshot diff / memory_profiler"My process OOM'd" → tracemalloc diff / memory_profiler
Q4→「Production 上的 daemon 卡住，不能改程式碼」 → py-spy --pid"Production daemon is stuck, can't touch code" → py-spy --pid
Q5→「我想比較三種寫法」 → bench::mark (R) / timeit 或 perfplot (Py)"Compare three implementations" → bench::mark (R) / timeit or perfplot (Py)
Q6→「R 中物件到底多大」 → lobstr::obj_size() (跨 promise/共享真正準)"How big is this R object really?" → lobstr::obj_size() (handles sharing)

📝 自我檢測 (3 題)

Q1. 你的 pipeline 跑 8 小時，flame graph 顯示 pd.concat 佔 65%、sklearn.PCA 佔 12%、plt.savefig 佔 0.4%。應先優化哪個？

Q1. An 8-hour pipeline. Flame graph says pd.concat = 65%, sklearn.PCA = 12%, plt.savefig = 0.4%. Which to optimize first?

A. 三個都要優化才公平A. Optimize all three equally
B. 從最簡單的 plt.savefig 開始建立信心B. Start with the easy one, plt.savefig
C. 先攻 pd.concat（佔 65%，潛在加速最大）C. Attack pd.concat first (65% — biggest payoff)
D. 改用更快的 sklearn 版本D. Upgrade sklearn for faster PCA
答案 C。80/20 法則：先打最寬的那一塊。即使把 PCA 變零秒，總時間也只省 12%；把 65% 砍半，全 pipeline 縮 32%。
Answer: C. 80/20 — go for the widest bar. Cutting PCA to zero saves only 12%; halving the 65% saves 32% of total runtime.

Q2. 在迴圈中持續 df = pd.concat([df, new_row])，10 萬列耗時 50 分鐘。為何如此慢？

Q2. Repeatedly df = pd.concat([df, new_row]) in a loop — 100K rows takes 50 minutes. Why?

A. pandas 沒裝最新版A. Pandas isn't on the latest version
B. 每次 concat 都複製整張 DataFrame，總成本 O(n²)B. Each concat copies the whole DataFrame — total cost O(n²)
C. Python GIL 在 loop 中拖慢C. Python GIL slows the loop
D. 應該改用 NumPy arrayD. Should switch to NumPy arrays
答案 B。Pandas 為了保持 contiguous memory，每次 concat 都會配置新陣列並複製舊資料。第 k 次成本 ≈ k，總和 ≈ n²。改用 list.append 累積 dict、最後一次建表，成本變 O(n)。
Answer: B. Pandas keeps memory contiguous, so each concat allocates a new array and copies. The k-th concat costs ≈ k, total ≈ n². Collect dicts in a list and build the DataFrame once → O(n).

Q3. 哪個工具最適合「附加到正在 production 跑的 Python process，找出它為什麼卡住，且不能停掉它」？

Q3. Best tool for "attach to a running production Python process, see why it's stuck, without stopping it"?

A. py-spy record --pidA. py-spy record --pid
B. cProfile.run()B. cProfile.run()
C. line_profilerC. line_profiler
D. tracemallocD. tracemalloc
答案 A。py-spy 是 sampling profiler，靠 ptrace 從外部讀 stack，不需要修改也不需重啟目標程式，overhead <1%，是 production 首選。其他三個都得改 code 或重跑。
Answer: A. py-spy is a sampling profiler that reads stacks via ptrace — no code change, no restart, <1% overhead. The others all require code modification or a fresh run.
