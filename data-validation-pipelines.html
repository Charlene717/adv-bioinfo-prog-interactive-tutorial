資料驗證與 ETL：別讓髒資料毀掉一篇論文

Differential expression 跑出來看不出 batch effect？clinical metadata 裡的 age 欄位有 "—" 與 -999？整篇分析最致命的 bug 通常在「資料還沒進統計模型之前」就埋好了。本章帶你用 Pandera、great_expectations、pointblank、validate 把資料污染擋在門外。

Why does differential expression look fine but the batch effect is invisible? Why does the age column have both "—" and -999? Most fatal bugs are baked in before the statistical model even runs. This chapter shows how Pandera, great_expectations, pointblank, and validate keep dirty data out.

動機

garbage in, garbage out — 但你會發現嗎？

真實世界的生醫資料天生帶有四類污染：

  • Missing：缺失值不只是 NA，還可能是 "", " ", "—", -999, "NULL"。
  • Outlier：RIN 值 = 25 (應 ≤ 10)、tumor purity = 1.4 (應 0-1)、age = 700。
  • Wrong dtype："chr1" 跑進了應該是整數的 position 欄；2024 vs "2024-01-01" 在同一欄混用。
  • Impossible value：survival days = -3、batch 標籤含 typo (B1 vs b1 vs B_1)。

沒有驗證層，這些值會直接污染下游：DE 列出的 top gene 是 batch effect；某個 cohort 的中位生存期是錯誤的；reviewer 在審稿期間發現一個 "—" → 整篇要重跑。schema-first 思維就是把這些檢查「寫成 code」放進 pipeline。

Real-world bio data carries four types of pollution:

  • Missing: Not just NA — also "", " ", "—", -999, "NULL".
  • Outlier: RIN = 25 (should be ≤10), tumor purity = 1.4 (should be 0-1), age = 700.
  • Wrong dtype: "chr1" lands in an integer position column; 2024 mixed with "2024-01-01" in one column.
  • Impossible value: survival days = -3; batch label with typos (B1 vs b1 vs B_1).

Without validation, these values pollute everything downstream: the top DE gene is a batch effect; a cohort's median survival is wrong; a reviewer spots a "—" mid-review → entire paper re-runs. Schema-first thinking means writing these checks as code in the pipeline itself.

💡
原則：raw → clean 之間一定要有一個明確的 validation step；validation fail 應該中斷 pipeline，而不是「警告繼續」。錯誤是訊號，不是 noise。 Principle: Between raw and clean, there must be an explicit validation step. A validation failure should halt the pipeline, not "warn and continue." Errors are signal, not noise.
主題 1

一、Pandera — DataFrame schema-as-code

Pandera 讓你用 Python class 描述 DataFrame 應該長什麼樣：欄位、型別、值域、相互關係。可以驗證 pandas、polars、Dask；錯誤訊息會列出所有違規列。是「pytest 在資料層的對應物」。

Pandera lets you describe a DataFrame's expected shape as a Python class: columns, dtypes, value ranges, cross-column rules. Validates pandas, polars, and Dask. Error messages list every offending row. Think of it as "pytest for data."

schema.py — declare expected shape
import pandera as pa
from pandera import Column, Check, DataFrameSchema

clinical_schema = DataFrameSchema({
    "sample_id": Column(str, Check.str_matches(r"^S\d{4}$"), unique=True),
    "age":       Column(int, Check.in_range(0, 120)),
    "sex":       Column(str, Check.isin(["M", "F"])),
    "tumor_purity": Column(float, Check.in_range(0.0, 1.0),
                              nullable=True),
    "condition": Column(str, Check.isin(["tumor", "normal"])),
    "batch":     Column(str, Check.str_matches(r"^B\d+$")),
    "rin":       Column(float, Check.in_range(0, 10), nullable=True),
    "date_collected": Column("datetime64[ns]"),
})

import pandas as pd
df = pd.read_csv("clinical.csv", parse_dates=["date_collected"])
clinical_schema.validate(df, lazy=True)   # raise with ALL errors
checks.py — cross-column / custom
from pandera import Check, DataFrameSchema, Column

# Custom check: tumor samples must have purity present
def _tumor_needs_purity(df):
    tumor = df[df.condition == "tumor"]
    return tumor.tumor_purity.notna().all()

clinical_schema = DataFrameSchema(
    columns={
        "condition":    Column(str, Check.isin(["tumor","normal"])),
        "tumor_purity": Column(float, nullable=True),
    },
    checks=[
        Check(_tumor_needs_purity,
              error="every tumor sample must have a tumor_purity value"),
    ],
)
decorator.py — guard a function
import pandera as pa

@pa.check_input(clinical_schema)
@pa.check_output(  # also validate the output

                     "log2fc":    Column(float),
                     "padj":      Column(float, Check.in_range(0, 1))})
)
def run_de(meta: pd.DataFrame) -> pd.DataFrame:

hypothesis.py — generate test data
import pandera as pa

# Pandera schemas can produce a hypothesis strategy automatically.

二、great_expectations、pointblank、validate

Pandera 偏向「程式內 schema」。當資料工程規模再大、需要 產生 HTML data docs、追蹤 expectations 隨時間變化、生成 profile 報告，three more tools shine:

  • great_expectations (Python) — 企業級 data quality 框架，產出 docs、可串入 Airflow / dbt。
  • pointblank (R) — R 端對應物，產出漂亮的 validation report；CRAN 套件。
  • validate (R) — 學術派、規則語言豐富，內建 confront / cells / values 三層檢查。

Pandera fits "in-process schema." At larger data-engineering scale — HTML data docs, expectation drift over time, profiling reports — three more tools shine:

  • great_expectations (Python) — enterprise-grade DQ framework, generates docs, plugs into Airflow / dbt.
  • pointblank (R) — R counterpart, beautiful validation reports; CRAN package.
  • validate (R) — academic-style, rich rule language, confront / cells / values layers.
ge.py — great_expectations
import great_expectations as gx

context = gx.get_context()
suite   = context.add_or_update_expectation_suite("clinical_v1")

batch = context.get_batch({"datasource_name":"my_pandas",
                           "runtime_parameters":{"batch_data": df},
                           "batch_identifiers":{"default":"v1"}})

batch.expect_column_values_to_be_between("age", 0, 120)
batch.expect_column_values_to_be_in_set("sex", ["M", "F"])
batch.expect_column_values_to_match_regex("sample_id", r"^S\d{4}$")
batch.expect_table_row_count_to_be_between(10, 10000)

result = batch.validate()
context.build_data_docs()   # browser-viewable HTML report

三、ETL 設計模式：idempotent、retries、dead-letter queue、lineage

把 raw data → analysis-ready 拆成 Extract → Transform → Load。三個關鍵原則：

  • Idempotency：同一 input 跑 1 次 vs 跑 100 次的最終狀態必須相同。用 UPSERT、deterministic 檔名、或 hash-based 去重。
  • Retries with backoff：網路 / API 必失敗，要設 tenacity 或 Airflow retry policy。
  • Dead-letter queue (DLQ)：無法處理的紀錄應該寫入單獨檔案 (而非 crash 全 pipeline)。
  • Data lineage：記錄「這份結果由哪些 raw + commit + parameters 產生」。OpenLineage 是開源標準。

Split raw → analysis-ready into Extract → Transform → Load. Three key principles:

  • Idempotency: Running once vs 100× must leave the same final state. Use UPSERT, deterministic filenames, or hash-based dedup.
  • Retries with backoff: Network/API will fail — set tenacity or Airflow retry policies.
  • Dead-letter queue (DLQ): Unprocessable records go to a separate file (not crash the whole pipeline).
  • Data lineage: Record "which raw inputs + commit + parameters produced this result." OpenLineage is the open standard.
etl_skeleton.py
from tenacity import retry, stop_after_attempt, wait_exponential
import pandera as pa, logging

log = logging.getLogger(__name__)

def extract(src) -> pd.DataFrame:
    raw = pd.read_csv(src)
    log.info(f"extract: {len(raw)} rows from {src}")
    return raw

def transform(raw: pd.DataFrame) -> pd.DataFrame:
    clean = (raw
             .replace({"—": pd.NA, "NULL": pd.NA, "": pd.NA, -999: pd.NA})
             .dropna(subset=["sample_id"])
             .astype({"age": "Int64"}))
    clinical_schema.validate(clean, lazy=True)   # <-- validation gate
    return clean

@retry(stop=stop_after_attempt(3),
       wait=wait_exponential(1, 2, 30))
def load(clean: pd.DataFrame, dst: str):
    clean.to_parquet(dst, index=False)
    log.info(f"load: {len(clean)} rows -> {dst}")

# orchestrate
load(transform(extract("raw/clinical.csv")), "clean/clinical.parquet")
idempotent.py — safe to re-run
import hashlib, pathlib

def output_path(input_path: str, params: dict) -> pathlib.Path:
    """Deterministic filename = same inputs ⇒ same output path."""
    h = hashlib.sha256()
    h.update(pathlib.Path(input_path).read_bytes())
    h.update(repr(sorted(params.items())).encode())
    return pathlib.Path(f"out/{h.hexdigest()[:12]}.parquet")

def run_step(input_path: str, params: dict):
    out = output_path(input_path, params)
    if out.exists():
        log.info(f"skip — {out} already produced")
        return out
    df = transform(extract(input_path))
    df.to_parquet(out)
    return out

# Re-run the pipeline 100x → only the first time does work.
dlq.py — don't crash on bad rows
import pandera as pa

def validate_with_dlq(df, schema, dlq_path: str):
    try:
        ok = schema.validate(df, lazy=True)
        return ok, 0
    except pa.errors.SchemaErrors as err:
        bad_idx = err.failure_cases.index.unique()
        bad  = df.iloc[bad_idx]
        good = df.drop(index=bad_idx)
        bad.to_csv(dlq_path, index=False)
        log.warning(f"{len(bad)} rows sent to DLQ {dlq_path}")
        return good, len(bad)

clean, n_bad = validate_with_dlq(raw, clinical_schema, "dlq/clinical_bad.csv")
lineage.py — record provenance
import json, subprocess, datetime as dt, hashlib, pathlib

def write_lineage(out_path: pathlib.Path, inputs: list[str], params: dict):
    git_rev = subprocess.check_output(["git","rev-parse","HEAD"]).strip().decode()
    rec = {
        "output":    str(out_path),
        "output_sha": hashlib.sha256(out_path.read_bytes()).hexdigest(),
        "inputs":    [{"path":p,
                       "sha": hashlib.sha256(pathlib.Path(p).read_bytes()).hexdigest()}
                      for p in inputs],
        "params":    params,
        "git":       git_rev,
        "timestamp": dt.datetime.now(dt.timezone.utc).isoformat(),
    }
    (out_path.with_suffix(".lineage.json")).write_text(json.dumps(rec, indent=2))

# OpenLineage (https://openlineage.io) standardises this for Airflow/dbt/Spark.

四、實例：differential expression 前的 clinical metadata 驗證

下面這段是真實 RNA-seq pipeline 的「驗證關卡」：clinical metadata 必須通過 schema、batch 必須與 condition 不混淆 (沒有 confounding)、count matrix 樣本名必須與 metadata 對齊。

Real-world "validation gate" before a RNA-seq DE analysis: clinical metadata must pass schema, batch must not confound condition, and count-matrix sample names must align with metadata.

before_de.py
import pandas as pd, pandera as pa
from pandera import Column, Check, DataFrameSchema

meta_schema = DataFrameSchema({
    "sample_id": Column(str, unique=True),
    "condition": Column(str, Check.isin(["tumor","normal"])),
    "batch":     Column(str),
    "age":       Column(int, Check.in_range(0, 120)),
})

def no_perfect_confounding(df):
    # every batch must contain both tumor and normal
    by_batch = df.groupby("batch")["condition"].nunique()
    return (by_batch >= 2).all()

meta_schema_full = meta_schema.add_checks([
    Check(no_perfect_confounding,
          error="some batch contains only one condition — batch confounds condition")
])

meta   = pd.read_csv("clinical.csv")
counts = pd.read_csv("counts.tsv", sep="\t", index_col=0)
meta_schema_full.validate(meta, lazy=True)
assert set(counts.columns) == set(meta.sample_id), "count/meta mismatch"
# Only now do DESeq2 / edgeR / limma

Q1. 下列何者最適合 pydantic 而非 Pandera？

Q1. Which is best suited for pydantic rather than Pandera?

A. 5000 列 RNA-seq metadata 表的欄位驗證A. Column-level validation of a 5000-row RNA-seq metadata table
B. 從 YAML 載入的 pipeline 設定檔 (cutoffs、paths、formula)B. Pipeline config loaded from YAML (cutoffs, paths, formula)
C. 50GB 的變異註釋 DataFrameC. 50GB variant-annotation DataFrame
D. great_expectations 的 HTML data docsD. great_expectations HTML data docs
答案 B。pydantic 是「單筆物件 / 設定檔級別」的驗證；DataFrame 級驗證選 Pandera。great_expectations 是另一種 DataFrame 工具，並非 pydantic 對應物。
Answer: B. pydantic validates "single objects / configs"; DataFrame-level work belongs to Pandera. great_expectations is another DataFrame tool, not a pydantic peer.

Q2. idempotent ETL step 的核心特性是？

Q2. What's the defining property of an idempotent ETL step?

A. 一定要用 Spark 才能達到A. Achievable only with Spark
B. 自動清理所有舊輸出B. Auto-deletes all old outputs
C. 同樣 input + params 跑 1 次與 100 次的最終狀態相同C. Same inputs+params produce the same final state on 1× or 100× runs
D. 必須儲存在資料庫而非檔案D. Must be stored in a DB, not files
答案 C。Idempotency = 結果只取決於 input + params；重跑不會造成資料重複、狀態漂移。典型實現是 UPSERT、deterministic 檔名、或 hash 命名。
Answer: C. Idempotency means results depend only on inputs + params; re-runs don't duplicate or drift state. Typical implementations: UPSERT, deterministic filenames, or hash-based naming.

Q3. 你的 clinical metadata 表有少部分壞列 (age = -999)。最合理的處理流程是？

Q3. Your clinical metadata has a few bad rows (age = -999). The right handling is?

A. 用 Pandera 驗證；壞列轉到 DLQ；好列繼續分析，並 log 出污染數A. Validate with Pandera; bad rows → DLQ; good rows continue; log contamination count
B. 把 -999 改成 0 就好B. Just replace -999 with 0
C. 直接跑 DE，反正只有少數列C. Run DE anyway, it's only a few rows
D. 不要驗證，相信原始資料D. Skip validation; trust the raw data
答案 A。正確處理：(1) 明確 schema 抓到問題；(2) 壞列另存 DLQ 以便人工檢查；(3) 好列才進下游；(4) log 與 lineage 留下記錄。B 是悄悄篡改資料，C/D 是 wishful thinking。
Answer: A. The right flow: (1) schema catches the issue; (2) bad rows go to a DLQ for human review; (3) good rows feed downstream; (4) log + lineage keep a record. B silently fabricates; C/D are wishful thinking.
