大型資料表處理：Polars、DuckDB、Dask、Spark

當 pandas 撐到 10GB 就 OOM、dplyr 撐到 1 億列就跑半天——你要的不是更大的機器，是更聰明的工具：lazy execution、columnar engine、out-of-core、distributed compute。

When pandas OOMs at 10GB and dplyr crawls at 100M rows — you don't need a bigger machine, you need smarter tools: lazy execution, columnar engines, out-of-core processing, distributed compute.

動機

為什麼 pandas / dplyr 撐不住了？

pandas 與 dplyr 的設計前提是「資料能完整放入記憶體」。一旦資料量超過 RAM / 2 (pandas 在中間步驟會複製)，你會碰到：

  • OOM crash：MemoryError 或被 OS 殺掉。
  • 單核心瓶頸：pandas 預設只用一個 CPU 核心，64 核機器形同浪費。
  • Eager 執行：每一步都立即實體化中間結果，無法整體最佳化。

四類解法：Polars (in-RAM 但 Rust + lazy + columnar)、DuckDB (in-process OLAP、SQL on Parquet)、Dask / Arrow Dataset (out-of-core，磁碟分塊)、Spark / Hail / Glow (跨機器分散式)。

pandas and dplyr assume the dataset fits in RAM. Once you exceed RAM/2 (pandas duplicates during intermediate steps), you hit:

  • OOM crash: MemoryError or the OS kills the process.
  • Single-core bottleneck: pandas uses one CPU by default — a 64-core box sits idle.
  • Eager execution: Every step materializes intermediates, blocking whole-query optimization.

Four solution classes: Polars (in-RAM Rust + lazy + columnar), DuckDB (in-process OLAP, SQL on Parquet), Dask / Arrow Dataset (out-of-core, disk chunks), Spark / Hail / Glow (multi-machine distributed).

💡
實務優先順序：單機 < 50GB → Polars 或 DuckDB；單機 50GB-1TB → DuckDB on Parquet 或 Dask；跨機器 / variant calling 全基因組 → Spark / Hail。Spark 不一定總是答案。 Pragmatic order: Single machine <50GB → Polars or DuckDB; 50GB-1TB → DuckDB on Parquet or Dask; multi-machine / whole-genome variant work → Spark/Hail. Spark is not always the answer.
主題 1

一、Polars — Rust + lazy 的新世代 DataFrame

Polars 用 Rust 寫核心，原生多執行緒、Arrow 記憶體格式、SIMD 向量化。最大的概念差異是 lazy mode：你寫 LazyFrame 操作會先建立 query plan，再一次最佳化執行 (像 SQL 的 query optimizer)。實測 5-20× 快於 pandas，且用同等 RAM。

Polars's Rust core is natively multithreaded, Arrow-formatted, and SIMD-vectorized. The big conceptual shift is lazy mode: operations on a LazyFrame build a query plan that is optimized as a whole before execution (like a SQL optimizer). Empirically 5-20× faster than pandas at similar RAM.

eager.py — basic API
import polars as pl

df = pl.read_csv("variants.tsv", separator="\t")
out = (df
       .filter(pl.col("qual") > 30)
       .group_by("chrom")
       .agg([
           pl.col("pos").count().alias("n"),
           pl.col("qual").mean().alias("mean_q"),
       ])
       .sort("chrom"))
print(out)
lazy.py — query plan, then collect
import polars as pl

# scan_*  → LazyFrame (no I/O yet)
lf = pl.scan_parquet("variants/*.parquet")

plan = (lf
        .filter(pl.col("qual") > 30)
        .join(pl.scan_parquet("gene_annotations.parquet"),
              on=["chrom", "pos"], how="left")
        .group_by("gene_id")
        .agg(pl.col("qual").mean()))

print(plan.explain())   # inspect optimized plan
result = plan.collect()  # NOW it runs (with predicate/projection pushdown)
streaming.py — bigger than RAM
import polars as pl

# streaming=True → process chunks, never load all at once
result = (pl.scan_csv("huge_variants.tsv.gz", separator="\t")
          .filter(pl.col("qual") > 30)
          .group_by("chrom")
          .agg(pl.col("pos").count())
          .collect(streaming=True))    # <-- key flag
vs_pandas.py — side by side
# Same query: count variants per chrom with qual > 30

# pandas
import pandas as pd
df = pd.read_csv("v.tsv", sep="\t")
df[df.qual > 30].groupby("chrom").size()

# polars (eager)
import polars as pl
df = pl.read_csv("v.tsv", separator="\t")
df.filter(pl.col("qual")>30).group_by("chrom").len()

# polars (lazy) — ~10× faster on 100M rows
(pl.scan_csv("v.tsv", separator="\t")
   .filter(pl.col("qual")>30)
   .group_by("chrom").len()
   .collect())
主題 2

二、DuckDB — In-process 的 SQL 神器

DuckDB 是「SQLite for analytics」：嵌入式、零組態、單檔。但骨子裡是 columnar OLAP 引擎，能對 Parquet / CSV / Arrow 直接下 SQL，跳過資料庫的 ETL 環節。最強的兩個特點：

  • SQL on files：SELECT * FROM 'variants/*.parquet' 直接跑，不必先 import。
  • Zero-copy with Pandas/Polars/Arrow：把 DataFrame 當 SQL 表查詢。

DuckDB is "SQLite for analytics" — embedded, zero-config, single file. But internally it's a columnar OLAP engine that runs SQL directly on Parquet/CSV/Arrow, skipping the usual DB ETL. Two killer features:

  • SQL on files: SELECT * FROM 'variants/*.parquet' works directly — no import needed.
  • Zero-copy with Pandas/Polars/Arrow: query a DataFrame as if it were a SQL table.
basic.py
import duckdb

# In-memory by default; pass a filename for persistent DB
con = duckdb.connect("genomics.duckdb")

con.execute("""
    CREATE TABLE variants AS
    SELECT * FROM read_csv_auto('variants.tsv', delim='\t')
""")

# Window function — count variants per 100kb bin per chrom
df = con.execute("""
    SELECT chrom,
           pos // 100000 AS bin,
           COUNT(*) AS n_var,
           AVG(qual) AS mean_q
    FROM variants
    WHERE qual > 30
    GROUP BY chrom, bin
    ORDER BY chrom, bin
""").df()  # .df() converts to pandas zero-copy via Arrow
sql_files.py — query files directly
import duckdb

# Parquet on disk → SQL → pandas, no import step
duckdb.sql("""
    SELECT chrom, COUNT(*) AS n
    FROM 'variants_partitioned/*/*.parquet'
    WHERE qual > 30
    GROUP BY chrom
""").df()

# CSV.gz on S3, no download:
duckdb.sql("INSTALL httpfs; LOAD httpfs;")
duckdb.sql("""
    SELECT COUNT(*) FROM 's3://bucket/variants/*.csv.gz'
    WHERE qual > 30
""").df()

# JOIN across formats: Parquet + CSV + Pandas DF in same query
interop.py — pandas/polars/arrow zero-copy
import duckdb, pandas as pd, polars as pl

pdf  = pd.DataFrame({"gene":["TP53","BRCA1"], "score":[9,7]})
pldf = pl.DataFrame({"gene":["TP53","EGFR"], "hits":[50,30]})

# DuckDB sees `pdf` and `pldf` as if they were SQL tables — zero copy
result = duckdb.sql("""
    SELECT pdf.gene, pdf.score, pldf.hits
    FROM pdf JOIN pldf USING (gene)
""")
result.df()    # back to pandas
result.pl()    # back to polars
result.arrow() # back to arrow Table
duckdb.R
library(duckdb)
library(DBI)
library(dplyr)
library(dbplyr)

con <- dbConnect(duckdb(), ":memory:")

# Query Parquet directly
dbGetQuery(con, "
  SELECT chrom, COUNT(*) AS n
  FROM 'variants_partitioned/*/*.parquet'
  WHERE qual > 30
  GROUP BY chrom
")

# Or use dbplyr — write tidyverse, run as SQL
v <- tbl(con, "variants")
v %>% filter(qual > 30) %>% count(chrom) %>% collect()
主題 3

三、out-of-core：Dask、Arrow Dataset、data.table、R arrow

當資料超過單機 RAM 但仍能放上一顆 SSD，out-of-core (磁碟分塊) 是首選。Dask 提供 pandas-like API + lazy DAG，能跨多核或多機。R 端的對等物是 arrow::open_dataset() + dbplyr 寫 lazy query；data.table 雖然是 in-RAM，但在記憶體允許下幾乎是 R 最快的單機方案。

When data exceeds RAM but fits on SSD, out-of-core (disk-chunked) is the way. Dask gives a pandas-like API + lazy DAG, scaling across cores or machines. The R equivalent is arrow::open_dataset() + dbplyr; data.table stays in-RAM but is essentially the fastest single-machine R option when memory allows.

dask.py — pandas API, out-of-core
import dask.dataframe as dd

# Loads only chunk metadata; data stays on disk
ddf = dd.read_parquet("variants_partitioned/")

result = (ddf[ddf.qual > 30]
          .groupby("chrom")
          .agg({"pos": "count", "qual": "mean"}))

# Nothing computed until .compute() — like Polars LazyFrame
out = result.compute()

# Distributed across machines:
from dask.distributed import Client
client = Client("tcp://scheduler:8786")
out = result.compute()    # now runs across the cluster
arrow_dataset.R — R out-of-core
library(arrow)
library(dplyr)

# Multi-file lazy dataset — partitioning auto-detected
ds <- open_dataset("variants_partitioned/",
                   partitioning = "chrom")

ds %>%
  filter(qual > 30) %>%
  group_by(chrom) %>%
  summarise(n = n(), mean_q = mean(qual)) %>%
  collect()      # now executes with pushdown

# Works on 500GB on a laptop with 16GB RAM
data_table.R — in-RAM speed king for R
library(data.table)

# fread is 5-10× faster than read.csv on big files
DT <- fread("variants.tsv")

# DT[i, j, by] grammar — extremely fast
DT[qual > 30, .(n = .N, mean_q = mean(qual)), by = chrom]

# In-place update without copy (key feature):
DT[, gene_id := paste0(chrom, "_", pos)]

# Set key for binary-search joins
setkey(DT, chrom, pos)
DT[CJ("chr17", 7575000:7590000)]   # instant region query
spark_hail.py — when you really need distributed
# PySpark — generic distributed DataFrame
from pyspark.sql import SparkSession, functions as F
spark = SparkSession.builder.appName("variants").getOrCreate()

df = spark.read.parquet("s3://genomics/variants/*.parquet")
(df.filter(F.col("qual") > 30)
   .groupBy("chrom").count()
   .show())

# Hail — genomics-native distributed (built on Spark) for variant data
import hail as hl
hl.init()
mt = hl.read_matrix_table("gs://bucket/gnomad.mt")
mt = mt.filter_rows(mt.info.AF[0] > 0.01)
mt.aggregate_rows(hl.agg.counter(mt.locus.contig))

📝 自我檢測 (3 題)

Q1. Polars 的 LazyFrame 相對於 pandas eager 執行的最大優勢是？

Q1. Polars's LazyFrame vs pandas eager — biggest win?

A. 自動把資料寫到雲端A. Auto-saves to the cloud
B. 永遠用 0 RAMB. Always uses 0 RAM
C. 整體 query plan 最佳化 (predicate / projection pushdown、合併運算)C. Whole-plan optimization (predicate/projection pushdown, op fusion)
D. 自動將 DataFrame 變成 GPU arrayD. Auto-converts DataFrame to GPU array
答案 C。Lazy 模式讓 Polars 在執行前看到整個 query plan，可以把 filter 推到掃描階段、只讀需要的欄位、合併 group_by 與 agg——這些都是 eager pandas 做不到的。
Answer: C. Lazy mode lets Polars see the full plan before running — push filters down to scan, project only needed columns, fuse group_by with agg. Pandas eager can't do these.

Q2. 你有 200 個 Parquet 檔，想用 SQL 跨檔聚合，且不想啟動資料庫服務。最佳工具？

Q2. 200 Parquet files, want SQL across all, but no database server. Best tool?

A. PostgreSQLA. PostgreSQL
B. DuckDBB. DuckDB
C. SQLiteC. SQLite
D. MongoDBD. MongoDB
答案 B。DuckDB 是 in-process OLAP 引擎，原生支援 SELECT * FROM 'data/*.parquet'，零組態、零伺服器。SQLite 是 OLTP，列式儲存與 Parquet 直查皆不擅長。
Answer: B. DuckDB is an in-process OLAP engine with native SELECT * FROM 'data/*.parquet' support — zero config, no server. SQLite is OLTP and lacks Parquet/columnar native support.

Q3. 何時 Spark / Hail 才是合理選擇？

Q3. When does Spark / Hail actually pay off?

A. 10MB metadata 表的清理A. Cleaning a 10MB metadata table
B. 一個樣本的差異表現分析B. Single-sample differential expression
C. 100 個樣本的 PCA 視覺化C. PCA visualization on 100 samples
D. 數萬人全基因組 variant 跨群體頻率分析 (gnomAD scale)D. Cross-cohort variant frequency on 10k+ whole genomes (gnomAD scale)
答案 D。Spark / Hail 的啟動成本很高，只在資料 > 單機 RAM 數十倍、且需要跨節點計算時才划算。小資料用 Spark 是反模式。
Answer: D. Spark/Hail have heavy startup cost; they pay off only when data dwarfs single-machine RAM and requires multi-node computation. Spark on small data is an anti-pattern.
