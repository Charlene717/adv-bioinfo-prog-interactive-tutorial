資料庫與 REST API：把外部資料工程化

SQL 不只是「資料庫人才學的東西」；當你的分析會被引用、會被另一個人重跑、會被 50 個樣本掃過，把資料放進 SQLite / PostgreSQL / DuckDB 比 CSV 散落 100 個資料夾要好得多。本章涵蓋 SQL essentials、SQLAlchemy / dbplyr、Ensembl / UniProt / E-utils REST API、與快取/重試策略。

SQL isn't just "for DBAs." Once your analysis gets cited, re-run by a colleague, or swept across 50 samples, putting data in SQLite / PostgreSQL / DuckDB beats 100 folders of stray CSVs. This chapter covers SQL essentials, SQLAlchemy / dbplyr, Ensembl / UniProt / E-utils REST APIs, plus caching and retry strategy.

動機

為什麼分析要靠資料庫？

許多生醫資料分析的瓶頸不在演算法，而在「資料管理」：

  • 資料散落在 50 個 CSV，重複欄位、單位不一致；
  • 每次重跑都得手動爬 NCBI / Ensembl / STRING，遇上限速或斷線就要重來；
  • 樣本中繼資料和變異註釋分散在不同 Excel，沒人能保證版本一致。

解法分兩條軸線：本地端用 SQL / 資料庫把資料正規化；對外部資料用 REST/GraphQL API + 快取把網路存取變可重現。本章把這兩條線整合。

The bottleneck in many bio analyses isn't the algorithm — it's data management:

  • Data scattered across 50 CSVs with overlapping columns and mismatched units;
  • Re-runs require manually crawling NCBI / Ensembl / STRING — rate limits or network blips force restarts;
  • Sample metadata and variant annotations live in disconnected Excel files; no one can prove the versions match.

Two axes of solution: locally, use SQL/databases to normalize data; externally, use REST/GraphQL + caching to make network access reproducible. This chapter integrates both.

💡
取捨：不是每個分析都要 PostgreSQL。先用 SQLite / DuckDB (零組態)；當有多人寫入或需要併發，再升級到 PostgreSQL。 Trade-off: Not every analysis needs PostgreSQL. Start with SQLite/DuckDB (zero config); upgrade to PostgreSQL when you need concurrency or multi-writer access.
主題 1

一、生醫分析必備 SQL：JOIN、window function、index

三類查詢在生醫分析裡反覆出現：(1) 將樣本表與變異表 JOIN；(2) 用 window function 算每個基因內變異的排名；(3) 對常用欄位建 index 讓查詢從 5 分鐘變 50 毫秒。

Three query patterns recur in bio analysis: (1) JOIN samples to variants; (2) window function for per-gene variant ranking; (3) indexes to cut a 5-minute query down to 50 ms.

joins.sql — sample × variant
-- Find all high-quality variants in BRCA1 across all "tumor" samples
SELECT s.sample_id, s.tissue, v.chrom, v.pos, v.ref, v.alt, v.qual
FROM   variants v
INNER JOIN samples s   USING (sample_id)
INNER JOIN gene_regions g ON v.chrom = g.chrom
                            AND v.pos BETWEEN g.start AND g.end
WHERE  g.gene_symbol = 'BRCA1'
  AND  s.condition  = 'tumor'
  AND  v.qual > 30
ORDER BY v.pos;
window.sql — per-gene ranking
-- Rank variants within each gene by quality; tag top-3
SELECT *
FROM (
  SELECT v.*, g.gene_symbol,
         RANK() OVER (PARTITION BY g.gene_symbol ORDER BY v.qual DESC) AS rk,
         AVG(v.qual) OVER (PARTITION BY g.gene_symbol) AS gene_mean_qual
  FROM variants v
  JOIN gene_regions g ON v.chrom=g.chrom
                       AND v.pos BETWEEN g.start AND g.end
) ranked
WHERE rk <= 3;
index.sql — make it fast
-- Without index: scan 50M rows = ~5 minutes
-- With index:    binary lookup  = ~50 ms

CREATE INDEX idx_variants_pos
  ON variants(chrom, pos);

CREATE INDEX idx_variants_sample
  ON variants(sample_id);

-- Composite index on (chrom, pos) supports both range and equality.
-- Column order matters: put high-cardinality / equality column first.

-- For Postgres + range queries:
CREATE INDEX idx_variants_gist
  ON variants USING gist(int4range(pos, pos+1));
explain.sql — read the plan
-- Always check the plan before declaring a query "slow"
EXPLAIN ANALYZE
SELECT chrom, COUNT(*) FROM variants WHERE qual > 30
GROUP BY chrom;

-- Look for:
--   Seq Scan       → no index used (often the bug)
--   Index Scan     → good
--   Hash Aggregate → expected for GROUP BY
--   Rows removed   → high count = predicate not pushed down to scan
主題 2

二、本地端資料庫：SQLite / DuckDB / PostgreSQL / MongoDB

四種代表性選擇：

  • SQLite — 單檔、零組態、OLTP 偏向；當「可攜分析資料庫」很完美。
  • DuckDB — 同為單檔零組態，但是 columnar OLAP；分析查詢比 SQLite 快 10-100×。
  • PostgreSQL — 多人寫入、嚴格事務、extensions 多 (pgvector 存 embeddings、PostGIS 存空間)。
  • MongoDB — 文件式、適合巢狀 JSON (e.g. clinvar 變異註釋、ClinGen 表型)。

Four representative choices:

  • SQLite — single file, zero config, OLTP-leaning. Perfect as a "portable analysis DB."
  • DuckDB — same single-file zero-config story, but columnar OLAP. 10-100× faster on analytic queries.
  • PostgreSQL — multi-writer, strict transactions, rich extensions (pgvector for embeddings, PostGIS for spatial).
  • MongoDB — document store, good for nested JSON (e.g. ClinVar variant annotations, ClinGen phenotypes).
sqlite.py + SQLAlchemy
from sqlalchemy import create_engine, text
import pandas as pd

eng = create_engine("sqlite:///genomics.db")

# Bulk ingest a TSV → table
pd.read_csv("variants.tsv", sep="\t").to_sql(
    "variants", eng, if_exists="replace", index=False, chunksize=100_000)

# Parameterised query — never string-format SQL!
with eng.connect() as con:
    df = pd.read_sql(text("""
        SELECT chrom, pos, qual FROM variants
        WHERE sample_id = :sid AND qual > :q
    """), con, params={"sid":"S001", "q":30})
postgres.py + psycopg + pgvector
import psycopg
from pgvector.psycopg import register_vector
import numpy as np

with psycopg.connect("postgresql://user:pw@host/genomics") as con:
    register_vector(con)
    con.execute("""
        CREATE TABLE IF NOT EXISTS gene_embeddings (
            gene_id text PRIMARY KEY,
            emb     vector(1536)
        )
    """)
    # Store an OpenAI / Geneformer embedding
    emb = np.random.rand(1536).astype("float32")
    con.execute("INSERT INTO gene_embeddings VALUES (%s, %s)", ("BRCA1", emb))

    # Approximate-nearest-neighbour: which 5 genes are closest to BRCA1?
    rows = con.execute("""
        SELECT gene_id FROM gene_embeddings
        ORDER BY emb <-> %s   -- L2 distance
        LIMIT 5
    """, (emb,)).fetchall()

主題 3

三、REST 與 GraphQL — Ensembl、UniProt、E-utils、STRING、GTEx

常用 API 速覽：

  • Ensembl REST — gene lookup、variant、xrefs；同時提供 GraphQL (新)。
  • UCSC REST — 基因組座標、track 資料。
  • UniProt — 蛋白序列、結構域、翻譯後修飾。
  • NCBI E-utils — PubMed、GenBank、SRA；強制 3 req/s (有 API key 可到 10)。
  • STRING — 蛋白互動網路。
  • GTEx — 組織特異性表現。
  • GEOquery / TCGAbiolinks — R 端 wrapper，把 GEO/TCGA 下載抽象成函式。

API quick reference:

  • Ensembl REST — gene lookup, variants, xrefs; also offers GraphQL (newer).
  • UCSC REST — genomic coordinates, track data.
  • UniProt — protein sequences, domains, PTMs.
  • NCBI E-utils — PubMed, GenBank, SRA; hard-limited to 3 req/s (10 with API key).
  • STRING — protein-protein interaction networks.
  • GTEx — tissue-specific expression.
  • GEOquery / TCGAbiolinks — R-side wrappers that abstract GEO/TCGA downloads into functions.
ensembl.py — REST + retry + cache
import requests, requests_cache
from tenacity import retry, stop_after_attempt, wait_exponential

# Persistent cache — same URL returns cached response across runs
requests_cache.install_cache(".ensembl_cache", expire_after=86400)

@retry(stop=stop_after_attempt(5),
       wait=wait_exponential(1, 2, 30))
def lookup_symbol(symbol: str) -> dict:
    r = requests.get(
        f"https://rest.ensembl.org/lookup/symbol/human/{symbol}",
        headers={"Accept":"application/json"}, timeout=10)
    r.raise_for_status()
    return r.json()

info = lookup_symbol("BRCA1")
print(info["id"], info["seq_region_name"], info["start"])

eutils.py — PubMed batch retrieval
import requests, time

# NCBI: 3 req/s without key, 10 with key
BASE = "https://eutils.ncbi.nlm.nih.gov/entrez/eutils"
API_KEY = "your_key_here"

def esearch(term: str, retmax: int = 100) -> list[str]:
    r = requests.get(f"{BASE}/esearch.fcgi", params={
        "db":"pubmed", "term":term, "retmax":retmax,
        "retmode":"json", "api_key":API_KEY})
    return r.json()["esearchresult"]["idlist"]

def efetch_batch(ids: list[str]) -> str:
    # Batch up to 200 IDs in one call — respect rate limit
    chunks = [ids[i:i+200] for i in range(0, len(ids), 200)]
    out = []
    for ch in chunks:
        r = requests.get(f"{BASE}/efetch.fcgi", params={
            "db":"pubmed", "id":",".join(ch),
            "rettype":"abstract", "retmode":"text",
            "api_key":API_KEY})
        out.append(r.text)
        time.sleep(0.11)         # keep under 10 req/s
    return "\n".join(out)

📝 自我檢測 (3 題)

Q1. 對 5000 萬列 variants(chrom, pos) 表，下列何者讓區段查詢速度提升最大？

Q1. On a 50M-row variants(chrom, pos) table, what speeds up range queries most?

A. 加更多記憶體A. Add more RAM
B. CREATE INDEX idx ON variants(chrom, pos)B. CREATE INDEX idx ON variants(chrom, pos)
C. 用 SELECT * 而不是 SELECT 部分欄位C. SELECT * instead of selected columns
D. 把資料存成 pickleD. Pickle the data instead
答案 B。沒有 index 時，WHERE chrom='chr17' AND pos BETWEEN ... 必須 sequential scan 全表；建索引後變成 B-tree 二分搜尋，從 5 分鐘變成毫秒級。
Answer: B. Without an index, WHERE chrom='chr17' AND pos BETWEEN ... requires a full sequential scan. A composite B-tree index turns it into a binary-search range scan — minutes become milliseconds.

Q2. 你寫了 for g in genes: requests.get(...) 拉 5000 個基因；第二天又跑了一次，結果 NCBI 把你 ban 了。最該加的是？

Q2. Your script does for g in genes: requests.get(...) for 5000 genes; you re-run the next day and NCBI bans you. What should you add first?

A. 改成 GraphQLA. Switch to GraphQL
B. 改成 async/awaitB. Switch to async/await
C. requests_cache + 適當的 rate-limit/retry (tenacity)C. requests_cache + proper rate-limit/retry (tenacity)
D. 把所有資料先下載到 SQLite 再查D. Download all data into SQLite first
答案 C。第一要務是「不打第二次相同的請求」(cache) 與「打太快會自動退讓」(rate-limit/retry)。async 在這裡只會讓 ban 來得更快。
Answer: C. Priority one: don't repeat identical requests (cache) and back off when rate-limited (tenacity). Async without these gets you banned faster.

Q3. 何時 MongoDB 比 PostgreSQL 更合適？

Q3. When does MongoDB beat PostgreSQL?

A. 你需要嚴格事務 (ACID)A. You need strict ACID transactions
B. 你的資料是固定 schema 的 5 欄整數表B. Fixed-schema 5-column integer table
C. 你要做大量 multi-table JOINC. Heavy multi-table JOINs
D. ClinVar 風格的巢狀變異註釋、欄位每月變化D. ClinVar-style nested variant annotations with schema changing monthly
答案 D。MongoDB 的優勢是「文件式 + 模式彈性」，適合 schema 還在演化、深巢狀的 JSON。固定 schema、需 JOIN/ACID 仍然選 PostgreSQL。
Answer: D. MongoDB shines for nested documents and evolving schemas. Fixed schemas, joins, and ACID still belong with PostgreSQL.
