進階 R 與 tidy eval：從 dplyr 使用者到 Bioconductor 開發者

S3 / S4 / R6 三種物件系統、environments、NSE 與 tidy eval ({{ }}, !!, :=)、Bioconductor 風格的 Methods/Generics——把 R 從「分析工具」變成「函式庫語言」。

Three object systems (S3/S4/R6), environments, NSE and tidy eval ({{ }}, !!, :=), Bioconductor-style Methods/Generics — turn R from "analysis tool" into "library language."

動機

為什麼進階 R 對生醫資訊重要？

大部分 R 使用者熟悉 dplyr 三件套 (filter / mutate / summarise) 與 ggplot2，但生醫資訊套件 (DESeq2、limma、Seurat、SingleCellExperiment) 走的是另一條路：

  • 物件系統不同：Bioconductor 採 S4，Seurat 用 S4 + Reference class，R6 在 Shiny / plumber 流行。讀別人的 source code 必須認得三種系統。
  • 非標準求值 (NSE)：當你寫 dplyr::filter(df, gene == "BRCA1")，gene 不是字串而是「未求值的符號」——這就是 NSE，也是 tidyverse 簡潔語法的代價。包裝 dplyr 函式時你會踩到。
  • 環境與 lexical scope：R 的 closure、parent.frame()、environment() 是 Shiny reactive、testthat fixture、ggplot2 圖層機制的基礎。

本頁六個主題：S3 / S4 / R6 物件系統、environments、NSE 基礎、tidy eval、Bioconductor 慣例、新管線 |>。

Most R users know dplyr's triumvirate (filter / mutate / summarise) and ggplot2, but Bioconductor packages (DESeq2, limma, Seurat, SingleCellExperiment) take a different road:

  • Different object systems: Bioconductor uses S4; Seurat mixes S4 + Reference class; R6 is popular in Shiny / plumber. Reading others' source code requires recognizing all three.
  • Non-standard evaluation (NSE): when you write dplyr::filter(df, gene == "BRCA1"), gene is not a string but an "unevaluated symbol" — that's NSE, the cost of tidyverse brevity. You'll hit it when wrapping dplyr functions.
  • Environments & lexical scope: R's closures, parent.frame(), environment() underpin Shiny reactives, testthat fixtures, and ggplot2 layer mechanics.

Six topics: S3/S4/R6, environments, NSE basics, tidy eval, Bioconductor conventions, the new pipe |>.

主題 1

一、S3 vs S4 vs R6：三種物件系統

R 有四種以上的物件系統 (S3, S4, RC/Reference class, R6, R7 草案中)。實務上你只需要熟悉前三，且 90% 場景用得到的是 S3 與 S4。

R has 4+ object systems (S3, S4, RC, R6, R7 in draft). In practice only the first three matter, and 90% of code uses S3 or S4.

系統 正式程度 可變性 Dispatch 代表套件
S3非正式copy-on-modify單一參數lm()、glm()、base R
S4正式copy-on-modify多重 dispatchBioconductor (DESeqDataSet, SummarizedExperiment)
R6OO 風格可變物件方法Shiny modules, plumber, processx
s3.R — informal: just set the class attribute
# Create an S3 "object" by attaching a class label to a list
new_qc_result <- function(n_cells, n_kept, params) {
  obj <- list(n_cells = n_cells, n_kept = n_kept, params = params)
  class(obj) <- "qc_result"
  obj
}

# Define a method by naming convention: <generic>.<class>
print.qc_result <- function(x, ...) {
  cat(sprintf("QC result: %d / %d cells kept (%.1f%%)\n",
              x$n_kept, x$n_cells, 100 * x$n_kept / x$n_cells))
}

res <- new_qc_result(5000, 4321, list(min_features = 200))
print(res)   # dispatches to print.qc_result
s4.R — formal: typed slots (Bioconductor style)
# Define a class with typed slots
setClass(
  "QCResult",
  representation = representation(
    counts   = "matrix",
    cell_meta = "data.frame",
    params   = "list"
  ),
  prototype = list(params = list())
)

# Define a *generic* and a *method* (multi-dispatch ready)
setGeneric("filterCells", function(x, min_features) standardGeneric("filterCells"))

setMethod("filterCells", signature(x = "QCResult", min_features = "numeric"),
  function(x, min_features) {
    keep <- x@cell_meta$nFeature >= min_features
    x@counts <- x@counts[, keep]
    x@cell_meta <- x@cell_meta[keep, ]
    x
  })

qc <- new("QCResult",
           counts = matrix(0, nrow = 100, ncol = 50),
           cell_meta = data.frame(nFeature = sample(100:5000, 50)))
qc2 <- filterCells(qc, min_features = 500)
r6.R — encapsulated, mutable (Shiny / plumber)
library(R6)

PipelineRun <- R6Class("PipelineRun",
  public = list(
    sample_id = NULL,
    n_reads   = 0,
    initialize = function(sample_id) {
      self$sample_id <- sample_id
    },
    add_reads = function(n) {
      self$n_reads <- self$n_reads + n   # mutates in place!
      invisible(self)                    # enable chaining
    },
    summary = function() {
      sprintf("%s: %d reads", self$sample_id, self$n_reads)
    }
  ),
  private = list(
    .hash = NULL
  )
)

run <- PipelineRun$new("SRR123")
run$add_reads(1e6)$add_reads(5e5)   # method chaining
run$summary()
python_equiv.py — closest Python idiom for each
# S3 ≈ duck typing + isinstance dispatch (informal)
class QCResult:
    def __init__(self, n_cells, n_kept): self.n_cells, self.n_kept = n_cells, n_kept
    def __repr__(self): return f"QC: {self.n_kept}/{self.n_cells}"

# S4 ≈ multipledispatch / functools.singledispatch (multi-dispatch)
from functools import singledispatch
@singledispatch
def filter_cells(x, min_features): raise NotImplementedError
@filter_cells.register
def _(x: QCResult, min_features: int): ...

# R6 ≈ regular Python class (mutable, reference semantics)
class PipelineRun:
    def __init__(self, sid): self.sample_id = sid; self.n_reads = 0
    def add_reads(self, n): self.n_reads += n; return self
⚠️
常見陷阱：S3/S4 是 copy-on-modify——x@slot <- val 不會修改傳入的 x，而是回傳新物件，你必須 x <- f(x)。R6 是 reference，傳入函式內修改會影響外部變數。混用兩種語意是 bug 的溫床。 Common pitfall: S3/S4 are copy-on-modify — x@slot <- val does NOT mutate the passed x; it returns a new object, so you must write x <- f(x). R6 is reference; mutating inside a function affects the outer variable. Mixing these semantics is a bug magnet.

主題 2

三、Environments 與 lexical scope

R 的每個函式都帶著定義它的 environment (closure)。這個機制支撐 ggplot2 圖層、Shiny reactive、testthat fixture 與大部分 metaprogramming。理解 environment()、parent.frame()、globalenv() 三者的差別是進階 R 的分水嶺。

Every R function carries the environment in which it was defined (a closure). This underpins ggplot2 layers, Shiny reactives, testthat fixtures, and most metaprogramming. Knowing environment(), parent.frame(), and globalenv() is the watershed for advanced R.

closure.R — function carries its defining env
# Factory: returns a configured function that "remembers" its threshold
make_qc_filter <- function(min_features) {
  function(cell_meta) {
    cell_meta[cell_meta$nFeature >= min_features, ]
  }
}

filt_strict <- make_qc_filter(500)
filt_loose  <- make_qc_filter(200)

# Each closure carries its own bindings — peek inside:
environment(filt_strict)$min_features   # 500
environment(filt_loose)$min_features    # 200
parent_frame.R — for NSE & deparse(substitute(x))
# parent.frame() = the env where the function was CALLED (dynamic)
# environment() = the env where the function was DEFINED (lexical)

inspect <- function(x) {
  # Capture the NAME the caller used:
  caller_name <- deparse(substitute(x))
  cat("You passed variable named:", caller_name, "\n")
  cat("Its value is:", x, "\n")
}

my_var <- 42
inspect(my_var)
# > You passed variable named: my_var
# > Its value is: 42

# This is how plot() gets "main = my_var" axis labels for free.

主題 3

四、Non-Standard Evaluation — substitute / quote / eval

大部分 R 函式採「標準求值」：呼叫 f(x + 1) 時，x + 1 先被求值。但 dplyr::filter(df, x > 10) 不會立即把 x > 10 求值——它要先看 df 裡有沒有 x 這個欄位。這就是 NSE，由三個函式撐起：substitute() 捕捉未求值表達式、quote() 建立未求值表達式、eval() 在指定 env 求值。

Most R functions use "standard evaluation": f(x + 1) evaluates x + 1 first. But dplyr::filter(df, x > 10) does NOT eagerly evaluate x > 10 — it first checks whether df has a column named x. That's NSE, built on three functions: substitute() captures the unevaluated expr, quote() builds one, eval() evaluates in a chosen env.

substitute.R — capture caller's expression
my_summary <- function(data, col) {
  col_expr <- substitute(col)        # capture: e.g. `gene_count`
  col_name <- deparse(col_expr)        # to string: "gene_count"
  values   <- eval(col_expr, envir = data)
  cat("Summary of", col_name, ":\n")
  print(summary(values))
}

df <- data.frame(gene_count = rpois(100, 50), mt_pct = runif(100, 0, 20))
my_summary(df, gene_count)   # pass column name unquoted
quote_eval.R — building expressions programmatically
# quote() captures whatever expression you write, without evaluating it
expr <- quote(mean(x) + sd(x))
expr                          # mean(x) + sd(x)
class(expr)                   # "call"

# Inject values into a pre-built call:
my_data <- list(x = rnorm(100))
eval(expr, envir = my_data)   # works — evaluates against my_data

# Build a call dynamically: glm(y ~ x1 + x2, ...) with chosen vars
make_formula <- function(response, predictors) {
  as.formula(paste(response, "~", paste(predictors, collapse = " + ")))
}
make_formula("survival", c("age", "stage", "BRCA1_expr"))

主題 4

五、tidy eval：{{ }}、!!、:= 三劍客

tidy eval 是 rlang 套件對 NSE 的現代封裝。三個關鍵操作：embrace {{ x }} 把參數從呼叫者「整段轉發」進 dplyr 動詞；!! (bang-bang) 取出已捕捉的 quosure 注入；:= (walrus) 在 LHS 動態命名新欄位。

tidy eval is rlang's modern wrapper around NSE. Three key operators: embrace {{ x }} forwards an argument straight into dplyr verbs; !! (bang-bang) injects a captured quosure; := (walrus) lets you name a new column dynamically on the LHS.

embrace.R — {{ x }} for thin wrappers
library(dplyr); library(rlang)

# Wrap dplyr in 1 line — works on ANY column the caller names
summarise_gene <- function(data, gene_col, group_col) {
  data %>%
    group_by({{ group_col }}) %>%
    summarise(
      mean_expr = mean({{ gene_col }}, na.rm = TRUE),
      median_expr = median({{ gene_col }}, na.rm = TRUE),
      .groups = "drop"
    )
}

# Caller uses bare column names, just like real dplyr
summarise_gene(scRNA_df, BRCA1, cell_type)
enquo_bang.R — capture once, inject many times
# For more complex flows, capture with enquo() and inject with !!
plot_qc <- function(data, x, y, color) {
  x_q     <- enquo(x)
  y_q     <- enquo(y)
  color_q <- enquo(color)

  ggplot2::ggplot(data, ggplot2::aes(x = !!x_q, y = !!y_q, color = !!color_q)) +
    ggplot2::geom_point(alpha = 0.5) +
    ggplot2::labs(x = as_label(x_q), y = as_label(y_q))
}

plot_qc(cells_df, nFeature, percent_mt, batch)

# For lists of args, use enquos() (plural) + !!! (bang-bang-bang)
group_summary <- function(data, ...) {
  groups <- enquos(...)
  data %>% group_by(!!!groups) %>% summarise(n = n(), .groups = "drop")
}
group_summary(scRNA_df, sample, cell_type, condition)
walrus.R — dynamic LHS column names
# := allows the NEW column name to come from a variable
add_log <- function(data, col) {
  col_name <- as_label(enquo(col))
  new_name <- paste0("log_", col_name)
  data %>% mutate(!!new_name := log1p({{ col }}))
}

add_log(counts_df, BRCA1)
# adds a column named "log_BRCA1"

add_log(counts_df, TP53)
# adds a column named "log_TP53"
multi_args.R — varargs across multiple genes
# Build n new "log_*" columns from n input columns
add_log_multi <- function(data, ...) {
  cols <- enquos(...)
  new_cols <- map(cols, ~ quo(log1p(!!.x)))
  names(new_cols) <- paste0("log_", map_chr(cols, as_label))
  data %>% mutate(!!!new_cols)
}

counts_df %>% add_log_multi(BRCA1, TP53, EGFR)
# adds 3 new columns: log_BRCA1, log_TP53, log_EGFR

記憶口訣：「想轉發 → {{ }}」、「想自己控制 → enquo() + !!」、「動態欄位名 → :=」、「一串參數 → enquos() + !!!」。 Mnemonic: "Just forward → {{ }}"; "want control → enquo() + !!"; "dynamic column name → :="; "list of args → enquos() + !!!".
主題 5

六、Bioconductor 慣例：Generics、BiocGenerics

Bioconductor 對於資料容器有一套近乎標準的介面：所有「樣本 × 特徵」型物件都實作 rownames() / colnames() / dim() / assay() / colData()。新套件透過 setMethod() 對 BiocGenerics 已宣告的 generic 註冊方法，使用者就能跨套件用一致的語法處理 SummarizedExperiment、SingleCellExperiment、DESeqDataSet。

Bioconductor enforces a near-standard interface for "sample × feature" containers: every such object implements rownames() / colnames() / dim() / assay() / colData(). New packages register methods on already-declared generics in BiocGenerics, so users get a uniform syntax across SummarizedExperiment, SingleCellExperiment, DESeqDataSet.

se_methods.R — extending a Bioconductor class
library(SummarizedExperiment)
library(S4Vectors)

# Define a subclass that adds QC metadata

# Register a method on the EXISTING BiocGenerics::show()

七、|> vs %>%：新管線 vs magrittr

R 4.1+ 內建 |> (base pipe)，與 magrittr 的 %>% 有微妙差異。新程式碼建議用 |>——更快、不需相依、語法清晰；但若需要 . placeholder 或 %>% 的特殊行為，仍可用 magrittr。

R 4.1+ ships |> (base pipe), subtly different from magrittr's %>%. New code should prefer |> — faster, no dependency, cleaner syntax; magrittr remains useful when you need its . placeholder or special behavior.

特性 |> (base, R 4.1+) %>% (magrittr)
需要套件否magrittr / dplyr
速度較快較慢
placeholder_ (R 4.2+, 限 named arg). (任意位置)
RHS 必為呼叫是否
side_by_side.R
# magrittr: . placeholder anywhere, RHS need not be a call
library(magrittr)
counts %>%
  filter(., gene == "BRCA1") %>%
  arrange(desc(.$expression))

# Base |>: cleaner, faster; placeholder is _ (and only for named args)
counts |>
  filter(gene == "BRCA1") |>
  arrange(desc(expression))

# Base |> requires the RHS to be a call. Lambda for non-call cases:
1:10 |> (\(x) x^2)()           # R 4.1+ lambda syntax
決策樹

該用哪個工具？

情境 → 工具對照

Q1→「我要快速包裝一個 list 給 print 用」 → S3 (設 class 屬性 + 寫 print.myclass)"Quick wrapper around a list with a print" → S3 (set class attr + print.myclass)
Q2→「我要寫 Bioconductor 套件或繼承 SummarizedExperiment」 → S4 + BiocGenerics"Writing a Bioconductor pkg or extending SummarizedExperiment" → S4 + BiocGenerics
Q3→「我要在 Shiny 模組中保留狀態 + 方法鏈」 → R6 (reference semantics)"Stateful Shiny module with method chaining" → R6 (reference semantics)
Q4→「我要寫 dplyr 的薄包裝、讓使用者傳裸欄位名」 → {{ col }} (embrace)"Thin dplyr wrapper that accepts bare column names" → {{ col }} (embrace)
Q5→「我要動態命名新欄位」 → := (walrus) + !!"Need a dynamic new column name" → := + !!
Q6→「我要做大量 O(1) 查表」 → new.env(hash = TRUE) 當 hash map 用"Heavy O(1) key lookups" → new.env(hash = TRUE) as a hash map

📝 自我檢測 (3 題)

Q1. 下列哪個物件系統具有 reference semantics (在函式內修改會影響外部變數)？

Q1. Which object system has reference semantics (mutation inside a function affects the outer variable)?

A. S3A. S3
B. R6B. R6
C. S4C. S4
D. 三者都是 copy-on-modifyD. All three are copy-on-modify
答案 B。R6 是 reference semantics — 物件在函式內被 mutate 後，外部變數也跟著變。S3/S4 都是 copy-on-modify。
Answer: B. R6 uses reference semantics — mutating inside a function affects the outer variable. S3 and S4 are copy-on-modify.

Q2. 你想寫一個 dplyr 薄包裝 my_filter(df, col, val)，讓使用者可以 my_filter(iris, Species, "setosa")。正確的實作是？

Q2. You want a thin dplyr wrapper my_filter(df, col, val) usable as my_filter(iris, Species, "setosa"). Correct implementation?

A. df %>% filter(col == val)A. df %>% filter(col == val)
B. df %>% filter(eval(col) == val)B. df %>% filter(eval(col) == val)
C. df %>% filter({{ col }} == val)C. df %>% filter({{ col }} == val)
D. df %>% filter(.data$col == val)D. df %>% filter(.data$col == val)
答案 C。{{ col }} (embrace) 是 tidy eval 中將參數整段轉發給 dplyr 動詞的標準方式。A 會去找名為 col 的欄位；D 需要傳字串而非裸名。
Answer: C. {{ col }} (embrace) is the canonical tidy-eval way to forward an argument to a dplyr verb. A looks for a literal column "col"; D needs the column as a string, not a bare name.

Q3. 關於 parent.frame() 與 environment() 的差別，下列何者正確？

Q3. About parent.frame() vs environment(), which is correct?

A. environment() 回傳函式被「定義」處的 env (lexical)；parent.frame() 回傳函式被「呼叫」處的 env (dynamic)A. environment() = the env where the function was DEFINED (lexical); parent.frame() = where it was CALLED (dynamic)
B. 兩者完全相同，只是別名B. They are aliases for the same thing
C. parent.frame() 一律回傳 globalenv()C. parent.frame() always returns globalenv()
D. environment() 只能用在 R6 物件中D. environment() only works inside R6 objects
答案 A。這是 R 的核心 — lexical scoping (定義處) vs 呼叫鏈 (呼叫處)。NSE 與 substitute() 通常搭配 parent.frame() 取得呼叫者的變數。
Answer: A. This is R's core distinction — lexical scoping (where defined) vs the call stack (where called). NSE with substitute() usually pairs with parent.frame() to access the caller's variables.
